ETR Study API — Scope and Requirements

This document defines the problem statement, service boundaries, requirements, and key scenarios for the ETR (Extract-Transform-Retrieve) backend built with FastAPI + SQLite + SQLAlchemy.

1. Problem Statement

Product goal: provide an API to manage ETR-based learning flows and control spaced reviews through A/B/C/D slots.

Problem: users struggle to retain material systematically and keep an actionable mistake log after review sessions.

Solution: the service stores note structure (Extract/Transform), manages Retrieve scenarios, moves cards across slots, and maintains an Error Log.

2. Context and Boundaries

In scope:

  • user registration/update;
  • creating and reading study notes;
  • retrieving notes that are due for review;
  • moving between slots by tags easy/hard/forgot;
  • error-log management;
  • ETR schedule recommendations.

Out of scope (current version): authentication/authorization, multi-user roles, push notifications, progress analytics, file attachments.

4. Functional Requirements

FR-1. User Management

  • The service must create a user on the first request by system_user_id.
  • The service must update username, full_name, and timezone on repeated registration requests.
  • The service uses internal id and external system_user_id for downstream operations.

FR-2. Note Creation

  • A note must include Extract fields: keywords, prompts/gaps, and hints.
  • A note must include Transform fields: a compact explanation in 5-7 bullet points.
  • On creation, a note must be placed into slot A and receive next_review_at.

FR-3. Retrieve Scenarios

  • The service returns notes whose review time has come (due).
  • The service updates the slot by review tag:
    • easy -> next slot (A->B->C->D);
    • hard/forgot -> return to slot A.
  • Every review action must be logged in retrieval_logs.

FR-4. Error Log

  • The service must store: question/key, incorrect answer, corrected answer.
  • The service must return the latest N mistakes from newest to oldest.

FR-5. Summary and Schedule

  • The service must provide distribution of active notes across A/B/C/D slots.
  • The service must provide a recommended ETR session schedule.

5. Non-Functional Requirements

Category Requirement
Performance Typical API requests should complete in under 300 ms with a local DB and light load; avoid N+1 queries and unnecessary serialization overhead in hot paths.
Reliability Write operations must be transactional; failures must trigger rollback semantics and return stable error contracts.
Maintainability Transport logic must be separated from business logic; domain layer must be reusable and endpoint behavior must be covered by automated tests.
Extensibility New delivery channels (CLI, gRPC, another bot) should be added via adapters without rewriting core logic.
API Contract Governance OpenAPI schemas, examples, and error contracts are part of public API and must stay backward-compatible (additive evolution, no silent contract breaks).
Validation Consistency All external input must be validated at API boundary and normalized into code-based error payloads with immutable code/key semantics.
Security Baseline Secrets are never committed; local .env is excluded from VCS, and production deployment must use environment-managed secrets.
Observability Readiness Logs and error payloads must be structured enough for support and debugging; critical failures should be diagnosable from logs and request context.
Release Quality Gate Deployment must be preceded by mandatory quality pipeline: environment check, automated tests, UML regeneration, and docs synchronization.
Documentation Governance Documentation is part of deliverable: README and docs page must be updated in the same change, and generated sections must remain sync-safe.

8. Error Matrix

This project uses code-based error contracts. Error Matrix documents the approach and extension rules, while full code catalogs stay in source-of-truth files.

Layer Contract shape Where managed Purpose
Business error (4xx) {"code","key","message","source":"business"} app/services/*, router responses Stable, domain-level failures
Validation error (422) {"error_type","endpoint","errors":[...]} app/errors/validation.py Consistent input-validation failures
Error schemas ApiErrorResponse, ValidationErrorResponse app/schemas/errors.py Typed contract for OpenAPI and clients
Swagger examples Code-key-message examples app/openapi/examples/errors.py Executable documentation in /docs

Full error code catalog is maintained in code to keep docs concise and always current for expanding API surface.

9. Development Guide

9.1 Project Philosophy

  • API contract is the product: request/response/error rules must always match runtime behavior.
  • Documentation evolves with code in the same change set.
  • Centralized source-of-truth files prevent drift as API scales.

9.2 Validation Standards (General)

  • Validate external input at API boundary (schemas/types/validators).
  • Do not duplicate request-shape validation in service/repository layers.
  • Normalize validation failures into stable error codes and keys.
  • Return consistent machine-readable payloads across endpoints.

9.3 Error Governance and Expansion

  • Error code + key are immutable public contracts.
  • Add new codes, do not repurpose existing ones.
  • Keep a fallback code path for unmapped validation failures.
  • Every endpoint should declare explicit error responses in Swagger.

9.4 Documentation Workflow at Scale

  • Maintain source of truth in: app/errors/validation.py, app/schemas/errors.py, app/openapi/examples/errors.py, router responses.
  • After API changes run make sync-docs; verify marker blocks only are auto-updated.
  • Keep documentation concise: explain standards and extension path, not full per-code dumps.

9.5 Testing Policy (Mandatory)

  • API changes without tests are considered incomplete and should not be released.
  • For each endpoint, keep at least one success and one failure-path test.
  • Keep tests deterministic and isolated from mutable local state.
  • Use make test for full run and make test-one path=... for targeted checks.

9.6 Pre-Deploy Procedure (Mandatory)

  • Before deployment, run make pre-deploy as a quality gate.
  • Pipeline includes: make check, make test, make docs, make sync-docs.
  • Deployment must be blocked if any step fails.
  • This keeps tests and documentation always aligned with deployed API behavior.
  • Use make deploy DEPLOY_CMD="..." to enforce pre-deploy gate before real deployment command.

10. Docs as Code Workflow

Documentation is treated as production artifact and evolves together with application code.

10.1 Source of Truth

  • Service behavior and contracts: app/
  • Error mapping: app/errors/validation.py
  • OpenAPI examples: app/openapi/examples/
  • UML sources: docs/uml/**/*.puml
  • Generated sections: marker blocks in README.md and this page

10.2 Required Commands

  • During implementation: make test, make docs, make sync-docs.
  • Before release: make pre-deploy.
  • Integrity check: make docs-check (must pass in PR/release flow).

10.3 Expansion Rule

  • Any API change must update tests, OpenAPI examples, and docs in the same change.
  • Contract changes must remain backward-compatible unless explicitly versioned.
  • Generated docs must be sync-safe and reproducible by command.
