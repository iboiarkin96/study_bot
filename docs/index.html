ETR Study API — Постановка и требования

Документ фиксирует постановку задачи, границы сервиса, требования и ключевые сценарии для backend-сервиса ETR (Extract–Transform–Retrieve) на FastAPI + SQLite + SQLAlchemy.

1. Постановка задачи

Цель продукта: предоставить API для управления учебным процессом по методике ETR и контролировать интервальные повторения через слоты A/B/C/D.

Проблема: пользователю сложно системно закреплять материал и вести журнал ошибок после сессий повторения.

Решение: сервис хранит структуру конспекта (Extract/Transform), управляет сценариями Retrieve, перемещает карточки по слотам и ведет Error Log.

2. Контекст и границы

Входит в scope:

  • регистрация/обновление пользователя;
  • создание и просмотр конспектов;
  • получение конспектов, готовых к повторению;
  • перемещение между слотами по тегам easy/hard/forgot;
  • ведение журнала ошибок;
  • предоставление рекомендованного расписания ETR.

Вне scope (текущая версия): авторизация, многопользовательские роли, push-уведомления, аналитика прогресса, файловые вложения.

3. Архитектурная декомпозиция

Сервис разделен на два домена обработчиков и общий доменный слой:

domains/general/logic_endpoints.py domains/http/handlers.py domains/telegram/handlers.py

  • General endpoints: переиспользуемая бизнес-логика, независимая от транспорта.
  • HTTP handlers: FastAPI-адаптер (валидация, OpenAPI, HTTP-коды).
  • Telegram handlers: Telegram-адаптер (формат команд/ответов).

4. Функциональные требования

FR-1. Управление пользователями

  • Сервис должен создавать пользователя при первом обращении по telegram_user_id.
  • Сервис должен обновлять username, full_name, timezone при повторной регистрации.
  • Сервис использует внутренний user_id для дальнейших вызовов.

FR-2. Создание конспекта

  • Конспект обязан содержать Extract-поля: ключевые слова, вопросы/пробелы, подсказки.
  • Конспект обязан содержать Transform-поля: плотный абзац из 5–7 пунктов.
  • При создании запись должна попадать в слот A и получить next_review_at.

FR-3. Сценарии Retrieve

  • Сервис возвращает список конспектов, срок повторения которых наступил (due).
  • Сервис должен обновлять слот по тегу:
    • easy → следующий слот (A→B→C→D);
    • hard/forgot → возврат в слот A.
  • Каждый акт повторения должен логироваться в retrieval_logs.

FR-4. Журнал ошибок

  • Сервис должен сохранять: вопрос/ключ, ошибочный ответ, исправленный ответ.
  • Сервис должен возвращать последние N ошибок в порядке от новых к старым.

FR-5. Сводка и расписание

  • Сервис должен отдавать распределение активных конспектов по слотам A/B/C/D.
  • Сервис должен предоставлять рекомендованное расписание ETR-сессии.

5. Нефункциональные требования

Категория Требование
Производительность Типовые API-запросы должны обрабатываться менее чем за 300 мс при локальной БД и малой нагрузке.
Надежность Операции записи выполняются транзакционно; при ошибке должна быть выполнена rollback-семантика.
Поддерживаемость Логика транспорта отделена от бизнес-логики; общий слой должен быть переиспользуемым.
Расширяемость Новые каналы доставки (CLI, gRPC, another bot) добавляются через адаптер без переписывания core-логики.
Документируемость HTTP-слой обязан публиковать OpenAPI-контракт и человекочитаемые описания эндпоинтов.

6. Контракты API (обязательно к поддержке)

GET /api/v1/health Проверка доступности сервиса.

POST /api/v1/users/register Регистрация или обновление пользователя.

POST /api/v1/conspectuses Создание конспекта в слоте A.

GET /api/v1/conspectuses/due Получение due-конспектов.

GET /api/v1/conspectuses/slots Сводка по слотам.

GET /api/v1/conspectuses/{id} Детальная карточка конспекта.

POST /api/v1/conspectuses/{id}/review Применение результата повторения.

POST /api/v1/errors Добавление ошибки в журнал.

GET /api/v1/errors Получение журнала ошибок.

GET /api/v1/schedule Получение рекомендованного расписания.

8. Критерии приемки

  • Все обязательные эндпоинты доступны и документированы в OpenAPI.
  • Логика слотов соответствует правилам: easy -> вперед, hard/forgot -> A.
  • Создание конспекта валидирует 5–7 bullets.
  • Error Log сохраняет и возвращает корректные поля.
  • Авторизация не внедрена, доступ контролируется уровнем окружения/сети.
