Overview

This page is the contract that says what we test, why, and to what depth. It is the single high-level reference; other QA pages refine it. If a page anywhere in the QA portal contradicts this strategy, this page wins until updated.

The strategy is intentionally narrow: it covers only the Study App (FastAPI service + docs portal). Cross-product testing standards do not apply here.

Mission & principles

Mission

QA exists to shorten the feedback loop between the team and reality: we catch defects before users do, surface unknowns early, and give the team confidence to release without drama.

Principles

  • Shift-left. Testing starts at the spec. A test plan is written against the API spec under docs/internal/api/ before code lands.
  • Risk-based. We do not test everything equally. Coverage follows risk (impact × likelihood), not feature count.
  • Automation first, manual where it pays. If a check repeats, it is automated. Manual time is reserved for exploration and UX.
  • Evidence-driven. Every "pass" must be reproducible. Screenshots, request/response captures, or log lines accompany manual checks.
  • One source of truth. A spec page in docs/internal/api/ is the contract. If the API behaves differently, the API is wrong, the spec is wrong, or the test is wrong — never "all three are right and disagree".
Scope

What we test

  • Public HTTP API — every operation under /api/v1/…. See internal API hub.
  • Documentation portal — pages under docs/: rendering, navigation, search, accessibility, theme, responsiveness.
  • Operational behavior — health/readiness, observability surfaces, error envelopes, structured logging.
  • Migrations — Alembic up/down on a representative dataset.
  • OpenAPI contract — the spec served at /openapi.json matches code and the baseline.

What we explicitly do not test

  • Third-party SaaS internals (only our integration boundaries).
  • Network/infrastructure beyond the container (DNS, upstream LBs, cloud platform).
  • Browsers older than the latest two major versions of Chromium and WebKit, unless an audit explicitly requests it.
  • Localization beyond English (the product is English-only).

Risk model

Risk model

We classify each feature on two axes — impact if it fails and likelihood of it failing — and pick coverage depth from a 3×3 grid.

Low likelihood Medium likelihood High likelihood
High impact Contract + manual happy path Contract + checklist + exploratory Contract + checklist + exploratory + regression baseline
Medium impact Contract only Contract + manual happy path Contract + checklist
Low impact Spot-check Contract only Contract + manual happy path

Examples. Auth, payments, persistent writes ⇒ high-impact. New free-form descriptive field on an internal admin page ⇒ low-impact. A region with frequent recent change (rapid commits, recent incidents) is high-likelihood by default.

Entry / exit

Entry & exit criteria

Entry to QA execution

  • Spec page exists under docs/internal/api/<resource>/ (or the equivalent for non-API features) and the spec DoD passes.
  • CI is green on the feature branch (unit, integration, contract, lint).
  • The feature is deployable to a test environment and reachable.
  • A test plan (or checklist reference) is written and reviewed.

Exit from QA (sign-off)

  • All in-scope checklist items pass; failures are linked to defect tickets.
  • No open defect at Severity 1 or Severity 2.
  • All defects below S2 either fixed, deferred with rationale, or accepted by the feature owner.
  • Evidence (screenshots, request/response captures, logs) is attached to the ticket.
  • The OpenAPI baseline is updated if the contract changed.
Oracles

Test oracles — how we decide pass vs fail

An oracle is the source of truth a test compares against. In priority order:

  1. The spec page under docs/internal/api/<resource>/ — it is the contract. The OpenAPI document is generated from code; the spec is the human contract.
  2. The OpenAPI baseline — for shape/schema. Drift is a contract change.
  3. Field conventions, error envelope, idempotency, pagination, versioning, observability conventions — see shared standards.
  4. Heuristics — when nothing else applies (HICCUPPS-F: History, Image, Comparable products, Claims, User expectations, Product, Purpose, Statutes, Familiarity).

If two oracles disagree, escalate — do not silently choose. Document the resolution as a spec change or a defect.

Metrics

Metrics we track

  • Defect escape rate — defects found post-release ÷ total defects on the release. Goal: trend down.
  • Coverage on golden paths — every API operation has at least one automated contract test and one acceptance checklist line.
  • Time to verify a fix — wall-clock from "Fixed" to "Verified" on a defect ticket.
  • Reopen rate — % of defects that returned to "In progress" after "Fixed". Target: < 10%.
  • Flaky test count — automated tests that fail intermittently in CI. Target: 0 outside an explicit allowlist.
