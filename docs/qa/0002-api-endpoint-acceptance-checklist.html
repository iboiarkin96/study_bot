QA checklist 0002: API endpoint acceptance

Internal — QA portal · Checklists

Overview

Run this checklist for every new or changed HTTP endpoint, against the operation's spec page under docs/internal/api/<resource>/. Mark each row Pass, Fail, or Needs follow-up; attach evidence per the API endpoint testing playbook.

Preconditions Contract Happy path Errors Auth Idempotency Observability Docs & OpenAPI Pass/fail
Preconditions

Preconditions

  • Spec page exists and passes spec DoD.
  • API is reachable on a test environment with the build SHA known.
  • You have a valid bearer token for an account with the required role.
  • HTTP client of your choice (curl, httpie, Postman) is ready and capturing.
Contract

Contract — schema and shape

# Check Expected
1.1 Path, method, version segment match the spec exactly. Identical.
1.2 Request body required/optional/types match the spec. Identical.
1.3 Response body schema matches for every documented status. Identical.
1.4 Field naming follows field conventions. Snake_case keys; UTC ISO-8601 timestamps; UUIDs as strings.
1.5 OpenAPI baseline test (tests/openapi/) is green. Pass; if the contract changed, baseline is regenerated and committed.
Happy path

Happy path

# Check Expected
2.1 Smallest valid body returns 2xx success. 200 / 201 per spec; body conforms to schema.
2.2 All optional fields, when supplied, round-trip unchanged. Echoed in the response and in a follow-up read.
2.3 List endpoints — pagination first → next → empty next_cursor. Per pagination.
2.4 Read-after-write — a follow-up GET returns the just-created resource. Visible immediately.
Errors

Error paths

# Check Expected
3.1 Missing required field. 422; envelope; field_errors points to the field.
3.2 Wrong type / out-of-range value. 422; envelope.
3.3 Resource not found (UUID for a missing row). 404; envelope; catalog code.
3.4 Conflict (forbidden state transition or unique constraint). 409; envelope; catalog code.
3.5 Wrong HTTP method. 405 with Allow header.
3.6 Wrong / missing Content-Type on body methods. 415; envelope.
3.7 Forced 5xx (e.g. break a downstream). 5xx; envelope; X-Request-ID traces in logs; no stack trace in the response.
Auth

Authentication / authorization

# Check Expected
4.1 No token. 401; envelope.
4.2 Malformed token / wrong scheme. 401; envelope; never 500.
4.3 Authenticated but lacking permission. 403; envelope.
4.4 Authenticated with sufficient role. 2xx.
Idempotency

Idempotency (write methods only)

# Check Expected
5.1 Same key + same body, twice within window. Both calls return identical body and status; no duplicate row.
5.2 Same key + different body. 409; envelope.
5.3 Same key after window expires. Treated as new request.
Observability

Observability

# Check Expected
6.1 Every response carries X-Request-ID. UUID; unique per request.
6.2 Structured log shows one record per request. Includes method, path, status, latency_ms, correlation id.
6.3 Sensitive headers / fields not logged in clear. Per observability conventions.
Docs & OpenAPI

Docs & OpenAPI

# Check Expected
7.1 Spec page passes DoD. All required sections non-empty.
7.2 Operation visible in Swagger UI with the expected request/response. Render is correct in both light and dark.
7.3 Examples in the spec match the live API. Copy-paste runs.
7.4 Sidebar entry exists under "API endpoints documentation". Active state highlights for this page.
Evidence

Evidence required for sign-off

  • Request/response capture for one happy path, one validation error, one auth error.
  • One X-Request-ID traced in the structured log.
  • Screenshot of Swagger UI showing the operation expanded.
  • Result line per row above (Pass / Fail / Needs follow-up), pasted into the ticket.
Pass/fail

Pass/fail rules

  • Pass: all rows above pass; evidence attached.
  • Fail: any contract, error envelope, auth, idempotency, or observability row fails.
  • Needs follow-up: non-blocking docs/wording issues with a tracked ticket.
