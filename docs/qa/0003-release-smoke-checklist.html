QA checklist 0003: Release smoke

Internal — QA portal · Checklists

Overview

Release smoke is the broad shallow check before tagging a release. The deeper walkthrough is in release smoke playbook. Run this checklist on the candidate build and paste the result into the release ticket.

Preconditions Health API golden paths Docs portal Observability Sign-off
Preconditions

Preconditions

  • CI green on the release commit; image built and tagged.
  • Migrations (if any) have been applied to the target environment.
  • Release ticket exists; build SHA and target environment are recorded.
  • Auth token, base URL, and a clean test account are ready.
Health

Health, readiness, OpenAPI

# Check Expected
1.1 GET /health 200; body matches the documented health envelope.
1.2 GET /ready 200 once dependencies are reachable.
1.3 GET /openapi.json 200; valid JSON; info.version matches release.
1.4 Container logs at startup No unhandled exception; one log line per route registered.
API golden paths

API golden paths

# Check Expected
2.1 Authenticated read endpoint returns 200 with valid shape. Schema validated.
2.2 Create + read-back on User. POST 201, GET 200; row present.
2.3 Create + read-back on Conspectus. POST 201, GET 200; row present.
2.4 Create + read-back on Error log. POST 201, GET 200; row present.
2.5 Idempotency replay on one POST with the same key. Same response twice; no duplicate row.
2.6 One forced 422 and one 401. Both conform to the error envelope.
Docs portal

Docs portal

# Check Expected
3.1 Open /docs/index.html. Loads with no console errors; sidebar populates.
3.2 Open one page from each major section (ADR, runbooks, internal API, QA). Page renders; breadcrumbs correct.
3.3 Search a known title. Result appears within first 5; click navigates.
3.4 Theme toggle cycles system → light → dark. Persists on reload.
3.5 Drawer (mobile width) opens, closes on Escape. Focus trapped while open.
3.6 Open Swagger UI page. Renders; one operation expanded shows correct request/response.
Observability

Observability

# Check Expected
4.1 Pick one X-Request-ID from a 2.x call. Found in the structured log; method/path/status match.
4.2 Pick one X-Request-ID from a 4xx call. Logged at warn level (or per policy); no PII leaked.
4.3 Metrics scrape (if enabled). Endpoint reachable; counters incremented after smoke calls.
Sign-off

Sign-off

  • Go: all rows pass; defects (if any) below S2.
  • Conditional go: S2 defects with explicit acceptance from the release owner.
  • No-go: any S1 open or any row in Health, API golden paths, or Observability failed.

Paste the sign-off note from the release smoke playbook into the release ticket with evidence.
