Overview

How a feature flows from idea to signed-off release, who does what at each stage, and what artefact is produced. This page is the choreography; the playbooks and templates are the steps.

Stages RACI Artefacts Cadence Release sign-off Hotfix path
Stages

Stages of the SDLC and what QA does at each

  1. Discovery / spec. Manager and developer draft a spec under docs/internal/api/<resource>/. QA reads the draft, asks questions, and identifies test risks before code is written. Output: spec page passing the spec DoD; risk notes attached to the ticket.
  2. Test planning. QA writes a test plan (template) for non-trivial features, or selects an existing checklist for routine work. Output: linked test plan or checklist with explicit scope and exit criteria.
  3. Implementation. Developer writes unit, integration, and contract tests alongside code. QA pairs on contract test design when the API surface is new. Output: green CI on the feature branch.
  4. QA execution. QA runs the test plan on a deployable build (staging or local). Defects are filed using the bug report template. Output: annotated test plan, defect tickets, evidence.
  5. Triage & fix. Defects are graded against the severity matrix; fixes reuse the defect lifecycle.
  6. Sign-off & release. QA confirms exit criteria are met, runs the release smoke checklist, and records the sign-off in the release ticket.
  7. Post-release. Track defect escapes; feed lessons back into the checklists and tests.

Artefacts

Artefacts produced by QA

  • Test plan — per non-trivial feature; links to checklists and risks.
  • Test cases — per scenario; descend from test case template.
  • Annotated checklist run — pass/fail per row plus evidence.
  • Defect ticket — using bug report template.
  • Release sign-off note — short summary attached to the release ticket.
  • Audit report (when scheduled) — see audit hub.
Cadence

Cadence

  • Per pull request — automated CI runs unit + integration + contract; QA reviews the PR if it touches public API or docs.
  • Per feature — full test plan + manual checklist run.
  • Per release — release smoke checklist + regression of impacted areas.
  • Quarterly — audit pass (UX/DX or REST API) recorded under audit hub.
Release sign-off

Release sign-off

QA's sign-off note on a release ticket must answer four questions:

  1. What scope was tested? (link to the test plan or checklists)
  2. What was found? (counts by severity, links to defects)
  3. What was deferred and why? (S3+ that won't ship as fixed)
  4. Recommendation: Go, Conditional go (with constraints), or No-go.
Hotfix path

Hotfix path

For an incident-driven hotfix the ceremonies compress, but the gates do not disappear:

  1. Reproduce on the affected environment; capture evidence.
  2. Add a failing test (contract or integration) before the fix.
  3. Apply the fix; the previously failing test passes.
  4. Run the release smoke checklist on a build of the hotfix branch.
  5. QA records sign-off on the incident ticket.
