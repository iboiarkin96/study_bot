Overview

A lightweight, IEEE-829-inspired test plan tailored to this project. Use it for any non-trivial feature or release. For routine work, link to an existing checklist instead.

When to use Sections Template Worked example
When

When to use a test plan

  • New feature touching multiple endpoints or pages.
  • Release ticket spanning > 1 area.
  • Migration affecting persisted data.
  • External-facing contract change.

For tiny single-PR fixes or doc edits, skip the plan and reference a checklist.

Sections

Sections — what each contains

Section Required Content
1. Identifier Yes Plan ID + version + date.
2. Scope Yes What's in, what's out, with explicit exclusions.
3. References Yes Links to spec, tickets, ADRs, related plans.
4. Risks Yes Top 3–7 risks with impact × likelihood and mitigations.
5. Test items & types Yes List of items × test types planned (contract, system, exploratory, …).
6. Approach Yes Tooling, environments, data, layers from the pyramid.
7. Pass / fail criteria Yes Per checklist and overall — anchors to severity matrix.
8. Entry / exit criteria Yes What must be true to start / stop testing.
9. Schedule Yes Calendar dates with single-owner responsibility.
10. Deliverables Yes Annotated checklists, defects, sign-off note.
11. Roles Yes Single accountable owner per row, RACI optional.
12. Open questions Yes Use the literal word None. if empty.

            - **Plan ID:** TP-<area>-<number>-v1
            - **Date:** <ISO>
            - **Author:** <name>
            - **Status:** Draft | In review | Approved

            ## 1. Scope
            - In: <list>
            - Out: <list>

            ## 2. References
            - Spec: <link>
            - Tickets: <ids>
            - ADRs: <ids>

            ## 3. Risks
            | # | Risk | Impact | Likelihood | Mitigation |
            |---|------|--------|------------|------------|

            ## 4. Test items and types
            | Item | Contract | System | Manual UI | Exploratory | A11y |
            |------|----------|--------|-----------|-------------|------|

            ## 5. Approach
            - Layers used (per test pyramid): …
            - Tooling: pytest, httpx, browser …
            - Environments: local, staging.
            - Data fixtures: …

            ## 6. Pass / fail criteria
            - Pass: all P1 cases pass; no S1/S2 open.
            - Fail: any S1/S2 open at exit.

            ## 7. Entry criteria
            - Spec passes DoD; CI green; build deployable.

            ## 8. Exit criteria
            - All in-scope cases executed; defects triaged; sign-off recorded.

            ## 9. Schedule
            | Date | Activity | Owner |
            |------|----------|-------|

            ## 10. Deliverables
            - Annotated checklist runs.
            - Defect tickets (linked).
            - Sign-off note on the release ticket.

            ## 11. Roles
            | Activity | R | A | C | I |
            |----------|---|---|---|---|

            ## 12. Open questions
            - <list> or "None."

Worked example — abbreviated

# Test plan: POST /api/v1/conspectuses idempotency

            - Plan ID: TP-API-CONSPECTUS-007-v1
            - Date: 2026-04-30
            - Author: Ivan B.

            ## 1. Scope
            - In: POST /api/v1/conspectuses with Idempotency-Key header — first call,
            retry within window, retry after window, conflict on different body.
            - Out: GET / PATCH / authentication scenarios (covered by TP-…-002).

            ## 3. Risks
            | # | Risk | Impact | Likelihood | Mitigation |
            |---|------|--------|------------|------------|
            | 1 | Replay attack creates duplicates | High | Med | Contract test on retry within window |
            | 2 | DB unique-constraint violation surfaces as 500 | Med | Low | Manual repro + log inspection |

            ## 4. Test items
            | Item | Contract | System | Manual UI | Exploratory | A11y |
            |------|----------|--------|-----------|-------------|------|
            | Idempotency key path | ✓ | ✓ |  | ✓ |  |

            ## 8. Exit criteria
            - All P1 cases pass.
            - No open S1/S2 defects.
            - OpenAPI baseline updated and contract test green.
