Overview

Canonical shape for a manually-executed test case. Copy the template block below into the ticket or test plan, fill every field, delete this commentary.

For automated contract tests, mirror the same fields in the test docstring so a future reader can map test ↔ behavior without leaving the file.

Fields

Fields and rules

Field Required Rule
ID Yes TC-<area>-<number>, e.g. TC-API-USER-014. Stable; never reused.
Title Yes Imperative, action + outcome. "Reject POST /user with missing email — 422".
Area Yes Top-level resource or page (User, Conspectus, Docs portal, Auth, …).
Type Yes One of: contract, system, manual UI, accessibility, exploratory, regression.
Priority Yes P1 / P2 / P3 — how often we run it (P1 every release, P3 ad-hoc).
Linked spec Yes Anchor under docs/internal/api/<resource>/ or other authoritative page.
Preconditions Yes System state required before step 1. List, never prose.
Test data Yes Inline values or a named fixture. Don't say "any user" — name it.
Steps Yes Numbered, one action per row. Past tense for the recorded run.
Expected Yes One row per step where the result is observable.
Postconditions No Cleanup steps if the test mutates state.
Evidence Yes What to capture for sign-off (screenshot of X, response body of Y).
Notes No Edge cases out of scope, related cases, or known flake conditions.

Worked example

Worked example

### TC-API-USER-014 — Reject POST /api/v1/user with missing email — 422

              - **Area:** User
              - **Type:** contract
              - **Priority:** P1
              - **Linked spec:** internal/api/user/operations/post-api-v1-user.html#request-body

              #### Preconditions
              - API is reachable on the local dev server at http://127.0.0.1:8000.
              - Caller has a valid auth token (see auth shared doc).

              #### Test data
              - Body: { "first_name": "Ivan", "last_name": "Boyarkin" }    (no email field)
              - Headers: Authorization: Bearer <token>, Idempotency-Key: tc-api-user-014-001

              #### Steps and expected results
              | # | Step | Expected |
              |---|------|----------|
              | 1 | POST /api/v1/user with the body above | HTTP 422 |
              | 2 | Inspect response body | Conforms to error envelope; contains code "validation_error" and a field error pointing at "email" |
              | 3 | Inspect response headers | X-Request-ID is present and is a UUID |

              #### Evidence to capture
              - Full request/response capture from the HTTP client.
              - The X-Request-ID header value (used to find the matching log line).

              #### Postconditions
              - None — request was rejected, no DB change.

              #### Notes
              - Pair test: TC-API-USER-015 covers blank string vs missing key (both reject as invalid).
Do / don't

Do / don't

  • Do link to a spec anchor as the oracle. Don't restate the spec in the case.
  • Do name fixtures (e.g. fixture: user_default). Don't say "any user".
  • Do capture the X-Request-ID on every API case. Don't rely on log timestamps to correlate.
  • Do split a multi-assertion case into multiple cases when assertions cover different concerns. Don't stuff happy-path + error path into one case.
  • Don't include implementation details (table names, internal classes) — they invalidate the case after refactors.
