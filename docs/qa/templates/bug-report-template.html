Overview

Every defect in this project is filed using this shape. A consistent shape lets a reviewer triage in 90 seconds and a developer reproduce without back-and-forth.

Title rules

Title rules

Title pattern: [area] <observed behavior> — <condition>. Aim for < 80 characters. State the bug, not the wish.

  • ✅ [API/user] POST /api/v1/user accepts empty email — should return 422.
  • ✅ [Docs] Search results panel does not close on Esc on Safari 17.
  • ❌ "Bug in user API" — what bug? where?
  • ❌ "Need to fix email validation" — that's the wish, not the observation.
Fields

Fields and rules

Field Required Rule
Title Yes See title rules above.
Severity Yes S1–S4. See matrix.
Priority Yes P1–P4. Set during triage; reporter may suggest.
Environment Yes Build SHA / version, env (local/staging), browser + version, OS.
Steps to reproduce Yes Numbered, minimal, deterministic. If non-deterministic, say so and provide repro rate (e.g. 3/10).
Expected Yes What should happen. Cite the spec.
Actual Yes What did happen. Quote, screenshot, or capture.
Evidence Yes Screenshots (annotated), HAR / request-response capture, X-Request-ID, log excerpts.
Impact Yes One sentence: who breaks, on what flow.
First observed Yes Build / commit where it first reproduced.
Workaround No If users can avoid the issue, describe how.
Suspect cause No Optional hypothesis; never required.
Linked items No Test case ID, spec anchor, related defects.

Worked example

Worked example

### Title
            [API/user] POST /api/v1/user accepts empty "email" — should return 422

            ### Severity / Priority
            - Severity: S2 (data integrity — invalid users persist)
            - Priority: P2 (suggested)

            ### Environment
            - Build: 2026-04-30 main @ ab5d00b
            - Environment: local
            - Client: httpie 3.2 / curl 8.4 — same result
            - Date observed: 2026-04-30

            ### Steps to reproduce
            1. Start API locally (`make run`).
            2. POST /api/v1/user with body {"first_name": "I", "last_name": "B", "email": ""}.
            3. Observe HTTP 201 and a new row in `users` table.

            Reproduction rate: 10/10.

            ### Expected
            Per internal/api/user/operations/post-api-v1-user.html#request-body, the email field
            must be a non-empty RFC 5322 address. Empty must yield 422 with envelope code
            "validation_error".

            ### Actual
            HTTP 201; response body returns a user_id; DB row exists with email = "".

            ### Evidence
            - screenshot-1.png — request and response in httpie.
            - X-Request-ID: 7c1a6f12-… — log shows `users.created`, no validation error.
            - HAR: post-user-empty-email.har

            ### Impact
            Any caller (CLI, integration partners) can persist invalid users. Downstream consumers
            will fail when they try to send mail to an empty address.

            ### Workaround
            Reject empty strings client-side until fixed.

            ### Suspect cause
            Pydantic field has min_length default of 0; needs `min_length=1` and an EmailStr type.

            ### Linked
            - Spec: internal/api/user/operations/post-api-v1-user.html#request-body
            - Test case: TC-API-USER-014
            - Related defects: none yet.
Do / don't

Do / don't

  • Do reproduce three times before filing. Intermittent ≠ unfilable, but say so.
  • Do capture the X-Request-ID. It is the magic link to the log.
  • Do attach annotated screenshots — arrows / boxes around the element under test.
  • Don't file the same defect twice. Search first; if a near-duplicate exists, comment with new evidence rather than open a new ticket.
  • Don't mix two defects in one report. Split.
  • Don't describe the fix. The defect is the observation; the fix is engineering's call.
