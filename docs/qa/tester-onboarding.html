Overview

Day-by-day path for a new tester. Follow it in order. By the end you will have run the app locally, exercised the API, verified a docs page, and filed a defect using the standard template.

Day 0 — access

Day 0 — access & tools

  • Get repo access (read at minimum, contributor for filing PRs).
  • Install the local toolchain — see local development. Confirm make, Python 3.x, Docker, and a Chromium-based browser are present.
  • Pick a defect tracker (GitHub Issues by default) and confirm you can create issues.
  • Install browser helpers: a JSON-formatted view, an HTTP client (Postman / Insomnia / httpie), and a screenshot tool that lets you annotate.
Day 1 — read

Day 1 — read the foundations

Read in this order. Each is short.

  1. Test strategy — what we test and why.
  2. Test pyramid — where each kind of check lives.
  3. QA process — how a feature flows end-to-end.
  4. Glossary — keep open in a tab.
  5. Severity matrix + defect lifecycle.
  6. Internal API hub — skim each shared page under _shared/ (error envelope, auth, idempotency, pagination, versioning).

After this, you should be able to answer: "where do I find the contract for endpoint X?", "what does Severity 2 mean here?", and "what is the difference between smoke and sanity testing?".

Day 2 — run

Day 2 — run the app locally

  1. Follow local development to start the API on its default port.
  2. Open Swagger UI at /openapi/index.html (served by the docs portal) and issue a GET to a read endpoint.
  3. Open the docs portal index and click around — use the search box, switch theme, open the drawer, scroll to bottom and use the back-to-top button.
  4. Run the test suite once: make test (or pytest). Note how long it takes; you'll come back to this baseline.
Day 3 — first run

Day 3 — first checklist run

Pick the smallest active checklist and run it on your local build:

  1. Open checklist 0001.
  2. Walk through every row. Take a screenshot for each section the checklist requires.
  3. Record pass / fail / needs-follow-up against each row in a notebook or ticket.
  4. Stop and ask if you find a row that is ambiguous — ambiguity in a checklist is a defect against the checklist itself.
Day 4 — first defect

Day 4 — file your first defect

  1. Find or seed a defect — if checklist 0001 came back clean, run exploratory testing for 60 minutes against the docs portal under "tab order and keyboard reachability".
  2. Reproduce the defect three times. If you cannot reproduce it, note the conditions and either keep digging or file as Severity 3 — intermittent, needs reproduction.
  3. Copy bug report template.
  4. Grade severity using the severity matrix.
  5. Attach evidence — screenshot with annotation, console log, request/response capture.
  6. File the defect; tag the area owner.
Day 5 — sign-off

Day 5 — verify a fix and shadow a sign-off

  1. When a developer marks your defect Fixed, follow the defect lifecycle to verify.
  2. Run the same reproduction on the new build. Pass → set Verified; fail → set Reopened with new evidence.
  3. Shadow a senior tester through a release sign-off using checklist 0003 so you see the cadence end-to-end.
