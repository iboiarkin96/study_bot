Overview

How to verify a docs page is correct, navigable, accessible, and consistent with the portal's design system. The line-by-line outcomes are in checklist 0001 and checklist 0005.

What to verify

What to verify on any docs page

Five concerns, in priority order: correctness (does it tell the truth), structure (does it follow the page contract), navigation (can a reader move around), themes (does it look right in light + dark), and accessibility (is it keyboard- and screen-reader-friendly).

Correctness

Correctness checks

  • Statements about endpoints, schemas, or behavior match the spec page in docs/internal/api/<resource>/ and the OpenAPI baseline.
  • Code samples compile / run as written. Copy them out and try.
  • Numbers and dates make sense (no "as of 2024" on a 2026 page).
  • Diagrams are PlantUML (per ADR 0020) and render in the lightbox without overflow.
Structure

Structural contract

Each page should have:

  • Hero section (.docs-page-hero) with overview paragraph and jump-to chips.
  • Section dividers (.docs-section-divider) between top-level groupings.
  • Cards (section.card) for each major block, each with a heading.
  • "See also" navigation (.docs-next-steps) before page history.
  • Page history table — at least one row.
  • Bug-report FAB and TOC mount in the footer area.

For API spec pages, see also the spec DoD for the full required-section list.

Navigation

Navigation checks

  • Internal sidebar shows the active page highlighted; surrounding group is open.
  • Breadcrumbs match the page's path in the docs tree.
  • Top navigation chips and "On this page" anchors all scroll to live targets.
  • Drawer mode (mobile / narrow) opens, traps focus while open, closes on Escape.
  • Back-to-top (rocket) appears after scrolling, returns to top on click.
  • External links use rel="noopener" and don't break navigation.
Search

Search

  • Search-index entry for the new page exists (rebuild the index if needed).
  • Searching for the page title returns it within the first 5 results.
  • Search results panel opens, navigates with arrow keys, closes on Escape, opens result on Enter.
  • Empty query state and no-results state both render without console errors.
Themes

Theme & responsiveness

  • Page renders correctly in light, dark, and system-default modes.
  • Theme toggle cycles system → light → dark with a tooltip and aria-label that match.
  • No element relies on a hard-coded color; tokens come through. Spot-check critical text contrast.
  • At 360px (small phone), 768px (tablet), 1280px (desktop): no horizontal scroll, no clipped text, tables scroll their own region.
A11y

Accessibility (quick pass)

  • Tab through the page — focus order matches reading order; focus rings are visible.
  • Headings are hierarchical (one h1, h2 below it, h3 inside h2).
  • Images have meaningful alt text or empty alt for decorative.
  • Diagram lightbox is reachable by keyboard and closable with Escape.
  • Use checklist 0005 for a deeper pass.
Assets

Assets & links

  • All hrefs resolve (no 404). Run a link checker if available.
  • Relative paths are correct for the page's depth (one level deeper = ../).
  • JS/CSS asset paths match — nothing in the console.
  • Inline scripts (theme bootstrap) are present and not blocked by CSP.
