Playbook: release smoke testing

Internal — QA portal · Playbooks

Overview

The minimum bar before a release tag goes out. Smoke is broad and shallow: confirm the build is alive, the golden paths still work, and the docs portal still loads. If smoke fails, the release does not ship.

When to run Preconditions Order of operations Rollback Sign-off
When

When to run smoke

  • Before tagging any release.
  • Immediately after a hotfix is deployed to staging or prod-like.
  • After any infrastructure change touching the runtime image, container, or DB.
Preconditions

Preconditions

  • CI is green on the release commit.
  • Image is built and pushed; the build SHA is known.
  • Database migrations have run (if any).
  • You have a usable bearer token and a clean test account.
Order

Order of operations

  1. Health. GET /health returns 200 with expected body. GET /ready returns 200.
  2. OpenAPI. GET /openapi.json returns the document; first 200 bytes look like valid JSON.
  3. Docs portal. Open the docs index; sidebar loads; one section page loads; search returns results.
  4. Auth golden path. An authenticated read endpoint returns the expected shape.
  5. Write golden path. Create + read-back on each top-level resource (User, Conspectus, Error log) using the API playbook.
  6. Idempotency. Repeat one POST with the same Idempotency-Key — second call returns the same response.
  7. Errors envelope. Force one validation error and one auth error; bodies conform to the envelope.
  8. Observability. Find the X-Request-ID from one earlier call in the structured log.
  9. Migrations sanity (if migrations ran). A read across the migrated table returns rows; counts look reasonable.
  10. UI smoke. Load the docs portal in light + dark, mobile + desktop. Tab through the page once.

The line-by-line pass/fail is in checklist 0003.

Rollback

If smoke fails

  1. Stop the release. Do not tag.
  2. File a defect with the failing step. Tag release blocker.
  3. If the failure is data-dangerous (writes wrong rows, exposes data), trigger the runbook for the affected area.
  4. Decide with the release owner: roll forward (fix on the branch) or roll back to the prior tag.
Sign-off

Sign-off note (paste into the release ticket)

QA smoke sign-off — release <tag>
                            - Build: <sha>
                            - Environment: <staging | prod-like>
                            - Result: GO | CONDITIONAL | NO-GO
                            - Defects opened: <count by severity, links>
                            - Notes: <optional>
                            - Evidence: <link to attached captures>
