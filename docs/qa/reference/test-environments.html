Overview

The environments QA uses, what each is for, who can change it, and what data it holds. Pick the right environment for the test type — a smoke against local won't tell you anything about staging-only configuration drift.

Summary

Environments at a glance

Env Purpose Data Reset cadence Owner
Local Developer + tester boxes; the only env where you mutate freely. Empty / fixtures. Anytime — make reset-db. Each user.
CI Automated test runs on every PR. Ephemeral fixtures. Every job. Platform.
Staging Manual smoke + acceptance against a deployed build. Synthetic seed; can be reset on request. On request, otherwise weekly. Release manager.
Prod-like Pre-release rehearsal mirroring production config. Anonymized fresh dump. Per release cycle. Release manager.
Local

Local

The default for exploratory work and any test that mutates data freely.

  • Setup: local development.
  • Database: SQLite by default; Postgres via docker-compose for closer parity.
  • OK to: drop the DB, run any migration, mutate any row.
  • Not OK to: assume timing or concurrency match production.
CI

CI

  • Runs unit + integration + contract + lint on every PR; merges blocked on red.
  • Each job is ephemeral — DB seeded, run, torn down.
  • Flakes are defects against the test, not "just retry".
  • QA does not "run on CI" — QA reads CI to confirm green before manual passes.
Staging

Staging

  • Auto-deployed from main after CI green.
  • Configuration mirrors production except for secrets and observability backends.
  • Use staging for: feature acceptance, integration with neighbouring services, release smoke.
  • Don't use staging for: load tests (use a dedicated environment per ADR 0012).
  • Data is synthetic. If you need a clean reset for a test, ask the release manager.
Prod-like

Prod-like

  • Released-tag rehearsal environment, used during the release cycle.
  • Data is an anonymized snapshot of production. Treat as "production minus PII".
  • Smoke and migration verification land here before tagging.
  • Do not run experimental scripts here. If in doubt, run on staging first.
Test data

Test data & fixtures

  • Per-test fixtures live in tests/conftest.py and adjacent modules.
  • For manual runs, use the seed CLI (see make commands).
  • Never test with real user PII even by accident — staging and prod-like are anonymized.
  • Each test creates the data it needs and cleans up; assume any other state may not be present.
Tools

Tools to keep handy

  • HTTP client — curl / httpie / Postman / Insomnia. Save collections per resource.
  • JSON viewer browser extension.
  • Annotated screenshot tool (Shotty / CleanShot / built-in macOS).
  • Log viewer — your terminal + jq; or the local Elasticsearch stack per ADR 0023.
  • Lighthouse + axe DevTools (browser extensions) for accessibility passes.
