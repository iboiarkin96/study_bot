Defect lifecycle

Internal — QA portal · Reference

Overview

The state machine every defect ticket follows. Eight states, deterministic transitions. Use the labels exactly as written so triage and reporting stay consistent.

States Diagram Transitions Who moves it Examples
States

States

State Meaning Owner
New Filed, not yet triaged. Reporter
Triaged Severity / priority / area set; assignee picked. QA / triage rotation
In progress Developer is working on the fix. Developer
Fixed Code merged on a build that QA can verify. Developer
Verified QA reproduced the original reproduction; failure no longer occurs. QA
Closed Verified and released, or accepted as intended behavior, or duplicate. QA / triage
Reopened Verification failed or the same defect returned post-release. QA
Won't fix Triage decided not to fix (deferred indefinitely or out of scope). Triage with owner

Transitions

Transition rules

  • New → Triaged. Triage sets severity, priority, area, owner. If duplicate of an existing ticket → Closed (duplicate) with link.
  • Triaged → In progress. Owner has begun coding the fix. SLA clock for "Fix by" starts here.
  • In progress → Fixed. Code merged on a build QA can pull. Comment must include the build SHA and a one-line summary of the change.
  • Fixed → Verified. QA reproduces the original repro on the new build; the failure no longer occurs. Evidence attached.
  • Fixed → Reopened. Repro still fails. New evidence attached, comment names the build that was attempted.
  • Verified → Closed. Defect is in a released build. The release tag is recorded in the ticket.
  • Closed → Reopened. Same defect returns post-release; new evidence required.
  • Triaged → Won't fix. Documented rationale; product owner accepts; ticket stays searchable.
Owner

Who moves the ticket

  • QA moves: New → Triaged, Fixed → Verified or Reopened, Verified → Closed.
  • Developer moves: Triaged → In progress, In progress → Fixed.
  • Triage rotation moves: Triaged → Won't fix (with feature owner consent).
  • Anyone may move: Closed → Reopened, with new evidence.
Examples

Worked examples

  • "POST /user accepts empty email" — New (filed) → Triaged (S2/P2) → In progress → Fixed on commit abc123 → QA reproduces on build abc123, original 201 no longer occurs → Verified → release tag v0.4.0 → Closed.
  • "Sidebar drawer focus trap missing on Safari" — New → Triaged (S2/P2) → In progress → Fixed → QA verifies, but discovers Tab still escapes after seven cycles → Reopened with new evidence → developer fixes, marks Fixed → Verified → Closed.
  • "Typo: documenttation" — New → Triaged (S4/P3) → owner fixes opportunistically with the next docs PR → Fixed → Verified → Closed.
