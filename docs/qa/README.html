QA portal

Overview

This is the home of testing for the Study App. Everything a tester needs to verify features, regress the system, document defects, and sign off a release lives here. The portal is structured so a new tester can go from "first day" to "first verified ticket" without asking anyone — and so a senior tester can find the exact template or playbook for the situation in two clicks.

The portal is organized as five layers, in increasing specificity: Foundations (why we test, the pyramid, the process), Templates (canonical artefacts to copy), Playbooks (how-to guides for a kind of test), Checklists (concrete pass/fail steps for a release or feature), and Reference (severity matrix, lifecycle, environments).

Templates

Templates

Copy-and-fill artefacts. Every test case, bug, and plan in this project should descend from one of these. If a template doesn't fit, raise it before inventing a new shape — consistency in artefacts is what lets reviewers move fast.

Template When to use
Test case template Designing a new test case (ID, preconditions, steps, expected, data, traceability).
Bug report template Filing a defect — title, environment, steps to reproduce, expected vs actual, severity, priority, evidence.
Test plan template Planning testing for a feature, release, or epic — scope, risks, entry/exit, schedule.
Playbooks

Playbooks

Step-by-step "how to test X" guides. Playbooks tell you how to think about a class of test; checklists give you the line-by-line pass/fail items.

Playbook Covers
API endpoint testing How to verify an HTTP endpoint against its internal spec — schema, errors, idempotency, auth, observability.
Documentation testing How to verify a docs page is correct, complete, navigable, and accessible.
Release smoke testing The minimal pass before a release tag is cut.
Exploratory testing Charter-based exploration to find issues that scripted tests miss.
Accessibility testing WCAG 2.1 AA checks for keyboard, contrast, focus, screen reader.
Checklists

Checklists

Numbered, line-by-line pass/fail steps. Run a checklist as written, attach evidence, and record the result in the ticket. Each checklist is owned by exactly one playbook.

Checklist Summary Status
0001 — Documentation pages visual Manual visual and interaction verification for documentation pages: navigation, sections, responsive behavior, controls. Active
0002 — API endpoint acceptance Per-endpoint acceptance: contract, success, errors, auth, idempotency, observability, evidence. Active
0003 — Release smoke Minimum bar before cutting a release tag — health, golden-path API calls, docs portal, observability. Active
0004 — Regression End-to-end regression coverage of all golden paths and the top defect-prone areas. Active
0005 — Accessibility (WCAG 2.1 AA) Keyboard, focus order, contrast, semantics, ARIA, motion safety. Active
