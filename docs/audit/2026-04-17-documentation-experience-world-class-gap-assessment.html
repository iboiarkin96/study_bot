1. Summary score

Current overall documentation experience maturity is estimated at 8.8/10 for a PET-scale service. This is a strong engineering-docs posture and above average for static docs sets.

The project is not yet at a stable top-decile level (9.2+) primarily due to feedback-loop and design-system gaps, not due to core architecture, search, or governance.

3. Critical gaps to close

3.1 Reader feedback loop is not visible enough

Search telemetry is implemented, but docs pages still lack a lightweight visible feedback control (e.g., “Was this page helpful?” and direct issue shortcut).

3.2 Design consistency is not fully systematized

Many pages follow shared styles, but section composition and visual hierarchy are still somewhat variable. A stronger page-template system is needed for repeatable UX quality.

3.3 A11y operations need KPI ownership

Accessibility checks run in CI, but there is no explicit published cadence for reviewing trends, recurring violations, and closure SLA by severity.

3.4 Golden onboarding path still needs measurable TTFS ownership

Content exists across hubs, but one strict path with explicit “you should see...” checkpoints and target time-to-first-success should be maintained as a first-class DX artifact.

4. Design maturity review

Current visual system provides a coherent baseline: shared stylesheet, top nav, in-page TOC, and consistent cards/tables. To match world-leading doc portals, the next maturity step is system-level enforcement.

Area Current state Next-level target
Page templates Mostly consistent, some drift between sections Strict templates by doc type (assessment, ADR, runbook, index, how-to)
Visual hierarchy Good core hierarchy, uneven detail-level execution Codified spacing/heading rhythm and reusable content blocks
Design governance Manual review and conventions Template conformance checks + documented design checklist in CI review flow
Feedback UX Search telemetry only Visible per-page feedback affordance and issue route

5. 90-day closure plan

Phase A (0-30 days)

  • Publish and adopt KPI policy for search quality (thresholds, warning, breach actions).
  • Add page-level feedback controls on top hubs and assessment pages.

Phase B (31-60 days)

  • Standardize doc-type templates and apply to top-traffic pages.
  • Introduce a11y trend reporting cadence and severity-based triage SLA.

Phase C (61-90 days)

  • Ship one canonical onboarding spine with explicit TTFS checkpoints.
  • Run focused design polish pass on highest-traffic 20 pages.
