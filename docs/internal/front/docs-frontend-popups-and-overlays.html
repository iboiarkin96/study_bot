Overview

This page is the source of truth for all floating surfaces in docs frontend: toasts, popups, overlays, and modal-like panels. The goal is one visual language and consistent interaction rules across all such elements.

  • Scope: visual contract, behavior contract, and implementation entry points.
  • Authoring model: this page follows the Frontend reference shape from RFC 0003 § Document taxonomy — anatomy diagram, visual gallery, behaviour cards, do/don't, page history.
  • Related pages: UI kit, navigation/search/discovery (sticky TOC widget itself), UI motion and adaptivity, tooltips and inline hints.

Stacking and lifecycle

All transient toasts mount inside one shared host — #docs-toast-stack (CSS class .docs-toast-stack) — so multiple toasts are visible at the same time. The previous behaviour (replace the existing toast or queue toasts one at a time) is retired: replacing a toast hid information the user had not yet read, queueing delayed urgent prompts until an unrelated promo finished its 3 s timer.

Host contract

  • Selector. #docs-toast-stack.docs-toast-stack — pinned position: fixed; right: 22px; bottom: 22px; with z-index: 130.
  • Layout. display: flex; flex-direction: column; gap: 12px; — newest toast is appended last, so it sits at the bottom of the stack closest to the anchor corner.
  • Pointer events. The host uses pointer-events: none; only its toast children re-enable pointer events. Empty space between toasts never blocks page clicks.
  • ARIA. The host carries role="region" and aria-label="Notifications". Each child toast carries its own role="status" (or role="alert" for errors) so screen readers announce them independently.
  • Width. width: min(420px, calc(100vw - 28px)); — same width on every breakpoint to avoid layout shift when a second toast joins.

Per-toast lifecycle

  1. Create. The runtime helper builds the toast element, sets role + aria-live, and calls getStackHost().appendChild(toast).
  2. Enter. 200 ms docs-toc-toast-enter keyframes — translate-Y 8 px → 0, scale 0.98 → 1, opacity 0 → 1.
  3. Visible. The progress bar (.docs-inpage-toc-toast__progress) animates scaleX(1) → scaleX(0) over 3 s as a non-textual countdown.
  4. Close trigger. Auto (timeout fires), manual (Hide / primary action button), or programmatic (instance.close()). Errors do not auto-close — they wait for acknowledgement.
  5. Leave. Add .docs-inpage-toc-toast--closing → 180 ms docs-toc-toast-leave keyframes → toast.remove(). Reduced-motion users skip the keyframes entirely.

Why stack and not replace

  • Replacing a toast hides messages the user has not yet processed — observed in the 2026-04 docs UX assessment.
  • Queueing single-active toasts delays urgent prompts (continue-reading) behind unrelated promos; users perceive the docs portal as “laggy”.
  • A bottom-right stack with a 12 px gap is the convention shipped by Stripe Dashboard, Linear, and Vercel — predictable for integrators reading our docs.

Cap and overflow: today there is no hard cap on the number of toasts. The toasts are short-lived (3 s informational, immediate dismiss for actionable) so practical concurrency stays at 1–3. If a future feature emits toasts at a higher rate, add a max-children cap on the host before introducing a queue.

Unified visual contract

Floating elements share one premium surface style via common CSS tokens:

  • --docs-overlay-border
  • --docs-overlay-bg
  • --docs-overlay-shadow
  • --docs-overlay-blur

Contract requirement: any new popup/overlay must use the same tokens unless there is a documented product reason to diverge.

Behavior contract

  • Toasts are non-blocking and must not trap focus.
  • Toasts stack inside .docs-toast-stack; never call remove() on a sibling toast just because a new one is being created.
  • Auto-dismiss toasts should have clear timing and reduced-motion-safe transitions.
  • Dialog-like overlays require semantic roles (role="dialog", aria-modal) and close affordances.
  • Desktop-only floating features (for example in-page TOC promo) must be disabled on tablet/phone where that UI is hidden.
  • Any action button in popup/toast must preserve existing page state and avoid destructive side effects.

Implementation entry points

  • docs/assets/docs.css — popup tokens, shared popup styling, and the .docs-toast-stack host (block ~ 2376).
  • docs/assets/docs-popups.js — centralized toast runtime helpers and window.DocsPopups.getStackHost() (Sticky TOC + Continue reading prompts).
  • docs/assets/docs-nav.js — generic enqueueDocsPromoToast + getDocsToastStackHost fallback for pages that load docs-nav.js before docs-popups.js.
  • docs/assets/docs-diagram-lightbox.js — diagram overlay behavior.
  • docs/assets/internal-sidebar.js — drawer overlay behavior for internal navigation.

QA checklist

  • All popup surfaces look visually consistent in light and dark themes.
  • No popup is clipped under sticky bars or under the internal drawer stack.
  • Desktop and tablet/phone policies are respected (including hidden desktop-only promos).
  • Keyboard navigation and focus-visible states are present and readable.

Do's and Don'ts

DoDon't
Reuse the unified surface tokens (--docs-premium-surface-*) when building a new floating surface. Re-invent the surface visuals for "just this one toast" — drift between popups is the most-flagged audit issue.
Add role="dialog" + aria-modal="true" + a focus trap for any modal overlay. Ship a modal that does not return focus to the launcher on close — keyboard and screen-reader users get stranded.
Auto-dismiss informational toasts after 3–5 s and animate respectfully under prefers-reduced-motion: reduce. Auto-dismiss error toasts — users miss them and lose data. Errors stay until acknowledged.
Hide desktop-only promos at tablet/phone widths via the documented breakpoint policy. Stack three popups on a 360 px screen — the screen becomes unusable.
Mount every new toast under .docs-toast-stack so the visual stack stays consistent and screen-reader announcements arrive in order. Append a toast directly to document.body with its own position: fixed coordinates — it will collide with the stack and break the lifecycle contract.
For an error toast, use role="alert" and skip the auto-dismiss timer; let the user acknowledge it. Auto-dismiss an error toast after 3 s — users miss the message and lose the recovery path.
