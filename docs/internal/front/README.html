Overview

This is the entry point for the docs portal frontend documentation. The portal is structured by category: foundations (cross-cutting visual rules), components (single UI parts), contracts (normative behavior), patterns (composed flows), and screens (full-page specs). Templates and the editorial workflow live in _shared/.

Categories

Each category answers a different question. The status column rolls up data-spec-status across the pages in that category; "Pages" counts only the typed specs (templates and untyped reference pages excluded).

Status counted as of 2026-05-06. To regenerate, run python3 scripts/front_spec_lint.py for the structural pass and inspect each category sub-folder for current statuses.

_shared (cross-cutting)

The _shared/ folder holds editorial rules, reference indexes, and golden templates used across every category. One source of truth — link to anchors here instead of duplicating text.

Editorial

  • Style guide — page skeleton, plain-language rules, page-history convention.
  • Maintenance process — document types, lifecycle, audit cadence, definition of done.
  • Glossary — canonical definitions for frontend UX, governance, and delivery terms.
  • Quick entry by role — onboarding map: analyst, frontend contributor, reviewer.

Spec templates (Phase 2)

Copy the matching template when adding a new page. Section markers (data-spec-section, data-spec-required) opt the page in to the front-spec linter.

  • Component spec template — for a single UI part: anatomy, states, accessibility, behaviour, do/don't, code handoff.
  • Foundation spec template — for a cross-cutting visual or runtime rule: tokens/values, usage rules, examples, code handoff.
  • Contract spec template — for a product-level normative contract: scope & breakpoints, MUST/SHOULD requirements, acceptance checklist, implementation entry-points.

Quality gates (Phase 3)

  • Definition of Done — DoD checklist per page-type (component, foundation, contract); plus the common metadata gate.
  • scripts/front_spec_lint.py — structural lint reading data-spec-section markers; wired into make docs-spec-check. Pre-template legacy pages are skipped.

Foundations

Cross-cutting visual and runtime rules. These pages do not describe a single component — they define the primitives that every component reuses.

  • Token gallery — auto-generated catalogue of CSS custom properties (light + dark).
  • UI kit — components vocabulary, badges, status pills, density presets.
  • UI, motion, and adaptivity — breakpoints, motion policy, mobile contract.
  • CSS architecture — file boundaries, naming, layer order.
  • JS modules reference — runtime entry points, ownership, lifecycle.
  • Hotkeys — keyboard shortcut catalogue.

Components

One page per UI part. Each is the source of truth for that component's anatomy, states, accessibility, and code handoff.

  • Tooltips and inline hints — Pattern A vs B; clipping rules.
  • Popups and overlays — anatomy, dismiss policy, focus.
  • Diagrams and lightbox — expand control, zoom/pan.
  • Sticky on-this-page TOC — desktop aside contract.
  • Resume reading and back-to-top — long-page reading aids.
  • Rocket launch animation — the back-to-top FAB's transient state.
  • Report-bug FAB — page feedback beacon.

Contracts

Normative behavior contracts for cross-cutting product surfaces. These pages describe what the portal must do at the product level; implementation lives in foundations/components.

  • Menu and theme — drawer / top-nav split, theme cycle, toast.
  • Navigation, search, and discovery — sidebar, breadcrumbs, search panel, command palette.
  • Feedback and editorial workflow — page feedback form, cadence, edited-by widget.

Patterns

Composed flows where multiple components participate in a single user-visible behaviour. Patterns name the participating components, the reader's journey, and the acceptance checklist that proves the flow holds together.

  • Long-form reading aids — composes the resume banner, back-to-top FAB, and rocket launch animation for pages that run past two viewport heights.

Screens

End-to-end spec for a single docs screen. Covers anatomy, visual gallery (light + dark + breakpoints), states, behaviour, and code handoff. Use the template for any new screen.

  • Screen spec template — copy this for new screens.
  • Home landing — rocket intro, hero, parallax.
  • Backlog cockpit — filters, board, list, timeline.

Architecture

  • Architecture map — feature ledger (DFEAT-xxx), runtime ownership, full implemented-feature inventory.
