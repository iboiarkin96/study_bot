Overview

Lookup table for every stylesheet under docs/assets/*.css: file responsibility, what loads it, its @import children, the main class families it ships, and the dark-mode override pattern it uses. Use this page when you need to answer "which file owns this style rule?" or "is it safe to add a new token to docs.css, or does it belong in a sub-layer?".

Sister pages: JavaScript modules reference, architecture map, UI / motion / adaptivity (token rationale).

@import chain

Every page loads two CSS files via direct <link> tags: docs.css and docs-theme.css. Everything else is pulled in transitively through @import declarations at the top of docs.css. The landing page additionally loads home.css. 

docs.css                       (entry point, loaded by every page)
            ├── @import → Google Fonts (Inter)
            ├── @import → docs-site-nav.css
            ├── @import → internal-layout.css
            ├── @import → docs-premium.css
            └── @import → docs-syntax-theme.css

            docs-theme.css                 (loaded separately via <link>, dark palette)
            home.css                       (only on docs/index.html)

Cascade order: Google Fonts → site-nav → internal-layout → premium → syntax-theme → (rest of docs.css) → docs-theme.css. Later rules override earlier ones, so the theme overrides everything; the syntax-theme overrides premium; premium overrides internal-layout; and internal-layout overrides site-nav. Reordering imports breaks specificity expectations — leave the order alone unless you are intentionally renegotiating the cascade.

docs-theme.css — dark-palette overrides

Loaded by
Every docs/**/*.html page via direct <link> tag (separately from docs.css so it sits at the very end of the cascade).
Pattern
One large @media (prefers-color-scheme: dark) block at :root redefines every visible variable from docs.css. Legacy :root[data-theme="dark"] rules at the top are kept for the explicit-toggle path used by the theme switcher.
What it overrides
~100 custom properties: surfaces, badges, navigation, internal layout, ADR/audit/status palettes.
Sample dark token values
--bg: #0d1117, --card: #161b22, --accent: #58a6ff.

internal-layout.css — deep dive

internal-layout.css owns the chrome that wraps every page under docs/internal/**: sidebar, drawer, sticky page title, collapsed-sidebar mode, and portal grids. It is the most layout-sensitive file in the portal — small changes here ripple through every internal page.

Sidebar component breakpoints

Component-level, not device tiers. The internal-layout sidebar reflows at its own cut-offs because the wide rail no longer fits — these numbers are not the canonical device tiers. Whenever you describe a device class (mobile / tablet / desktop) in any other doc, use the canonical viewport breakpoints (≥1025 / 761–1024 / ≤760).

Drawer state machine

  • Trigger: viewport ≤1024 px and shell carries .is-drawer-mode.
  • Open class: .internal-layout__drawer.is-open → display: block.
  • Closed: .internal-layout__drawer[hidden] → display: none !important (HTML attribute, set by internal-sidebar.js).
  • Body lock: body.internal-sidebar-drawer-open → overflow: hidden (prevents page scroll while drawer is open).
  • Drawer-only cut-off (<900 px): sidebar moves below page title row in document order (no longer sticky). Component-level cut-off; spans the bottom of the tablet tier and the entire mobile tier per the canonical viewport breakpoints.

Collapsed sidebar

  • Toggle: button rendered by internal-sidebar.js with data-internal-sidebar-toggle; only visible on desktop.
  • Class added: .internal-layout__shell.is-sidebar-collapsed → grid-template-columns: 56px 1fr.
  • Sidebar title rotates (CSS transform); toggle button shrinks to icon-only.
  • Persisted via localStorage key docs.internal.sidebar.collapsed.

Stylesheet conventions

Where does a new style belong?

  • Brand-new visual variable (color, radius, spacing, motion) → :root block in docs.css, then a matching dark override in docs-theme.css.
  • New component class shared across page types → docs-premium.css. Keeps docs.css focused on tokens and globals.
  • New layout primitive for internal pages (shell modifier, sticky region, drawer behavior) → internal-layout.css. Update the deep dive in this page in the same PR.
  • Top-nav widget → docs-site-nav.css. Do not stash nav rules in docs.css.
  • Landing-page-only styles → home.css. Anything that shows up only on docs/index.html must live here, not in docs.css.
  • Code-syntax token → docs-syntax-theme.css. Pair with a token-emitting rule in docs-syntax.js (currently dormant).

Dark-mode override pattern

Two mechanisms work together:

  1. Variable redefinition: docs-theme.css redefines tokens at :root under @media (prefers-color-scheme: dark). Most components inherit the new palette automatically because they read var(--bg), var(--card), etc.
  2. Explicit selector overrides: where a component needs a different look in dark (not just different colors), docs.css ships :root[data-theme="dark"] .my-component { … } blocks colocated with the light rules.

The explicit toggle path is required because the theme switcher writes data-theme="dark" on <html> independently of OS preference. Both code paths must visually agree.

Naming

  • Use kebab-case BEM for layout/component classes: .docs-callout, .docs-callout--warn, .internal-layout__sidebar.
  • Prefix all global tokens with -- and a domain word: --docs-premium-*, --internal-*, --st-*. Generic tokens (--bg, --accent, --line) stay un-prefixed but are scarce by design.
  • Avoid one-off ID selectors. The mount points listed in the JS reference are the only DOM IDs styled.
