Docs frontend UI kit (component style spec)

Overview

This page is retained as a component style specification for premium docs frontend surfaces. It focuses on visual standards only and avoids runtime behavior duplication.

  • Scope: style system for toolbar, command palette, hotkey tip, continue-reading card, and page feedback card.
  • Behavior docs: see UI system, motion, and adaptivity and navigation/search/discovery.
  • Inline hints: Tooltips and data-tooltip contract (normative for hand-authored pages).
  • Design intent: compact, modern, low-noise premium UI with strong accessibility support.

Visual principles

  • Premium surface: layered gradient plus subtle inner highlight, never flat gray blocks.
  • Low-noise density: compact spacing and restrained shadows to keep focus on content.
  • Modern corners: soft rounded rectangles (not old pill-heavy controls by default).
  • Predictable states: hover, focus, and active use one coherent accent language.
  • Theme parity: dark mode uses dedicated tokens, not naive inversion.

Premium UI surfaces must use shared token variables from :root and :root[data-theme="dark"], not hard-coded per-component colors.

Interaction styling contract

  • State consistency: hover/focus/active must stay visually coherent across all premium controls.
  • Focus visibility: controls must expose visible :focus-visible in both light and dark themes.
  • Motion policy: micro-interactions use short transitions; reduced-motion mode disables non-essential effects.
  • Density policy: compact spacing should not reduce readability of labels, hints, and keycaps.

Accessibility requirements

  • Palette panel must preserve dialog semantics: role="dialog", aria-modal, and labeled title.
  • Interactive controls must have visible :focus-visible states and sufficient contrast in both themes.
  • Feedback and toast surfaces must use live-region semantics where status updates are dynamic.
  • Keyboard shortcuts should not hijack typing behavior in form fields.
  • Reduced-motion mode must suppress non-essential transforms and animations.

Implementation map

  • docs/assets/docs.css — token definitions, theme overrides, component visuals, and state styling.
  • docs/assets/docs-theme.css — dark-theme token overrides and parity rules.
  • docs/assets/docs-site-nav.css — top navigation and flashlight control styling.
  • docs/internal/front/docs-frontend-ui-motion-and-adaptivity.html — runtime and UX behavior details.

QA checklist

  1. Verify all premium surfaces in both light and dark themes (toolbar, palette, tip, continue-reading, feedback).
  2. Run keyboard-only pass: focus rings are visible and readable on all premium controls.
  3. Validate focus-visible rings and glow contrast on interactive controls.
  4. Check reduced-motion mode to ensure animations are disabled or near-instant.
  5. Check responsive behavior at desktop, tablet, and phone breakpoints.
  6. Run make docs-html-check after UI changes.

Change management rules

  • When adding a new premium docs component, define tokens first, then implement styles.
  • Avoid introducing one-off colors, borders, or shadows in component rules without token coverage.
  • Any visual update in one theme must be reviewed for parity in the other theme.
  • Keep runtime behavior docs in dedicated pages to prevent duplication in this style specification.
