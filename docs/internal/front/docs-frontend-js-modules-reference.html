Docs frontend JavaScript modules Reference

Docs portal · JavaScript runtime

Summary Roles, exports, contracts, and storage keys for every JS module. Status Availability v1 · Internal Owner Ivan Boyarkin Last reviewed 2026-04-30 Spec template v1

Overview

Lookup table for every JavaScript module under docs/assets/*.js: file role, where it is loaded, what it exports, what data-* contracts it consumes, what custom events it dispatches, and which storage keys it owns. Use this page when you need to answer "what runtime owns this behavior?" or "is it safe to rename this attribute?".

For runtime block diagram and ownership zones, see the architecture map. For UI-level component descriptions, see the UI kit. For CSS layer organization, see the CSS architecture reference.

At-a-glance summary

All sizes are source-file bytes (not minified, no transpile step in this project). "Loaded?" reflects whether the file is reachable from a normally-loaded HTML page; "—" means dynamic-load by another module rather than a direct <script> tag.

docs-syntax.js — syntax highlighter (currently unwired)

Status: file exists in docs/assets/ but is not loaded by any HTML page. The Phase 1 portal redesign created it together with docs-syntax-theme.css intending to wire it into the six hub pages (adr/, developer/, howto/, runbooks/, qa/, rfc/ READMEs). docs-resume.js — long-page reading aids

Wrapped in IIFE
Yes
Init trigger
DOMContentLoaded
Loaded by
Pages that opt-in via direct <script defer src="…/assets/docs-resume.js">. Currently used on long internal API operation pages (e.g. conspectus/operations/post-…actions-review.html). Gracefully no-ops if the page has no scrollable content.
Behavior
  • Resume reading banner — if the reader scrolled past 25 % of the page on a previous visit (within 14 days), shows a top banner with a one-click "jump to where I left off" action.
  • Back to top button — floating button appears after 600 px of scroll; smooth-scrolls to top on click.
Storage keys owned
localStorage: docs.resume.<path> — JSON { scrollY, ts }, TTL 14 days.
sessionStorage: docs.resume.seen.<path> — "1" when the banner was acted on or dismissed in this tab.
Tuning constants
TTL_MS = 14 days, MIN_SCROLL_FRACTION = 0.25, BACK_TO_TOP_THRESHOLD = 600 px, SAVE_THROTTLE_MS = 750 ms.
CSS dependency
Visual rules live in internal-layout.css (banner and floating button). See internal-layout deep dive.
Status
Recently added (2026-04-29) — does not yet have a dedicated component spec page. Backlog: write a Component-type spec under docs/internal/front/ covering banner anatomy, button states, opt-in pattern, and the long-page acceptance criteria.

Naming and ownership conventions

  • One module per concern. If a behavior crosses two ownership zones from the architecture map, split it.
  • IIFE by default. Modules wrap their internal helpers in (function(){…})(); and only expose what is needed via window.X.
  • Prefix all globals with window.__DOCS_ or window.Docs. Anything else collides with page-level scripts.
  • Custom events use a namespace. Examples: docs-search-telemetry, internal-sidebar:drawer-state, home:intro-done. New events must use one of: docs-*, internal-sidebar:*, home:*.
  • data-* attributes are a contract. Renaming an attribute is a breaking change for every HTML page that consumes it. Update the consumer module, every HTML page, and this reference in the same PR.
  • Storage keys are versioned where state can drift. Add -v1, -v2 suffixes when format changes; never silently rewrite existing payloads.
  • Mount discovery is permissive. Modules tolerate missing mount points — pages that do not need a feature simply omit the mount, and the module exits silently.
