Overview

The sticky TOC is the right-side "On this page" rail that appears next to the article on tablet-wide and desktop viewports. It is built at runtime from the article's h2/h3 headings — authors don't write it by hand. As the reader scrolls, the entry matching the section currently at the top of the viewport is highlighted (scroll-spy). When the article has more headings than fit inside the rail's max-height, the rail's inner <nav> becomes its own scroll container and the active link is kept in view without ever moving the page.

This page is the UI spec for the rail and its scroll-spy. The collapse-toggle (the < / > chevron in the rail header) and the one-time "Sticky TOC available" promo toast are still documented in Navigation, search, discovery §On this page collapse control and Popups and overlays respectively.

Source-of-truth for the contract is this page; source-of-truth for the code is docs/assets/docs-nav.js (functions initAutoInPageToc(), initInPageTocScrollSpy(), ensureTocAnchorIds(), collectTocEntries()) and docs/assets/docs.css (selectors prefixed .docs-inpage-toc* and .docs-page-layout__inner*).

Goals and non-goals

GoalsNon-goals
Always show the reader where they are in the article without forcing them to scroll the rail. Replicate the left sidebar tree — the TOC is page-scoped, not site-scoped.
Build itself from the actual content (h2/h3) so docs authors don't drift. Index every heading level — h4+ are deliberately excluded to keep the rail scannable.
Keep window scroll under the reader's control: the rail's auto-scroll only moves its own overflow, never the page. Smoothly animate nav.scrollTop while the user is mid-flick — that would fight the reader's input.
Survive page-level DOM rewrites (e.g., page-history fold, lifecycle status injection) without pinning the active state to a stale entry. Re-build the TOC on every layout change. Targets are captured once; the spy is defensive but the list is stable.
Hide entirely on phone widths so the article gets the full reading column. Replace the sticky rail with a drawer/sheet on phone — that role belongs to the in-page hero chips.

Anatomy

  1. Aside container — aside.docs-inpage-toc in column 1 of .docs-page-layout__inner (article is column 2 with order: 2, so DOM order is article → aside but visual order is aside → article). Sticky via position: sticky; top: 1rem;.
  2. Header row — .docs-inpage-toc__head with the title ("On this page") and a chevron .docs-inpage-toc__toggle. The entire header is clickable (collapse/expand); the chevron is a decorative SVG inside the button.
  3. Inner scroll container — .docs-inpage-toc nav with max-height: min(75vh, 36rem); overflow-y: auto; overscroll-behavior: contain;. This is the container the auto-scroll moves; the page never moves because of the spy.
  4. Entry list — flat <ul> of li > a. h3 entries get .docs-inpage-toc__item--nested for the indent.
  5. Active link — .docs-inpage-toc__link--active sets the accent left border, accent text colour, and a subtle background. The same anchor also gets aria-current="location".
  6. Heading anchor ids — every h2/h3 gets a stable id assigned by ensureTocAnchorIds() if the author didn't write one. Slugs are based on the heading text and de-duplicated against ids already present on the page (this matters — see Failure modes).
  7. Promo toast — one-shot "Sticky TOC available" toast on first visit to a wide-desktop page where the TOC starts collapsed. Lives in docs-popups.js; see Popups and overlays.

Behaviour contract

The scroll-spy in initInPageTocScrollSpy() runs on every scroll, resize, and once on window.load. Each run it computes a single active target against this contract:

RuleWhy
Activation line at 96 px from viewport top. The active entry is the last one whose h2/h3 top is ≤ 96 px. Picks the section the reader is "in" rather than the one they scrolled past five lines ago. 96 px clears the sticky top-bar zone.
Default before first crossing = first entry. When no heading has crossed the line yet (reader is at the top of the page), entry #0 is active. Otherwise the rail would render with no active highlight at the canonical reading position (top of page).
Near-bottom safety net is gated on scrollY > 0. The "you've reached the end → activate last entry" branch only fires after the reader actually scrolls. At scrollY = 0 a short page (or a long page with assets still loading) trivially satisfies scrollBottom ≥ docBottom − 8; without the gate the last entry is incorrectly pinned on initial load. See Failure modes #2.
Detached targets are filtered out per tick. updateActive() recomputes liveTargets = targets.filter(t => t.el.isConnected) before measuring. Detached elements return top: 0 from getBoundingClientRect(), which would always satisfy top ≤ 96 and pin that entry forever. See Failure modes #1.
Auto-scroll inside the rail keeps the active link visible. When nav.scrollHeight > nav.clientHeight, keepActiveLinkVisibleInNav() nudges nav.scrollTop just enough to land the active <a> within a 24 px margin from the top or bottom of the visible band. On long articles the TOC overflows its max-height: min(75vh, 36rem). Without this the active highlight scrolls out of sight and the rail looks broken.
Window scroll is never moved. The auto-scroll only writes to nav.scrollTop; it never calls scrollIntoView() on the link. scrollIntoView() walks every scrollable ancestor including the document. Moving the page in response to scroll-spy would feel like the browser fighting the user.
Updates are rAF-throttled. Multiple scroll events between two animation frames coalesce into one measurement. getBoundingClientRect() in a tight scroll loop is the canonical reflow trap.
Page-history section is excluded from the TOC. normalizeDocsPageHistory() runs before initAutoInPageToc(), removing the <h2>Page history</h2> (it becomes the <summary> of a fold) before the TOC sees it. If the TOC captured that h2 first, the later normalization would detach it. See Failure modes #1 for the historical bug.

States

StateTriggerEffect
Idle No scroll, no hover. Active entry from the contract above; other entries muted (color: color-mix(...var(--text)...var(--muted))).
Hover Pointer over a non-active entry. Accent text + accent left border + soft --docs-premium-item-hover-bg background; transform: translateX(1px).
Active Section h2/h3 top crossed the activation line and is the latest such crossing. .docs-inpage-toc__link--active: accent colour, accent left border, accent-tinted background, font-weight: 600, aria-current="location".
Active & off-screen inside rail TOC overflows its max-height; active link is above or below the visible band by more than 24 px. nav.scrollTop nudged so the active link sits exactly 24 px from the top (when above) or 24 px from the bottom (when below).
Collapsed rail Reader clicked < in the rail header (or wide-desktop default on long pages). .docs-inpage-toc--collapsed hides the nav; the chevron flips to >; the spy keeps running but visual state is invisible until expanded.
Phone max-width: 900 px. Aside is display: none; the page-layout grid collapses to a single column; the spy still mounts but does no visible work.

Implementation

Init order in document.addEventListener("DOMContentLoaded", …) (docs/assets/docs-nav.js):

  1. normalizeDocsPageHistory() — folds the page-history section into a <details> and removes its <h2>. Must run before the TOC is built.
  2. initAutoInPageToc() — wraps content in .docs-page-layout__inner, calls ensureTocAnchorIds(), builds the rail from collectTocEntries(), attaches the collapse toggle and the (optional) promo toast.
  3. restoreInitialHashPosition() — re-runs scrollIntoView() on the URL hash target after layout transforms.
  4. initInPageTocScrollSpy() — captures targets via document.getElementById, registers scroll + resize + one-shot load listeners, runs updateActive() once.

Accessibility

  • Container is <aside> with aria-labelledby="inpage-toc-heading"; the heading paragraph has a unique id.
  • The inner <nav> is also labelled by the same heading id.
  • Active link gets aria-current="location"; the class is purely visual.
  • Collapse toggle exposes aria-expanded, aria-controls, and a label that toggles between "Hide On this page" and "Show On this page".
  • Auto-scroll only writes scrollTop on the nav — focus is never moved as a side effect.
  • prefers-reduced-motion: reduce is honoured by upstream CSS (html { scroll-behavior: smooth } is overridden to auto); the spy itself does instant updates.

Failure modes

  1. Detached active target pinned forever. If a later DOM step removes a heading the spy already captured, getBoundingClientRect() on the now-detached node returns top: 0, which always satisfies the ≤ 96 px rule and pins that entry as active.
    Guard: updateActive() filters targets by el.isConnected every tick. Root-cause fix: normalizeDocsPageHistory() runs before initAutoInPageToc() so its <h2> never enters the TOC.
  2. Last entry pinned at scrollY = 0 on short or slow-loading pages. The "you scrolled to the end" branch incorrectly fires because innerHeight ≥ scrollHeight when assets (e.g., PlantUML images) haven't grown the document yet.
    Guard: nearBottom requires scrollY > 0; a one-shot window.load listener re-runs the spy after subresources settle.
  3. Heading-id collision with section id. If the author writes <section id="page-history"><h2>Page history</h2>, ensureTocAnchorIds() assigns the h2 a de-duplicated id (e.g., page-history-2) and the TOC links there. Anything that later removes that h2 orphans the link.
    Guard: avoid putting id on a wrapping <section> that has the same slug as its inner heading. The page-history fold sidesteps this by removing the h2 before the TOC sees it.
  4. Author writes a manual "Contents" block. removeLegacyManualContents() scrubs <nav> elements that contain a "Contents" heading; if a custom variant slips through, the page renders both the manual and the auto rail. Mitigation: rely on the auto-TOC; do not write inline contents.

Acceptance checklist

  • Rail renders to the right of the article on viewports > 900 px; hidden at ≤ 900 px.
  • At scrollY = 0 the first entry is active; "Page history" is not in the list.
  • Scrolling past each h2 moves the active highlight to that entry within one frame.
  • Scrolling the very last h2 past the activation line activates the last TOC entry; further scroll to the document end keeps the last entry active.
  • On a TOC longer than the rail's max-height, the active link is always inside the visible band — never clipped above or below — and the page itself does not move when the rail auto-scrolls.
  • Collapse chevron toggles the rail; the spy keeps tracking; expanding restores the active highlight without flicker.
  • No regressions on the related fall-back paths: pages with no h2/h3 render as .docs-page-layout__inner--single (full-width article, no aside).
