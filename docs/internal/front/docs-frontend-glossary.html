Docs frontend glossary Foundation

Overview

This glossary defines frontend documentation terms used across the docs platform so contributors, analysts, and reviewers share the same language when discussing behavior and changes.

Core terms

Term Definition Primary references
Architecture map The canonical inventory of frontend features, ownership boundaries, and implementation ledger entries. Architecture map
Feature ID (DFEAT-xxx) Stable identifier for a frontend capability used in tasks, QA notes, changelog entries, and reviews. Feature ID governance
Internal sidebar The left navigation tree for internal docs pages, including nested groups and active-page resolution. Internal sidebar and drawer
Hint tooltip (data-tooltip) Compact, no-underline hint on controls and table headers; shared styles in docs.css. Distinct from dotted glossary .docs-tooltip. Tooltips and inline hints
Drawer mode Tablet/phone navigation model where sidebar content opens as a modal-style left panel. Navigation contract
Flashlight control The theme toggle button that cycles automatic/light/dark modes and shows a lit state in dark mode. Theme system and flashlight control
Quick actions (command palette) Desktop-only action dialog opened by keyboard shortcuts for navigation, actions, and section jumps. Command palette
Page actions toolbar Desktop toolbar with high-frequency page-level actions (copy link, edit/report, reading mode, theme). Feature ledger
In-page TOC Automatically generated section navigation from headings (h2/h3) with active-section tracking. In-page navigation
Reading mode A focused content mode that minimizes surrounding UI noise for long-form reading. Reading UX helpers
Continue reading prompt A persisted prompt that restores a previous section anchor and scroll position for returning readers. Continue reading details
Rocket back-to-top A floating control that appears near the page bottom and scrolls to top with launch animation. Rocket and reading UX
Premium surface Shared visual styling model based on --docs-premium-* tokens (cards, chips, list items, focus). UI kit
Edited by widget Page-level metadata control that displays maintainer avatars and an expandable editor list. Edited by workflow
Page feedback card Embedded page form that creates prefilled GitHub feedback issues with context and category. Feedback form workflow
Telemetry contract Agreed event channels and payload expectations for search and command palette interactions. Telemetry contract
Diagram lightbox Full-screen diagram viewer with zoom, pan, keyboard controls, and focus-safe open/close behavior. Diagrams and lightbox
Page-type badge Small inline label placed next to every page <h1> declaring one of six page types: Foundation, Architecture, Screen, Component, Reference, Contract. Drives size budget and required sections. Page-type taxonomy
Reference page Page type whose body is dominated by a normalized lookup table; minimal narrative. Examples: JS modules reference, CSS architecture, hotkeys. Page-type taxonomy
Contract page Page type that holds only stable normative requirements; implementation lives elsewhere. Size budget 80–200 lines. Navigation and theme contract
Public surface What a JavaScript module exposes to the rest of the runtime: CustomEvents emitted/listened, data-* attributes consumed/set, window globals, storage keys, mount-point IDs. Renaming any of these is a breaking change. JavaScript modules reference
data-* contract The set of data--prefixed HTML attributes a runtime module reads or writes. Acts as the bridge between markup pages and JS behavior. Naming and ownership conventions

Home screen terms (engineering hub landing)

Terms specific to the docs portal home screen (docs/index.html) and its boot intro overlay. Each term has a short, plain-language definition first, then a precise implementation note. If you don't know a term while reading code or specs, look it up here before guessing.

Term Plain-language definition Implementation note Reference
Hero The large, attention-grabbing block at the top of a landing page. Holds the title, tagline, and primary CTAs. .home-hero in home.css; isolated stacking context with background layers behind and content grid in front. Home spec — Hero
Eyebrow A short uppercase label sitting above a heading, used to label the section ("Engineering Documentation"). .home-hero__eyebrow; small, accent-colored, with a leading 18×2px accent bar. Home spec — Eyebrow
Tagline One-line subtitle below the H1 that summarizes the page's purpose. .home-hero__tagline; muted color, max-width 58 characters. Home spec — Tagline
Wordmark A textual logo: the product name set in a distinctive type style, treated as a brand mark. The home boot intro renders ETR // API as a wordmark with per-character spans for the scramble effect. Home spec — Wordmark scramble
Glyph A single visual character — a letter, digit, symbol, or pictogram. Used in this codebase to mean "one drawn character" (e.g. one cell of the rain canvas, one slot of the wordmark). —
Glyph rain (Matrix rain) A background of falling random characters in vertical streams, styled after the "code rain" from The Matrix. Canvas-based; columns of glyphs (Katakana + ASCII) fall at random speeds with bright head + faded trail. Implemented in startGlyphRain(). Home spec — Glyph rain
Glyph scramble (decrypt effect) A reveal animation where a piece of text first cycles through random characters and then "settles" into the real text, character by character. Used in two places: the intro wordmark (startWordmarkScramble) and the H1 (bindHeroDecrypt). Each char rolls random glyphs every ~50ms, then locks at a per-char timestamp. Home spec — Wordmark scramble
Lock-in / lock-flash The moment when a scrambling character settles to its final letter, accompanied by a brief brightness flash and scale pulse. CSS animation ig-lockflash (scale 1.18 → 0.98 → 1, 380ms). Triggered by adding .ig-char--locked. Home spec — Wordmark scramble
HUD bracket An L-shaped corner mark used in heads-up displays (HUD) to frame an object of interest, like the targeting reticle in a video game. Four .home-intro__bracket elements at corners of .home-intro__stage; idle pulse + one-shot snap on full lock. Home spec — HUD brackets
Crosshair tick A short vertical line above and below an object, marking its center axis — like the tick marks of a sniper scope. .home-intro__crosshair--top/bot at the vertical center of the HUD stage. Home spec — HUD brackets
Radar pulse ring (sonar ring) A circle that grows from a center point and fades out, mimicking a radar or sonar ping. Three .home-intro__radar elements expand from scale(0.5) to scale(7.2) over 3.6s, staggered 0/1.2/2.4s. Home spec — Radar rings
Vignette A dark border-fade around the edges of an image or screen that pulls the eye toward the center. .home-intro__vignette — radial gradient that darkens the outer 35% of the overlay. Home spec — Boot intro
Scanlines Faint horizontal stripes overlaid on a screen, evoking a CRT television or technical instrument display. .home-intro__scanlines — 1px-on / 3px-off repeating linear gradient at low opacity. Home spec — Boot intro
Iris shutter A circular wipe that closes (or opens) like a camera aperture, used here to dismiss the intro overlay. runShutterExit() animates a clip-path: circle(150% → 0%) over 720ms with cubic-bezier(0.87, 0, 0.13, 1). Home spec — Iris shutter
Mask reveal An animation that reveals content by uncovering it from behind a moving mask — a wipe, a curtain, or a clip-path. Pattern used in earlier intro iterations; current intro uses iris shutter on exit and per-character scramble on entry. —
Skip button (skip control) A button that lets the user dismiss an intro/animation early instead of waiting it out. .home-intro__skip; absolute top-right; Esc bound as keyboard equivalent. Home spec — Skip
Status pill A small rounded label showing the current state of a system. Often paired with a colored dot. .home-intro__status — monospace text with a pulsing cyan dot; transitions to steady white on .is-locked. Home spec — Status pill
Progress bar A horizontal bar that fills from 0% to 100% to show how much of a process is complete. Two on this page: (a) .home-intro__bar tied to wordmark glyph lock count; (b) .home-scroll-progress tied to scroll position via scaleX. Home spec — Progress bar
Scroll progress bar A thin bar fixed at the top of the viewport that fills as the user scrolls down the page. .home-scroll-progress; updated each scroll/resize via rAF in bindScrollProgress(). Home spec — Scroll progress
CTA (Call To Action) A clickable button or link inviting the user to take a primary action ("Start", "Sign up", "Open the docs"). .home-hero__cta; primary uses gradient fill + magnetic effect, secondaries use card border. Home spec — CTAs
Magnetic CTA A button that subtly slides toward the cursor when the cursor is near, as if attracted to it. Opt-in via data-home-magnetic; pointer offset is mapped to a damped translate, clamped to ±9/±7px. See bindMagneticCta(). Home spec — Magnetic CTA
Mono-chip A small rounded pill of monospace text used to display short metadata (a version, an environment, a tag). .home-chip; backdrop-blurred 6px, monospace 0.7rem, optional leading status dot. Home spec — Mono-chips
Number ticker (count-up) A number that animates from zero up to a target value when it scrolls into view. .home-ticker__num with data-home-ticker-target + optional data-home-ticker-suffix; cubic-out easing over 900ms; stagger 80ms; uses tabular-nums. Home spec — Tickers
Terminal panel (live code panel) A card styled like a terminal window (with traffic-light dots and monospace text) that shows a code or shell example. .home-hero__terminal; macOS chrome, syntax-colored typing of TERMINAL_SCRIPT via bindLiveTerminal(); gated on intro-done + visibility. Home spec — Terminal
Caret (typing cursor) The blinking vertical bar shown at the current insertion point in a text editor or terminal. .home-terminal__caret 8×1em block; blinks via @keyframes cursor-blink at 720ms steps(2, end). —
Cursor-reactive dot grid A regular grid of small dots that brightens and grows where the cursor is hovering, as if the cursor lights up nearby dots. Canvas-based; spacing 26px, react radius 140px. Implemented in bindDotGrid(); rAF-throttled, theme-aware. Home spec — Hero background
Parallax orb A blurred glowing disc placed in the background that drifts slightly when the user moves the cursor, creating a sense of depth. .home-hero__orb--one/two/three; depth multipliers 0.28 / 0.18 / 0.12. Driven by bindHeroParallax() with damping 0.12. Home spec — Hero background
Section tint shift A subtle change in the background color tone between scroll sections, used to give each section a distinct mood. [data-section-tint="cool|teal|mono"]; .is-tinted applied via IntersectionObserver; tint rendered in ::before as a radial gradient. Home spec — Section tints
Card stagger reveal A reveal animation where cards in a row fade/slide in one after another with a small time delay, instead of all at once. bindRevealOnScroll() sets transition-delay: i * 65ms per card and adds .is-visible in a double-rAF. Home spec — Card sections
Shimmer sweep A diagonal translucent band that sweeps across a card on hover, like light catching the surface. .home-card::after; translateX(-110% → 110%) over 620ms. Home spec — Card hover
3D card tilt A small 3D rotation applied to a card on hover so it feels like it's leaning toward the cursor. .home-card:hover applies translateY(-3px) rotateX(1.2deg) rotateY(-1.4deg) (motion-allowed only). —
Variable font (wght axis) A font file that can interpolate between many weights/widths along an axis, instead of shipping discrete weight files. font-variation-settings: "wght" var(--home-h1-wght) on .home-hero__title; --home-h1-wght animated 500 → 760 on scroll. Home spec — H1
View Transitions API A browser API that animates the change between two states — including the transition between two pages — automatically. Currently Chromium-only. document.startViewTransition() in bindViewTransitions(); per-CTA view-transition-name via CSS for shared-element morphs. Home spec — CTAs
Scan beam A bright bar that sweeps once across an element, like a barcode scanner reading the wordmark. .home-intro__wordmark-beam; one-shot animation ig-beam on .home-intro__wordmark.is-locked. Home spec — Wordmark
Reduced motion A user preference (operating system setting) that asks websites to minimize non-essential animations. Respecting it is an accessibility requirement. @media (prefers-reduced-motion: reduce) gate; everything decorative is disabled or simplified, intro is removed, terminal renders statically. Home spec — Accessibility
Anatomy diagram A labelled illustration that names every visible region of a screen or component, like a wiring diagram for the UI. Annotated SVG embedded in screen specs (preferred over screenshots — version-controllable, doesn't go stale on style tweaks). Home spec — Anatomy
Screen spec A single-page document that fully describes one screen of the product: layout, behavior, data hooks, edge cases, and where to change it. Lives in docs/internal/front/screens/docs-screen-*.html. Always start from the template. Screen spec template

How to use this glossary

  • While reading code: if a CSS class or function name uses an unfamiliar word ("orb", "wordmark", "shimmer"), find it here first. The implementation note tells you exactly which selector or function to look at.
  • While writing PR descriptions: use these terms verbatim. "Adjusted radar pulse timing" beats "made the circle thing slower" — reviewers know exactly what changed.
  • While writing a new screen spec: use the same term across the spec, glossary, and code. Add new terms here in the same PR that introduces them in the screen.
  • If a term feels ambiguous: open an issue or fix it directly — glossary precision is the foundation of every other doc.

Usage guidelines

  • Use glossary terms in issue titles, task descriptions, and PR notes to reduce ambiguity.
  • When introducing a new frontend concept, add it here in the same change set as feature documentation updates.
  • When a term changes meaning, update this page and cross-references in related docs pages.
