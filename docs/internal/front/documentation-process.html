Overview

This page describes how we keep documentation alive in this repository. It is the lifecycle companion of the documentation style guide (which covers how words look): this page covers how docs are produced, updated, audited, and retired over time.

The process is modelled on patterns used by widely-respected engineering documentation programs — Stripe, Linear, Vercel, GitHub, Atlassian, and Storybook — adapted to the constraints of a small team shipping a static docs/ tree alongside the application code.

  • Goal: every screen, ADR, runbook, and API page is the single source of truth for its topic, and stays correct as the code evolves.
  • Scope: all hand-authored HTML pages under docs/, plus the screen-spec corpus under docs/internal/front/screens/.
  • Out of scope: auto-generated reference (OpenAPI explorer, pdoc) — those are generated artifacts and updated by the build, not by the docs process.

Operating principles

  1. Single source of truth. One topic = one canonical page. Other pages link to it; they never re-state it. If two pages contradict, fix the canonical page and update the others to point at it.
  2. Co-located with code. Documentation lives in the same repository, on the same branch, and changes in the same PR as the code it describes. There is no separate "docs site" or wiki.
  3. Living, not archived. A page is either current or replaced. There are no "deprecated but kept around" pages — outdated content is removed in the same change that replaces it (with redirect stubs only when external links exist).
  4. Screen specs over screenshots. Visual reality is captured by anatomy diagrams (annotated SVG) checked into git, not pixel-perfect screenshots. A diagram aligns to architecture; a screenshot rots on every style tweak.
  5. Glossary first. No new domain term goes into a page without a definition added to the frontend glossary in the same PR. Term consistency is the foundation of every other doc.
  6. Plain language. Documentation is read by engineers whose first language is rarely English. Short sentences. Common words. One idea per sentence. The style guide is normative.
  7. Page history is required. Every non-redirect page ends with a Page history table. Every meaningful edit appends a row. This is the ledger that lets a future reader see when something was last verified.
  8. Definition of done includes docs. A code change that touches user-visible behavior is not done until the corresponding docs are updated in the same PR.

Information architecture

The frontend documentation tree is organized into five buckets, mirroring the structure used by Stripe Elements, Atlassian Design System, GitHub Primer, and Material Design. Every page belongs to exactly one bucket; pages that span buckets are split.

  1. Foundations — meta-rules for documentation itself: style guide, maintenance process (this page), glossary, role-based onboarding paths.
  2. Architecture — system-level inventory: feature ledger, runtime ownership, contracts.
  3. Screens — one page per real user-facing screen, with screenshots, anatomy diagram, and full behavior. Lives under docs/internal/front/screens/.
  4. Components — reusable UI primitives (UI kit, tooltips, popups/overlays, diagrams/lightbox, hotkeys).
  5. Cross-cutting systems — concerns that span the surface: navigation contracts, search/discovery, UI motion and adaptivity, feedback/editorial workflow.

The left navigation in the docs site (docs/assets/internal-sidebar.js → INTERNAL_SIDEBAR_NAV) renders these five buckets as nested groups. When you add a page, place it in the right bucket and update the sidebar tree in the same PR.

Document types and their templates

Each document type has a fixed shape so readers know where to look. Use the linked template when creating a new page; do not invent a new structure.

Type Purpose Template / location

Lifecycle of a documentation change

  1. Trigger. A change happens in code (new screen, new feature, behavior tweak), or audit flags drift between docs and reality, or the team decides to start documenting an area that wasn't covered.
  2. Identify the canonical page. Use the architecture map to find the existing page; if no canonical page exists, create one from the appropriate template.
  3. Plan the doc edit alongside the code. Add the doc edits to the same branch / PR as the code change. If the doc is large enough to be its own PR, link it from the code PR description.
  4. Update the page. Edit only what changed. Preserve the page skeleton (Overview → topic cards → Page history). Use plain language; reference the style guide when in doubt.
  5. Update the glossary if any new term appears. Add a row with a plain-language definition and an implementation note in the same PR. Cross-link from the page.
  6. Update the architecture map if scope changed. If the change adds or removes a feature, update the block model and feature-implementation ledger.
  7. Append a Page history row. One sentence summarizing what changed; today's date in YYYY-MM-DD; a link to the maintainer's portal page.
  8. Self-audit. Run through the DoD checklist before requesting review.
  9. Review. A second pair of eyes verifies that the doc matches reality and is readable to someone who didn't write the code.
  10. Merge with code. Docs and code land together. Never merge a behavior change without the accompanying doc edit.

Screen-spec policy

Screen specs are the highest-leverage documentation artifact: they describe a real user-facing surface in full. They cost the most to write and pay back the most when changes are needed.

Which screens get a spec

  • Required: any screen that is a user entry point (home, backlog cockpit, OpenAPI explorer landing).
  • Required: any screen with non-trivial interaction logic, multiple states, or animations beyond a fade.
  • Optional: simple content pages whose behavior is covered by the global skeleton.

Required sections in every screen spec

  1. Overview — primary URL, owner, audience, last verified date.
  2. Anatomy diagram — annotated SVG with numbered callouts.
  3. Layout map — text walk-through of every region.
  4. One section per major behavior (e.g. "Boot intro overlay", "Hero region", "Card sections").
  5. Data contract — every data-* hook and class state used by the runtime.
  6. Interaction logic — page lifecycle, event contract, timing budget.
  7. Responsive behavior — breakpoints and what changes at each.
  8. Accessibility and motion policy — reduced motion, ARIA, focus, contrast, theme.
  9. Where to change what — file-by-file mapping of behavior to source.
  10. Acceptance checklist — items a reviewer can tick off post-change.
  11. Known edges and design decisions — non-obvious choices and historical fixes that explain why the code looks the way it does.
  12. Related internal docs — links to glossary, architecture map, sister specs.
  13. Page history — the change log table.

Required visual gallery (screenshots)

  • Every screen spec opens with a Visual gallery card containing real screenshots of the screen.
  • Minimum coverage (using canonical viewport breakpoints):
    • Desktop default state (≥1025 px), light theme
    • Desktop default state (≥1025 px), dark theme
    • Tablet breakpoint (761–1024 px)
    • Mobile breakpoint (≤760 px)
    • Each transient state (intro overlay, modal, loading, empty, error)
  • Files live in ./assets/ next to the spec; filenames follow <screen-id>-<variant>.png (kebab-case).
  • Update the gallery in the same PR that changes the screen — outdated screenshots are worse than no screenshots.
  • Capture via the project's screenshot script (Playwright + WebKit) so the output is reproducible. Manual screenshots are accepted only when the script can't reach the state (e.g. interactive demo).
  • Each <figure> gets a one-line <figcaption> describing the variant ("Desktop · light · default", "Intro · phase 2 · mid").
  • Wrap each image in a link to the full-size PNG so reviewers can zoom.

Anatomy diagram standards

  • Format: inline SVG embedded in the page; never external <img>.
  • Numbered callouts: small accent-colored circles with monospace numbers; legend below the figure repeats every number.
  • Two phases when applicable: if the screen has a transient state (intro, modal, loading), the diagram shows both states stacked vertically with phase labels.
  • Accessibility: SVG must have <title> and <desc> children, plus a <figcaption> below.
  • Aspect-ratio: use viewBox, not fixed dimensions; figure scales with viewport.
  • Maintain by hand. SVG is text — edit it the same way you edit any other code. Don't paste binary screenshots.

Visual evidence policy

Every screen spec carries both a real-screenshot gallery and an annotated anatomy diagram. Each answers a different question:

  • Screenshots answer "what does this look like right now?" — fidelity, recency, dark/light parity, breakpoint reflow.
  • Anatomy diagrams answer "what are the named regions?" — structure, callouts, vocabulary that the rest of the spec references.

Screenshot rules

  • Captured with the project's reproducible script (scripts/capture_screen_specs.py), not ad-hoc.
  • Stored in ./assets/ next to the spec, in PNG.
  • Filename format: <screen-id>-<variant>.png, kebab-case. Variants: desktop-light, desktop-dark, tablet, mobile, intro-phase-N, etc.
  • Refreshed in the same PR that changes the screen.
  • Each image wrapped in a link to itself so reviewers can open full-size.
  • Each image gets descriptive alt text and a one-line <figcaption>.

Anatomy diagram rules

  • Inline SVG, hand-authored, never an external <img>.
  • Numbered callouts repeated in a legend below the figure.
  • For multi-phase screens (intro overlay + page surface), stack two panels in one SVG.
  • SVG must have <title> and <desc> children plus a <figcaption>.
  • Use viewBox, not fixed dimensions; figure scales with the viewport.
  • Edit the SVG in place — it is text in git, not a binary artifact.

Why both, not one or the other

  • Screenshots rot on style tweaks but are the only reliable evidence of current visual reality. The rule is: keep them current via the script.
  • Anatomy diagrams describe structure, are diff-readable in code review, screen-reader-friendly, and survive cosmetic changes.
  • Together they cover the full "show + name" loop that international design systems use.

Live previews as supplementary aid

  • Link directly to the page (../../../index.html) — the reader can open the actual screen.
  • Do not embed cross-origin iframes or third-party preview services.

Page history rules

  • Required on every non-redirect page. Mount it as the last <section class="card"> before the in-page TOC.
  • One row per meaningful change. Trivial whitespace, link rewrites, or auto-formatting do not require a row.
  • Reverse chronological order. Newest row at the top of <tbody>.
  • Date format: ISO YYYY-MM-DD in <time datetime> and rendered as the same string.
  • Author link: point to the maintainer's portal page (../../portal/people/<name>/index.html).
  • Change description: one sentence, present-tense imperative summary of what the change accomplishes for the reader. Do not say "fixed typo" — describe the substance ("clarified that decrypt only runs on motion-allowed visits").

Audit cadence

Documentation drifts. The audit cadence below catches drift before it becomes wrong-by-default.

  • Per-PR audit: every PR that touches user-visible behavior is reviewed for matching doc updates. Reviewer responsibility — failing to update docs is grounds for "request changes".
  • Quarterly screen-spec re-verify: the listed owner re-runs the acceptance checklist of each owned screen spec. If anything diverges, an update PR follows. Update the "Last verified behavior date" in the Overview.
  • Glossary scan on every release: grep new code for unfamiliar terms; if any has no glossary entry, add one or fix the naming.
  • Annual architecture-map cleanup: retire features that no longer exist; reassign ownership; renumber feature IDs only with a clear migration note.

Definition of done (for any change)

A code change is not "done" until all of the following are true. Use this as a self-check before review.

  1. The behavior change is implemented and tested in code.
  2. The relevant screen spec or reference page is updated in the same PR.
  3. If new terminology was introduced, it is defined in the frontend glossary in the same PR.
  4. The architecture map is updated if the change adds, removes, or reassigns a feature.
  5. A page-history row is appended to every page that was edited, with today's date and a meaningful description.
  6. For screen specs: the anatomy diagram still matches reality, and the acceptance checklist is reviewed.
  7. For new pages: the required skeleton (Overview → topic cards → Page history → in-page TOC mount) is present.
  8. No links are broken; data-* attribute names match between markup and runtime code.
  9. Plain-language self-check: short sentences, common words, no unexplained jargon.
  10. The change description in the PR mentions both the code change and the doc change.

Anti-patterns to avoid

  • "Docs PR follows." Code merges with the promise that docs will land later. The follow-up almost never ships. Always bundle.
  • Two pages, same topic. Splitting a topic across two pages because each one is "almost about it" creates contradictions. Merge or pick one canonical home.
  • Stale screenshots. A PNG of a screen that no longer looks like that. Either remove it or replace with a current SVG anatomy.
  • Empty page-history. A page with no history table is invisible to future maintainers and gets quietly deleted. Always add the table, even on the first commit.
  • Jargon without a glossary entry. A page introduces "the conspectus orchestrator" with no definition; future readers either guess or ask. Define it.
  • Long noun stacks. "The configuration value resolution preference cascade order rule" — break it down into a sentence with a verb.
  • Marketing copy in engineering docs. "Effortlessly seamless modern API" is for landing pages. Engineering docs say what it does, not how exciting it is.
  • "TBD" and "WIP" sections in shipped docs. If a section isn't ready, omit it. Placeholder text encodes ambiguity.
  • Inconsistent terminology. Same thing called three different names across three pages. Pick one and unify in a single PR.
  • Hand-edited generated content. If the page is generated from OpenAPI / pdoc, never hand-edit the output — fix the source.

Onboarding: first time editing docs

  1. Read the style guide end-to-end (~10 min).
  2. Skim this process page (~5 min).
  3. Skim the frontend glossary to learn the canonical terms (~5 min).
  4. Open the home screen spec as a reference for what a complete screen spec looks like.
  5. For your first edit, start with a small change to an existing page; copy the structure of nearby cards.
  6. Always use the templates (screen template, ADR template, etc.) — do not start from a blank file.
