Overview

This page describes contributor-facing frontend features that support docs quality loops: page feedback intake, page ownership visibility (Edited by), and generated maintainer metadata.

Page feedback form workflow

  • Injection point: injectDocsFeedbackCard() appends the FAB and modal to document.body at runtime; no HTML mount required on the page.
  • Input model: feedback type select + required textarea with minimum length validation.
  • Status UX: live status text and non-blocking toast while redirecting to GitHub issue creation.
  • Issue target: prefilled URL from docsFeedbackIssueUrl() to iboiarkin96/study_bot.
  • Fallback: message copied to clipboard when available before opening GitHub.

Runtime owner: docs/assets/docs-nav.js; styles: docs/assets/docs.css (.docs-feedback-card* selectors).

UI contract for the floating "Report issue" beacon (the round bug FAB at the bottom-right of every page) and its modal — sizing, positioning vs. the back-to-top rocket, states, accessibility, GitHub URL payload — lives in Docs frontend report-bug FAB.

GitHub feedback contract

Edited by widget (page ownership visibility)

  • HTML contract: data-maintainer-ids on <body> and <div id="docs-page-meta-mount" hidden></div>.
  • Runtime: renderPageMeta() in docs/assets/docs-internal-meta.js.
  • Presentation: compact avatar stack (up to 3 + counter) and expandable full list via Edited by button.
  • Data source: person profiles and maintainer-page mappings from generated docs/assets/docs-portal-data.js.
  • Unknown IDs: explicit inline warning with instructions to add portal profile and run make docs-fix.

Portal data model and generation flow

  • Generated file: docs/assets/docs-portal-data.js is built from profile pages under docs/internal/portal/people/.
  • Generation process: documented in file header and changelog entries; generated by repository scripts and make docs-fix.
  • Consumed by frontend: docs/assets/docs-internal-meta.js for page editors, people groups, and maintained pages lists.
  • Profile UX: maintained pages view uses pagination and skeleton loading for visual consistency.

Operational and telemetry touchpoints

  • Search telemetry: emitted as custom events and optional POST endpoint if configured by meta tag/localhost fallback.
  • Palette telemetry: open/filter/execute/reading-mode actions emitted via docs-palette-telemetry events.
  • Feedback governance: weekly cadence workflow enforces triage discipline for user-reported docs issues.

Telemetry contract (frontend docs UX)

This contract defines the minimum event surface expected by analytics consumers and QA checks.

Channel Events Required payload context Implementation owner
Search search_query, search_result_click, search_success, load/error events page path, query, result metadata, status/error context emitDocsSearchTelemetry() in docs/assets/docs-nav.js
Command palette palette_open, palette_close, palette_filter, palette_execute page path, trigger type, action id/group, query length emitDocsPaletteTelemetry() in docs/assets/docs-nav.js
Feedback workflow Issue creation is observable in GitHub issues stream and cadence workflow artifacts page URL/path, feedback type, issue template alignment injectDocsFeedbackCard() + GitHub template/workflow artifacts

Endpoint configuration for search telemetry can be provided via page metadata; if unset, runtime uses safe defaults.

Practical checklist for analysts and frontend contributors

  1. When adding a new page, include data-maintainer-ids and docs-page-meta-mount so ownership is visible.
  2. For feedback-related changes, validate issue template and cadence workflow together.
  3. After profile or maintainer updates, regenerate portal data and verify Edited by rendering.
  4. Document material frontend behavior changes in docs/CHANGELOG.md.
