Docs portal · Reading aids

Summary Resume-reading banner and floating back-to-top button. Status Availability v1 · Internal Owner Ivan Boyarkin Last reviewed 2026-04-30 Spec template v1

Overview

Long-page reading aids for internal docs: a top resume-reading banner that offers a one-click jump back to the reader's previous scroll position, and a floating back-to-top button that appears after a scroll threshold. Both are opt-in per page — the runtime (docs-resume.js) loads only where the page declares it.

Status: in-progress — first deployed on internal/api/conspectus/operations/post-…actions-review.html (long page, lots of state-table scrolling). Stabilization criteria below.

Anatomy

  1. Resume banner — appears at the top of .internal-layout__main after DOMContentLoaded if a saved scroll position exists for this URL within the 14-day TTL. Includes a primary action ("Resume reading at section X") and a dismiss button. Single instance per page.
  2. Back-to-top button — fixed-position circular button at the bottom-right of the viewport. Hidden by default; reveals once scrollY > 600 px and smooth-scrolls to 0,0 on click. Persists across navigations within the page; respects safe-area-insets on mobile.

Both controls are visual children of .internal-layout; they are not portaled into document.body, so the drawer overlay (z-index 1000) still covers them when active. CSS rules live in internal-layout.css.

Visual reference

The two reading aids appear at opposite ends of the page and on different triggers. The mocks below show where each one lives and when it shows up. Left: the resume banner anchored to the top of .internal-layout__main on first load. Right: the back-to-top FAB revealing at the scrollY > 600 px threshold near the bottom-right of the viewport.

Why these two together. The resume banner is the first reading aid the reader sees on returning to a long page; the back-to-top FAB is the last after they reach the bottom. Pairing them in one spec keeps the scroll-aware UX in a single page so we can reason about overlap (none — the banner is hidden by the time the FAB shows) and stacking (both children of .internal-layout, both covered by the drawer overlay at z-index 1000).

What the threshold means. BACK_TO_TOP_THRESHOLD = 600 px is the first reveal — at this point the FAB appears as the compact "sleeping rocket" pill. A second, deeper threshold (~ 320 px from the footer) arms the rocket; only the armed click actually launches the rocket animation. The full state machine — Compact → Armed → Boil → Flight → Cleanup — is the subject of Rocket launch animation.

Public surface

Loaded by
Pages that opt in via <script defer src="…/assets/docs-resume.js">. The script gracefully no-ops on pages without scrollable content or without an .internal-layout host.
Storage keys
localStorage · docs.resume.<path> — JSON { scrollY, ts }, throttled write at 750 ms, TTL 14 days.
sessionStorage · docs.resume.seen.<path> — "1" when the banner has been actioned or dismissed in the current tab. Cleared on tab close.
Tuning constants (in source)
TTL_MS = 14 × 24 × 60 × 60 × 1000
MIN_SCROLL_FRACTION = 0.25 — banner only offers if reader passed 25% of the page before
BACK_TO_TOP_THRESHOLD = 600 px
SAVE_THROTTLE_MS = 750 ms
CSS hooks
.docs-resume-banner, .docs-resume-banner__action, .docs-resume-banner__dismiss, .docs-back-to-top — all defined in internal-layout.css.
CustomEvents
None emitted or listened. The module is self-contained.
Window globals
None.

Do's and Don'ts

DoDon't
Opt in only on pages where the reader is likely to scroll past 25% — long API operation pages, deep ADRs, audit reports. Add the script globally — short pages get a banner that never resolves and a button that never appears, wasting bytes and runtime.
Bump the SAVE_THROTTLE_MS constant if the page has very long scroll regions and CPU-sensitive behavior (e.g. heavy SVG diagrams). Reduce the throttle below 250 ms — repeated localStorage writes degrade performance and battery on mobile.
Test the prefers-reduced-motion: reduce path: button must scroll instantly, banner must show without fade. Use scrollIntoView({ behavior: "smooth" }) unconditionally — it ignores reduced-motion and animates anyway. Branch on the media query.
Treat the resume banner as informational — keep the dismiss button always reachable via Tab + Esc. Make the banner sticky or full-screen — it would block the page and erode trust.

Acceptance checklist

  1. Loading a previously-scrolled page (within 14 days, scrollY > 25% of page height) shows the banner above all other content.
  2. Dismissing the banner persists for the current tab session — banner does not return on F5 in the same tab.
  3. Closing and reopening the tab brings the banner back (sessionStorage cleared).
  4. After 14 days, the banner does not appear even if a save exists — entry has expired.
  5. The back-to-top button appears at scrollY = 600 px on every breakpoint and disappears below it.
  6. Click on the back-to-top scrolls smoothly to top under prefers-reduced-motion: no-preference; jumps instantly under reduce.
  7. Both controls are reachable via Tab, dismissable via Esc or activation.
  8. No console errors and no stray timers after navigation.
