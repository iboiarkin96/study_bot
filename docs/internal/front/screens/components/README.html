Component screenshots — convention Reference

Docs portal · Visual reference

Purpose

This folder holds raster screenshots of internal docs portal components — bug FAB, back-to-top rocket, drawers, modals, tooltips, the docs hero, and any other UI surface whose cross-viewport behaviour is part of its specification. Each component spec page (e.g. report-bug-fab, rocket-launch-animation) includes a Visual gallery section that links into ./assets/.

Page-level screen captures (home landing, backlog cockpit) live one folder up in ../assets/; this components/ folder is the parallel surface for component-grain visuals.

Viewport canon — required four captures

Every component gallery must ship at least these four viewports. They cover the canonical breakpoint tiers (Desktop / Tablet / Mobile) and add a small-mobile capture so iPhone SE-class devices don't regress quietly.

Capture CSS pixels (width × height) Tier Filename suffix
Desktop 1440 × 900 Desktop ≥ 1025 px -desktop.png
Tablet (iPad Pro portrait) 1024 × 1366 Tablet 761–1024 px -tablet.png
Mobile (iPhone Pro Max) 430 × 932 Mobile ≤ 760 px (large) -mobile-pro-max.png
Mobile (iPhone SE) 375 × 667 Mobile ≤ 760 px (small floor) -mobile-se.png

Optional fifth capture — dark theme — when the component renders meaningfully different in data-theme="dark" (e.g. the docs hero, syntax-highlighted panels). Use the suffix -dark.png appended to any of the four (e.g. report-bug-fab-mobile-pro-max-dark.png).

Filename pattern

<component-id>-<viewport>[-<state>][-<theme>].png
  • component-id — kebab-case, matches the spec page basename without docs-frontend- prefix when reasonable. Examples: report-bug-fab, rocket-launch-animation, internal-sidebar-drawer.
  • viewport — one of desktop, tablet, mobile-pro-max, mobile-se.
  • state (optional) — hyphenated state name (idle, hover, armed, launching, modal-open). Omit for the default visible state.
  • theme (optional) — light (default; usually omitted) or dark.

Full examples:

  • report-bug-fab-desktop.png — bug FAB at default state, desktop, light theme.
  • report-bug-fab-mobile-pro-max-modal-open.png — modal opened, mobile.
  • rocket-launch-animation-mobile-se-armed.png — armed state at iPhone SE width.
  • rocket-launch-animation-desktop-launching-dark.png — launching, desktop, dark theme.

How to capture (manual, until Playwright lands)

  1. Run make docs-serve (or python3 -m http.server 8765 --directory .) and open the page that hosts the component in Chrome.
  2. Open DevTools → toggle device toolbar → choose viewport from the canon table above. Use Responsive mode and type the exact CSS width × height (DevTools defaults like "iPhone 14 Pro Max" round to non-canonical sizes; type the explicit number).
  3. Set DPR (device pixel ratio) to 2 for retina parity. Set data-theme via html[data-theme="dark"] in the elements panel for dark captures.
  4. Trigger the state if the gallery requires it (hover, focus, click — use Chrome's Force element state). For animations (rocket flight, antenna twitch) capture at the canonical phase listed in the spec page (e.g. t = 1.0 s for boil).
  5. Use Capture node screenshot (right-click on the component DOM node) for tight crops, or Capture full-size screenshot for whole-viewport context.
  6. Save the PNG into ./assets/ using the canonical filename. Re-export at the same dimensions when updating; never resize after the fact.

Yandex Browser / Safari: the Yandex DevTools mirror Chrome's, so the same flow works. iOS Simulator (Xcode) gives the most accurate iPhone capture and is preferred when a manual re-shoot is needed for a notch / safe-area concern.

When to update

  • Same PR as the visual change. If a CSS/markup change touches the component, the gallery must be re-shot in the same PR. Stale screenshots are worse than missing ones — they lie.
  • Quarterly drift sweep. Even with no logical change, re-shoot all galleries once per quarter to catch typography / theme / browser drift.
  • Whenever a new viewport tier is added to the canon (canonical-breakpoints). Adding a tier means re-shooting every existing gallery.

Future: Playwright auto-generation

Manual capture is the bridge convention. The mandatory follow-up — tracked as a backlog item — is a Playwright harness in tests/visual/ that auto-generates every PNG in this folder against the live docs portal, with a make docs-screenshots entry point. See backlog for the entry; until that lands every update is manual and reviewed via PR.

When the harness ships, this README is updated to make manual capture a fallback, not the default. The viewport canon and filename pattern remain authoritative for both flows.
