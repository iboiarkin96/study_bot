Overview

This guide is the rule set for every hand-written page under docs/. Reviewers apply it during PR review, and authors paste it into LLM prompts when a model assists with the audit.

  • Goal: keep the docs predictable and easy to read for engineers who use English as a second language.
  • Scope: every HTML page under docs/, including index pages and redirect stubs.
  • Related policy: ADR 0001, ADR 0018, ADR 0024, ADR 0026.

Global style rules

  • Language: English. Plain engineering prose. No slang, no marketing language.
  • Tone: directive. State what is required, what is optional, and what is forbidden.
  • Tense: present tense for current rules and behavior. Future tense only for planned work.
  • Heading case: sentence case. Labels stay short and concrete.
  • Sectioning: one topic per <section class="card">.
  • Terminology: one term per concept, repeated everywhere. Pick “OpenAPI contract test” and never write a variant.
  • Links: name the destination, not the action. Write ADR 0027, not “Open ADR 0027”.
  • Commands and paths: wrap in <code>. Examples must be safe to copy and paste.
  • Lists: ordered for procedures, unordered for policies and reference rows.

Plain language

The audience reads English as a second language. Stay formal and accurate, but prefer plain words and short sentences over academic phrasing.

  • Sentence length: one main idea per sentence. Split chained clauses into separate sentences.
  • Vocabulary: pick the common word when both options are accurate. Write “use”, not “utilize”; “show”, not “surface”.
  • Jargon: keep necessary technical terms. Spell out every acronym the first time it appears on a page (<abbr title="…"> or a short gloss).
  • Voice: active voice for instructions: “Run make verify.” Passive voice only when the actor does not matter.
  • Nouns: do not stack noun phrases. “The pipeline runs tests” beats “the test execution phase of the pipeline”.
  • Consistency: when one row of a hub page is simplified, every row gets the same treatment.

Apply these rules to every page you edit. Older pages catch up as they come up for review; do not block a small PR on a sweep through legacy text.

Required page skeleton

Every non-redirect page follows this skeleton.

  1. <h1> page title.
  2. <div id="docs-top-nav"></div> top-navigation mount.
  3. First content card: Overview or Status, depending on document type.
  4. Body cards, one topic per card.
  5. <section id="page-history" class="card"> with the Page history table (see Page history).

The in-page table of contents and the report-bug FAB are injected by docs-nav.js at runtime. Do not write them into the HTML.

Redirect pages are the one exception: a short “Page moved” card and the canonical link.

Frontend documentation page-type taxonomy

Every page under docs/internal/front/ belongs to one of six page types. Place a single <span class="docs-page-type docs-page-type--<variant>">Label</span> directly after the <h1>, so the type is visible to readers and to link-graph tooling. Pick the type before you write — size budgets and required sections differ.

Type Badge Purpose Size budget Required sections
Foundation Foundation Cross-cutting principles, vocabulary, and process used by every other page. 200–500 lines Overview, the foundational content, governance, page history
Architecture Architecture System map: blocks, ownership, feature ledger, runtime contracts. 300–600 lines Block model, ownership map, feature inventory, page history
Screen Screen Full-page recipe — anatomy, gallery, behavior, and data contract for one URL. 250–900 lines At a glance, gallery, anatomy, layout map, data contract, behavior, accessibility, do and don't, where to change what, page history
Component Component Reusable UI surface — anatomy, states, props, accessibility, do and don't. 150–350 lines Inventory or anatomy, states, public surface, accessibility, do and don't, page history
Reference Reference Lookup tables — JS modules, CSS architecture, tokens, hotkeys, telemetry events. 100–400 lines One-line introduction, normalized table, page history
Contract Contract Stable normative requirements only. Implementation lives elsewhere. 80–200 lines Overview with links to the implementation pages, requirements as bullet lists, acceptance checklist, page history

A 140-line page is short for a Screen and complete for a Contract. The badge tells reviewers which bar to apply.

Screen spec template baseline (frontend pages)

The authoritative skeleton lives at screens/docs-screen-template.html. A screen spec missing any required section is rejected at review.

  1. Title block with <span class="docs-page-type docs-page-type--screen"> badge
  2. At a glance: status, owner, audience, primary URL, tracked feature IDs, last verified
  3. Visual gallery: desktop light and dark, tablet, mobile, every transient state — required
  4. Anatomy diagram (numbered SVG) and legend — required
  5. Layout map: a reading-order region walk-through — required
  6. Behavior matrix: events, states, side effects
  7. Timing budget: per-phase milliseconds
  8. Data contract: every data-* attribute
  9. Responsive behavior per breakpoint
  10. Browser and device support matrix — required
  11. Accessibility and motion: global policies and a per-component table — required
  12. Do and Don't table — required
  13. Where to change what: file-by-file map and tuning knobs
  14. Acceptance checklist
  15. Known edges and design decisions
  16. Related internal docs
  17. Page history

Component spec template baseline (frontend reusable surfaces)

  1. Title with <span class="docs-page-type docs-page-type--component"> badge
  2. Component inventory or single-component anatomy
  3. States table: default, hover, focus, active, disabled, loading, error
  4. Public surface: CSS hooks, data-* contract, JS events — required
  5. Accessibility notes: roles, keyboard, screen reader, reduced motion — required
  6. Do and Don't table — required
  7. Positioning and sizing table for any surface with a fixed footprint (FABs, drawers, modals, sticky chrome). List values per viewport. When all viewports share a value, write “uniform across all viewports” so future authors do not silently drift it. Required for any component that appears at the same coordinates on more than one page.
  8. Visual gallery with raster screenshots — required for any component whose appearance shifts across viewports (FABs, drawers, modals, sidebar, page header). Filenames and capture rules: screens/components/README. Minimum four captures (desktop 1440×900, tablet 1024×1366, iPhone Pro Max 430×932, iPhone SE 375×667) plus one capture per non-default state. Capture is manual until the Playwright harness lands; gallery slots stay as dashed placeholders so reviewers see the gap.
  9. Implementation map: which files, which tokens
  10. Page history

Reference page template baseline

  1. Title with <span class="docs-page-type docs-page-type--reference"> badge
  2. One-line introduction stating the lookup question this page answers
  3. The normalized table — no per-row prose; everything in columns
  4. Notes section for footnotes and conventions
  5. Page history

Contract page template baseline

  1. Title with <span class="docs-page-type docs-page-type--contract"> badge
  2. Overview with explicit links to the implementation pages
  3. Requirements as bullet lists, grouped by area
  4. Acceptance checklist of concrete observable behaviors
  5. Page history

Authoring for mobile

The docs portal must remain readable on a phone (≈360–440 CSS px) with no shrunk type and no horizontal overflow. CSS handles most of the work. The rules below cover the cases authors still get wrong, and are normative — reviewers reject pages that break them.

  • Wrap multi-column tables. Any table with three or more columns, or any column with long identifiers or code, lives inside <div class="docs-table-scroll" role="region" aria-label="…">…</div>. The aria-label is required for screen-reader navigation.
  • Do not paste raw long URLs into prose. A URL longer than about 50 characters belongs inside a named anchor: <a href="…">readable label</a>.
  • Use a block <pre> for code longer than about 20 characters. Inline <code> is for short tokens. Even with the global wrap rule, long inline code crowds the content column on a phone.
  • Do not add a breakpoint below 760 px without a written reason. The canonical device tiers are defined here. A 640, 720, or 900 px rule is allowed only as a layout reflow inside an existing tier, never as a new device tier.
  • Test on a phone-sized viewport before merge. Run the mobile-contract acceptance smoke on iPhone 17 Pro Max (440 px) and iPhone SE (375 px). A page that fails any item does not merge.

Naming and linking conventions

  • File names: lowercase kebab-case. Use a numeric prefix when ordering matters (ADRs, RFCs, runbooks).
  • Operation page names: HTTP method plus path pattern, for example post-api-v1-user.html.
  • Link labels: stay consistent inside one index. Use ADR 00XX everywhere; do not mix it with “Open ADR 00XX”.
  • Index descriptions: one sentence per row, the same grammar across rows.

Page history

Every hand-written page ends with a Page history card, placed before the in-page TOC mount. Add one row per substantive edit; skip pure typo fixes. In Author, link to the maintainer's Employee portal profile when one exists (<a href="…/internal/portal/people/<slug>/index.html">…</a>, path relative to the current page). Use an em dash (—) for older rows where the author is unknown.

Audit protocol for documentation changes

Run this protocol on every docs PR and on periodic audits.

  1. Run the quality gates: make docs-fix, make docs-html-check, plus the checks the project requires.
  2. Verify links and navigation on every changed or added page.
  3. Compare section structure to the template for that document type.
  4. Check terminology against nearby index and hub pages.
  5. Confirm that the changelog and cross-links were updated when policy or process changed.
  6. Record deviations and remediation in the PR checklist or in the audit notes.

LLM conformance checklist (copy and paste)

Paste this prompt when an LLM reviews a docs change.

Validate changed docs against docs/internal/front/documentation-style-guide.html.
Return:
1) Violations grouped by severity (critical/major/minor).
2) For each violation: file path, violated rule, short evidence, concrete fix.
3) Missing required sections by document type.
4) Terminology inconsistencies (same concept, different naming).
5) Broken or suspicious links.
6) Final pass/fail verdict and residual risks.

Governance and updates

  • A style-rule change lands in this page first, then propagates to related guides.
  • When a rule changes architecture or process, the matching ADR or RFC moves in the same PR.
  • On any conflict, this guide overrides per-page wording.
