Overview

This document is the single source of truth for documentation style, structure, and quality checks across docs/. Use it for both human review and LLM-driven conformance checks.

  • Goal: keep all docs predictable, readable, and auditable over time.
  • Scope: all HTML documents under docs/ plus index pages and redirect stubs.
  • Related policy: ADR 0001, ADR 0018, ADR 0024, ADR 0026.

Global style rules (all document types)

  • Language: English, clear engineering prose, no slang, no hype.
  • Tone: neutral and directive; describe what is required, optional, and prohibited.
  • Tense: present tense for current behavior and policy; future tense only for planned work.
  • Heading case: sentence case for section headings; keep labels short and explicit.
  • Formatting: one primary topic per <section class="card">.
  • Terminology: use stable terms consistently (for example: "OpenAPI contract test", not variants).
  • Links: link by destination name (for example: ADR 0027, Runbook: Tests failing).
  • Commands and paths: wrap in <code>; use copy-paste-safe command examples.
  • Lists: use ordered lists for procedures and unordered lists for policies/reference.

Plain language (readability)

Docs should be easy to read for engineers who use English as a second language. Stay formal and precise, but prefer simple wording and short sentences over ornate or academic style.

  • Sentence length: aim for one main idea per sentence; split long chains with semicolons into two sentences.
  • Vocabulary: use common terms (e.g. “use”, “show”, “end”) instead of heavy synonyms when both are accurate.
  • Jargon: keep necessary technical terms; define acronyms on first use in a page ( plus title or a short gloss).
  • Voice: prefer active voice for steps (“Run make verify”) and passive only where the actor does not matter.
  • Nouns: avoid stacking long noun phrases; rephrase with a short verb (“The pipeline runs tests” not “The test execution phase of the pipeline”).
  • Consistency: when you simplify a hub or index, apply the same pattern to similar rows (same grammar, same level of detail).

Rolling updates: apply this section when editing any page; large legacy pages (older ADRs, long internal specs) can be updated in smaller PRs over time.

Required page skeleton

All non-redirect pages should follow this baseline skeleton.

  1. <h1> page title.
  2. <div id="docs-top-nav"></div> top navigation mount.
  3. First content card: either Overview or Status (document type dependent).
  4. Body cards grouped by one topic per card.
  5. <section id="page-history" class="card"> with a Page history table (see Page history).

The in-page TOC mount and the report-bug FAB are injected by docs-nav.js at runtime and should not be authored in HTML.

Redirect pages are the only exception: short "Page moved" card plus canonical link.

Frontend documentation page-type taxonomy

Pages under docs/internal/front/ belong to one of six page types. Place a single <span class="docs-page-type docs-page-type--<variant>">Label</span> immediately after the page <h1> so the page type is visible to readers and to automated link-graph tooling. Each type has a different size budget and a different set of mandatory sections — choose the type before you start writing, not after.

Type Badge Purpose Size budget Required sections
Foundation Foundation Cross-cutting principles, vocabulary, process — applies to every other page. 200–500 lines Overview, the foundational content itself, governance, page history
Architecture Architecture System map: blocks, ownership, feature ledger, runtime contracts. 300–600 lines Block model, ownership map, feature inventory, page history
Screen Screen Full-page recipe — anatomy, gallery, behavior, data contract for one rendered URL. 250–900 lines At a glance, gallery, anatomy, layout map, data contract, behavior, accessibility, do/don't, where to change what, page history
Component Component Reusable UI surface — anatomy, states, props, accessibility, do/don't. 150–350 lines Inventory or anatomy, states, public surface, accessibility, do/don't, page history
Reference Reference Lookup tables — JS modules, CSS architecture, tokens, hotkeys, telemetry events. 100–400 lines One-line introduction, normalized table, page history
Contract Contract Stable normative requirements only — implementation lives elsewhere. 80–200 lines Overview (with links to implementation pages), the requirements as bullet lists, acceptance checklist, page history

Why this matters: a 140-line page reads as "incomplete" only if its type is unknown. A Contract page at 140 lines is correctly sized; a Screen page at 140 lines is missing required sections. The badge resolves that ambiguity at first glance and lets reviewers apply the right acceptance bar.

Component spec template baseline (frontend reusable surfaces)

  1. Title with <span class="docs-page-type docs-page-type--component"> badge
  2. Component inventory or single-component anatomy
  3. States table (default, hover, focus, active, disabled, loading, error)
  4. Public surface (CSS hooks, data-* contract, JS events) — required
  5. Accessibility notes (roles, keyboard, screen-reader, reduced-motion) — required
  6. Do's and Don'ts table — required
  7. Positioning & sizing table for any surface that has a fixed footprint (FABs, drawers, modals, sticky chrome). State the values per viewport — even when they are uniform, naming "uniform across all viewports" explicitly stops drift. Required when the component appears at the same coordinates on multiple pages.
  8. Visual gallery with raster screenshots — required for any component whose appearance changes meaningfully across viewports (FABs, drawers, modals, sidebar, page header). Filenames and capture canon at screens/components/README; minimum four captures (desktop 1440×900, tablet 1024×1366, iPhone Pro Max 430×932, iPhone SE 375×667) plus one capture per non-default state. Until the Playwright harness lands, capture is manual; gallery slots stay as dashed placeholders so reviewers can see the gap.
  9. Implementation map (which files, which tokens)
  10. Page history

Reference page template baseline

  1. Title with <span class="docs-page-type docs-page-type--reference"> badge
  2. One-line introduction stating the lookup question this page answers
  3. The normalized table (no per-row prose; everything in columns)
  4. Notes section for footnotes and conventions
  5. Page history

Contract page template baseline

  1. Title with <span class="docs-page-type docs-page-type--contract"> badge
  2. Overview with explicit links to the implementation page(s)
  3. Requirements as bullet lists, grouped by area
  4. Acceptance checklist (concrete observable behaviors)
  5. Page history

Authoring for mobile

The docs portal must read on a phone (≈360–440 CSS px) without degraded typography or overflowing layout. The CSS does most of the work, but author-side mistakes can still break readability. The rules below are normative; reviewers should reject pages that violate them.

  • Wrap multi-column tables. Any table with three or more columns, or any column carrying long identifiers / code, must live inside <div class="docs-table-scroll" role="region" aria-label="…">…</div>. The aria-label is required for screen-reader navigation.
  • Don't paste raw long URLs into prose. If the URL is longer than ~50 characters, prefer a named anchor link (<a href="…">readable label</a>) instead of the bare URL.
  • Use a block <pre> for code samples longer than ~20 characters. Inline <code> is for short tokens only — long inline code crowds the content column on mobile even with the global wrap rule.
  • Don't introduce a new component cut-off below 760 px without a documented reason. Canonical device tiers are defined here. If you add a 640 / 720 / 900 px rule, it must be a layout-specific reflow inside the mobile or tablet tier, not a new device tier.
  • Test on a phone-sized viewport before merge. Use the mobile-contract acceptance smoke checklist on iPhone 17 Pro Max (440 px) and iPhone SE (375 px). Pages that fail any item are not mergeable.

Naming and linking conventions

  • File names: lowercase kebab-case with numeric prefix where series requires ordering (for example ADR/RFC/Runbook).
  • Operation page names: HTTP method plus path pattern (for example post-api-v1-user.html).
  • Link labels: keep style stable per index (for example always ADR 00XX, not mixed with Open ADR 00XX).
  • Index descriptions: one sentence, same grammatical style across rows.

Page history

Every hand-written HTML page must end with a Page history card (before the in-page TOC mount). Use one row per substantive edit (skip trivial typo-only fixes). In Author, use a link to the maintainer’s Employee portal profile when it exists (<a href="…/internal/portal/people/<slug>/index.html">…</a>, path relative to the current page). Use an em dash (—) when the author is unknown for older rows.

Audit protocol for documentation changes

Use this protocol for every docs PR and for periodic audits.

  1. Run quality gates: make docs-fix, make docs-html-check, and project-required checks.
  2. Verify links and navigation paths for all changed/added pages.
  3. Verify section structure against this standard for the affected document type.
  4. Check terminology consistency against nearby index/hub pages.
  5. Confirm changelog and cross-links were updated when policy/process changed.
  6. Record deviations and remediation in audit notes or PR checklist.

                              Return:

                              2) For each violation: file path, violated rule, short evidence, concrete fix.

Governance and updates

  • Any style-rule change must be updated in this page first, then reflected in related guides.
  • If a rule affects architecture/process, update the relevant ADR/RFC in the same PR.
  • When uncertainty exists, this document wins over local wording patterns in individual pages.
