Frontend spec — Definition of Done

Internal — Docs frontend · _shared

Overview

A frontend spec page is Done when every box for its page-type is ticked. The structural checks are automated by scripts/front_spec_lint.py; content quality is confirmed by peer review. PR cannot merge with a failing lint.

Templates: component, foundation, contract. Cross-cutting topics (tokens, motion, breakpoints) live once in foundations/ — link to anchors there instead of repeating.

Which DoD applies?

Each page declares its type via data-spec-page on <body>. The DoD section for that type is the one to follow.

data-spec-page Folder Template DoD
component front/components/ component-spec-template Component DoD
foundation front/foundations/ foundation-spec-template Foundation DoD
contract front/contracts/ contract-spec-template Contract DoD
screen front/screens/ docs-screen-template (governed by the screen template's own checklist)
hub · architecture · reference various — Style-guide skeleton + page history only.
template front/_shared/ (self) Lint-exempt.

Common to every page

  • Body attributes: data-spec-page, data-spec-status, data-spec-owner set on <body>.
  • Status in metadata strip matches data-spec-status via [data-spec-status-mount] hydrated by docs-spec-status.js.
  • Owner linked to a portal people page.
  • Page-type badge in H1 matches data-spec-page.
  • Last reviewed date refreshed on each spec change.
  • Page history table has at least one row.

Component DoD

For pages with data-spec-page="component" under front/components/.

  • 1. Overview — what the component is and when to use it.
  • 2. Anatomy — labelled inline SVG or numbered parts list. Each part name reused in States and Code handoff.
  • 3. States & variants — visual matrix covering default + at least one of hover/focus/active/disabled. Theme parity called out.
  • 4. Accessibility — keyboard, focus management, ARIA, contrast, motion all addressed.
  • 5. Behavior contract — interactions, lifecycle, edge cases.
  • 6. Do / Don't — at least three concrete pairs.
  • 7. Code handoff — JS module, CSS selector prefix, DOM hooks, tokens used, feature ledger ID.
  • 8. Related (optional) — siblings, foundations, patterns, or section removed.

Lint command: python scripts/front_spec_lint.py --paths docs/internal/front/components/<file>.html.

Foundation DoD

For pages with data-spec-page="foundation" under front/foundations/.

  • 1. Overview — what the foundation defines and which consumers depend on it.
  • 2. Tokens / values — table or definition list of canonical values; light + dark columns when relevant.
  • 3. Usage rules — both when to apply and when not to apply; at least three positive rules and two anti-patterns.
  • 4. Examples — visual or code demonstrations; every token / value used at least once.
  • 5. Code handoff — source files, selector group / API, generation script if auto-generated.
  • 6. Related (optional) — sibling foundations, dependent contracts, affected components, or section removed.

Contract DoD

For pages with data-spec-page="contract" under front/contracts/.

  • 1. Overview — what the contract governs and the cost of violation.
  • 2. Scope & breakpoints — viewport tiers covered (referenced from canonical breakpoints); explicit out-of-scope.
  • 3. Normative requirements — numbered MUST / SHOULD / MAY clauses, each independently verifiable.
  • 4. Acceptance checklist — pass/fail items mapped to requirement IDs.
  • 5. Implementation entry-points — every JS/CSS file the contract relies on.
  • 6. Visual reference (optional) — inline SVG mocks or screenshots, or section removed.
  • 7. Related (optional) — referenced foundations, participating components, sibling contracts, or section removed.

Quality gates

  • python scripts/front_spec_lint.py --paths <file> exits 0.
  • Page renders cleanly: no broken links, no missing diagrams, examples readable, in-page TOC populated.
  • make docs-fix changes nothing afterwards (no drift).
  • Reviewed by at least one frontend contributor who agrees the spec is implementable as written.
