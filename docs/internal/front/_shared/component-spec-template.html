How to use this template

Golden template for any single-component page under docs/internal/front/components/. Copy this file, rename to <component-name>.html, then fill in every required section in the order shown below.

Required vs optional. Each section is tagged with a badge in its heading and with a data-spec-required="true|false" attribute on its <section> wrapper. Required sections must exist and be non-empty. Optional sections may be removed when they do not apply (for example, a component with a single state has no states matrix to draw) — never reorder, only remove.

Authoring rules and the full Definition of Done for a component spec are tracked separately in _shared/spec-definition-of-done.html (Phase 3 — to be created). Cross-cutting topics (foundations, motion policy, breakpoints) live once in foundations/ — link to anchors there instead of repeating their content per component.

Lint checklist (block-merge gate, future)

This block lives only on the template page. Real component specs do not duplicate it — when a front-spec linter is added (Phase 3), it will read data-spec-section / data-spec-required attributes from the page and apply the rules below.

  • Body attributes: data-spec-page="component", data-spec-status, data-spec-owner on <body>.
  • Page-type badge in the H1 reads Component.
  • Every required section exists as <section data-spec-section="…" data-spec-required="true"> with non-empty content (no lone TODO, no empty placeholder).
  • Anatomy includes either an inline SVG diagram or a numbered parts list, with labelled regions matching the prose.
  • States & variants includes a visual matrix (table or figure) covering at minimum default, hover (or focus), active, disabled when applicable.
  • Accessibility covers keyboard, focus, ARIA roles/state, contrast, and motion.
  • Do / Don't has at least three pairs.
  • Code handoff names every JS module, CSS selector prefix, and DOM hook.
  • Page history section present with at least one entry.

4. Accessibility required

Keyboard, focus, ARIA, contrast, motion. All five must be addressed.

  • Keyboard: which keys reach and operate the component (Tab, Enter, Space, Esc, arrows).
  • Focus management: where focus moves on open/close; focus indicator contrast.
  • ARIA: role, aria-* state attributes, labelling. Prefer native semantics over ARIA; only add ARIA when the native element does not provide the role.
  • Contrast: text ≥ 4.5:1; non-text UI ≥ 3:1; focus rings ≥ 3:1 against adjacent colors.
  • Motion: behaviour under prefers-reduced-motion: reduce.
