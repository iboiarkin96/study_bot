Overview

The report-bug FAB is the always-on, page-scoped beacon for reader feedback. It sits at the bottom-right of every internal docs page as a small round button shaped like a beetle/bug, gently pulsing a radar ring that says "we're listening". A click opens a side-panel modal with a short form; submitting the form redirects to a prefilled GitHub issue against iboiarkin96/study_bot.

This FAB is the visible sibling of the back-to-top rocket — same launchpad palette, same right-edge stack, but a different personality: the rocket departs when the page is done; the bug listens the entire time. Together they form the two-FAB cluster shown above the footer.

Source-of-truth for the contract is this page; source-of-truth for the code is docs/assets/docs-nav.js (function injectDocsFeedbackCard()) and docs/assets/docs.css (selectors prefixed .docs-report-bug-fab* and .docs-feedback-modal*). Editorial governance (cadence workflow, GitHub template, triage labels) is described in Feedback and editorial workflow; this page is the UI spec for the floating control and the modal it opens.

Goals and non-goals

GoalsNon-goals
Stay visible on every internal docs page so any reader can report a defect without leaving the page. Replicate a generic chat widget — there is no inbox, no streaming, no realtime conversation.
Carry a calm, low-frequency motion ("we are listening") that does not compete with content. Spin or flash to demand attention. The FAB is ambient, not promotional.
Match the rocket FAB visually so the two controls read as a single cluster. Drift in shape, palette, or motion vocabulary from the back-to-top rocket.
Submit a clean, prefilled GitHub issue (page path, URL, type, body) so triage is one-click. Persist drafts, attach files, or call any internal write API — submission is GitHub-only.
Degrade quietly under prefers-reduced-motion: reduce and on mobile widths. Fall back to a compact alternative form on mobile that hides the GitHub redirect.

Anatomy

  1. Pill — round button (border-radius: 999px) painted in the launchpad palette: background: var(--card), 1 px var(--line) border, var(--card-shadow), text colour var(--muted).
  2. Bug glyph — inline SVG with three semantic groups: .bug-antenna--l, .bug-antenna--r (transformable on hover) and a single .bug-body group containing the body ellipse, the spine line, and three pairs of legs. Stroke 1.6 px, fill="none", currentColor.
  3. Radar pulse — purely cosmetic box-shadow animation (@keyframes docs-fab-radar): a 0–16 px expanding ring of accent that fades to transparent. No DOM nodes; no overflow concerns.
  4. Ambient antenna twitch — each antenna runs its own keyframe loop (@keyframes docs-bug-antenna-l-idle at 7.3 s; @keyframes docs-bug-antenna-r-idle at 9.1 s) so the left and right antennae flick at different phases and almost never lock into a pattern. Each loop spends ~95 % of its duration at 0° and produces two short flicks per cycle. Hover overrides the loop with the fixed ±12° lift; reduced-motion disables both loops entirely.
  5. Visually hidden label — .docs-report-bug-fab__label with the SR-only clip pattern. The accessible name comes from aria-label="Found a bug? Report issue"; the label is a fallback for environments that ignore aria-label.
  6. Tooltip hook — data-tooltip="Report issue" wires the FAB into the docs tooltip system. Tooltip contents are kept short because the FAB is icon-only.
  7. Open trigger — data-feedback-open="true". Any element on the page carrying this attribute opens the same modal; the FAB is the canonical one.
  8. Modal (lazy-rendered) — .docs-feedback-modal portal at document.body, right-aligned panel with a feedback-type select, textarea (≥ 10 chars), submit, status line, and toast. Detailed contract in Modal contract.

Why portal to <body>. Ancestors inside .internal-layout__main occasionally introduce transform/filter/contain properties, which create a containing block for position: fixed. That would anchor the FAB to the column edge instead of the viewport. The runtime reparents the FAB to document.body at injection time and dedupes any stray copies (keeps the first, removes the rest).

Visual reference

What the FAB looks like in situ. Left: a viewport mock with the bug pulsing above the rocket — the two-FAB launchpad cluster as readers see it. Right: a 4× detail of the glyph alone, with the antenna and leg groups labelled (these are the parts the hover state animates).

On mobile (≤ 740 px) both pills shrink to 38 × 38 px, the bug glyph to 18 px, and the stack gap to +58 px. The ambient antenna twitch keeps running on touch (it is idle, not hover-driven) — only the hover-driven behaviours (full ±12° antenna lift, body scale, faster radar pulse) are absent on tap; :active is the press feedback.

Why two coprime-ish loop durations. Real insects don't flick both antennae at once. The left antenna runs on a 7.3 s loop and the right on 9.1 s; their lowest common multiple is ~66 s, so the two animations rarely return to the same combined phase during a normal page visit. Each antenna spends roughly 95 % of its loop at 0° with two short flicks per cycle, so the bug looks alert without ever appearing busy or twitchy.

Positioning and stacking

The two FABs share the bottom-right corner. The bug always lives one slot above the rocket so the rocket is the first reachable target with a tap-up gesture and the bug sits within thumb arc on mobile. Sizing and offset are identical on every viewport — desktop, tablet, and mobile — per the uniform-typography-and-controls rule pinned in #mobile-contract. The companion rocket FAB is locked to the same numbers in rocket launch animation §positioning-and-stacking.

Property Value (uniform across all viewports)
Outer size 52 × 52 px — meets WCAG 2.1 Level AAA tap-target (≥ 44 × 44 px)
Bug icon size 28 × 28 px — matches .docs-back-to-top__rocket-wrap
right max(1.25rem, env(safe-area-inset-right, 0px)) — same as rocket
bottom (above rocket) calc(max(1.25rem, env(safe-area-inset-bottom, 0px)) + var(--docs-footer-visible-h, 0px) + 64px)
z-index 39 (one below the rocket at 40, both below modal/drawer)
Stack order with rocket Bug above, rocket below; 12 px gap between rocket top edge (≈ 72 px from screen bottom) and bug bottom edge (≈ 84 px from screen bottom).

The --docs-footer-visible-h variable lifts both FABs together when the site footer scrolls into view, so neither overlaps the footer card. env(safe-area-inset-*) respects iOS notches and home indicators (active when the device asks for it; otherwise the 1.25rem fallback wins).

History: until 2026-05-01 this page documented a smaller mobile variant (38 × 38 px at ≤ 740 px). That override was below the WCAG 44 × 44 tap-target floor and broke visual parity with the rocket. The mobile @media block in docs.css was removed; the bug FAB now inherits its desktop sizing on every viewport. See Do's and Don'ts: any new font-size / width / height rule for either FAB inside a width-conditional @media block is a review blocker.

States and accessibility

State Trigger Visual Motion Keyboard / ARIA
Pre-mount Before injectDocsFeedbackCard() runs. Hidden (opacity: 0, visibility: hidden, pointer-events: none). None. Not focusable.
Visible (idle) Class .docs-report-bug-fab--visible applied at end of mount. Fades in (translateY(8px) → 0, opacity 0 → 1, 0.22 s ease). Three ambient loops: radar pulse (docs-fab-radar 3.2 s, accent ring 0 → 16 px), left antenna flick (docs-bug-antenna-l-idle 7.3 s, two short twitches per cycle), right antenna flick (docs-bug-antenna-r-idle 9.1 s, twitches at different phases). Antennae rest at 0° for ~95 % of each cycle. Tab reaches it; aria-label="Found a bug? Report issue"; tooltip on hover/focus.
Hover Pointer over the pill. Border tints toward accent; colour shifts to var(--accent); pill nudges translateY(-2px). Pulse speeds up (animation-duration: 1.6 s); antennae rotate ±12°; body scales 1.04. Cursor: pointer.
Focus-visible Keyboard focus. 2 px outline of accent at 52 % alpha, 2 px offset. Same as hover (twitch + faster pulse). Enter/Space opens the modal.
Active (pressing) Pointer down or key down. translateY(-1px) scale(0.96), transition 0.08 s. — —
Reduced motion @media (prefers-reduced-motion: reduce). Static; no nudge on hover. Radar pulse disabled; both antenna idle loops disabled; antenna/body transitions disabled. Behaviour identical; only motion suppressed.

The FAB never carries an aria-expanded attribute — the modal is a separate dialog (role="dialog" aria-modal="true") at body level, not a popover anchored to the button. Screen readers announce "Found a bug? Report issue, button" on focus and "Page feedback form, dialog" on open.

Modal contract

One modal instance is created at mount time and reused for every open. The runtime never builds a second copy; opening while already open is a no-op (the second click toggles closed).

  1. Shape. .docs-feedback-modal covers the viewport with position: fixed; inset: 0; the panel (.docs-feedback-modal__panel) is right-aligned, max 440 px wide, max-height: calc(100vh - 2rem). On mobile (≤ 740 px) it switches to a bottom sheet: full width, max-height 85 vh, rounded top corners.
  2. Header. Heading "Report issue", a short paragraph explaining the GitHub redirect, and a top-right close button (data-feedback-close="true").
  3. Form fields.
    • <select id="docs-feedback-type"> — five categories, default Other: Incorrect or outdated content, Missing explanation, Broken link or navigation, Accessibility issue, Other.
    • <textarea id="docs-feedback-input"> — required, minLength=10, 4 rows, placeholder example. Live status text below explains current state.
  4. Submission. On submit:
    1. Trim message; if length < 10 → status flips to --error and focus returns to textarea.
    2. Build the GitHub URL with docsFeedbackIssueUrl(message, type).
    3. Best-effort navigator.clipboard.writeText(message) as a backup; failure is silent (file:// previews lack clipboard).
    4. Toast appears: "Opening GitHub with prefilled feedback…".
    5. Textarea cleared; window.location.href = targetUrl after 160 ms.
    6. After 900 ms, status resets to "Thanks!" and the modal closes if still mounted.
  5. Close affordances. Close button, backdrop click (anywhere outside the panel), Esc key (document-level listener). Each restores focus to lastFocusTarget.
  6. Body lock. While the modal is open, body.docs-feedback-modal-open sets overflow: hidden to prevent background scroll bleed.

GitHub issue payload

docsFeedbackIssueUrl() builds a deterministic https://github.com/iboiarkin96/study_bot/issues/new?… URL with these query parameters:

Param Value
template docs_feedback.md (so GitHub picks the right Issue Form).
title [Docs feedback] <rel-path-of-current-page>
labels docs-feedback
body Markdown with five sections: ## Page, ## URL, ## Feedback type, ## What should be improved, ## Additional context. The first three are filled automatically; the fourth is the user's textarea contents (or a placeholder if empty); the fifth is always a placeholder for screenshots/refs.

The page-actions menu's "Report issue" link uses the same builder with no message; only the modal submit path passes a non-empty extraFeedback. Cadence and triage of these issues are governed by .github/workflows/docs-feedback-cadence.yml — see GitHub feedback contract.

Do's and Don'ts

DoDon't
Keep the bug pill the same diameter as the rocket pill at every viewport: 52 × 52 px button, 28 × 28 px glyph, identical right inset and offset stack — desktop, tablet, mobile alike. They form a visual cluster; drift breaks the "two siblings on the launchpad" reading. Add a @media (max-width: …) rule that changes width, height, padding, or icon size on either FAB. Per #mobile-contract the FAB cluster is uniform across viewports — review blocker.
Reach for data-feedback-open="true" when adding inline "Report issue" links elsewhere on the page — the same modal handles them. Open the modal by mounting a second instance into the page DOM. There is exactly one modal; multiple instances duplicate listeners.
Verify the prefers-reduced-motion: reduce path on every visual change. Pulse, hover bobble, antenna twitch all must stop under reduce. Add motion-only feedback that the click was registered. The modal opening is the confirmation; the pulse is ambient, not informational.
Keep the SVG strictly currentColor + 1.6 px stroke. The bug must inherit the launchpad palette so theme switches just work. Hardcode hex colours into the SVG or rely on fill. Stroke-only currentColor is the contract.
Treat DOCS_FEEDBACK_CARD_ENABLED as a hard kill-switch for incidents (e.g. GitHub outage). Setting it to false hides the FAB entirely. Disable individual paths (clipboard, redirect) by editing the runtime — the kill-switch is one boolean, do not branch.
Let the runtime own the FAB end-to-end: injectDocsFeedbackCard() creates the button, glyph, and modal at document.body on every page. Pre-render <button class="docs-report-bug-fab"> in HTML or hand-author the SVG glyph. The static path is removed; baking the FAB into pages duplicates a contract that must be kept in lock-step.
If you need a containing block above the FAB (e.g. a popover), portal it to document.body too. The FAB lives at body root precisely to escape ancestor transforms. Wrap the FAB in a transform/filter/contain: layout ancestor. position: fixed will be trapped to that ancestor's box.

Failure modes and degradation

Condition Behaviour Mitigation in code
Page has no main.container. Injection is skipped; FAB never appears. Early return at the start of injectDocsFeedbackCard().
FAB is pre-rendered inside a transformed ancestor. Without mitigation: pinned to ancestor edge, not viewport. Runtime reparents to document.body and dedupes extras.
Multiple .docs-report-bug-fab on the page. The first is upgraded and kept; the rest are removed from the DOM. Loop after the reparent block.
Clipboard write rejected (e.g. file:// preview, denied permission). Submission still proceeds; the GitHub redirect is the source-of-truth. try/catch with empty handler.
User submits a textarea shorter than 10 chars. Status flips to --error, focus returns to textarea, no redirect. Length check before URL build.
GitHub is down. Page navigates and the user lands on the GitHub error page; their text is in the clipboard backup so they can retry later. Clipboard backup is intentional for this case.
DOCS_FEEDBACK_CARD_ENABLED = false. FAB and modal are not injected; no listeners; no DOM nodes. Single early return at the start of injectDocsFeedbackCard().

Acceptance checklist

  1. The FAB is visible at the bottom-right on every internal docs page after first paint, sized 52 × 52 px uniformly on every viewport (desktop, tablet, mobile). No @media-conditional shrink; per #mobile-contract this rule is enforced for the entire FAB cluster.
  2. The FAB sits exactly one slot above the back-to-top rocket with a fixed 12 px gap; both pills share the right margin (max(1.25rem, env(safe-area-inset-right, 0px))) and never overlap each other or the site footer.
  3. The radar pulse runs at ~3.2 s idle and accelerates to ~1.6 s on hover/focus.
  4. Each antenna runs its own ambient twitch loop (left 7.3 s, right 9.1 s); twitches occur at different phases and the antennae rest at 0° for the majority of each cycle. The two loops never lock into visible synchrony during a normal page visit.
  5. On hover, the ambient twitch is overridden: antennae rotate ±12° and the body scales 1.04; both reset (and twitch resumes) on pointer-leave.
  6. Under prefers-reduced-motion: reduce: no radar pulse, no antenna twitch, no hover lift — click behaviour is unchanged.
  7. Keyboard: Tab reaches the FAB once visible; Enter/Space opens the modal; Esc closes it; focus returns to the FAB.
  8. The modal panel appears on the right (or as a bottom sheet on mobile), focus moves to the textarea, and body scroll is locked while open.
  9. Submitting with < 10 characters surfaces an error and does not navigate.
  10. Submitting with valid text copies the text to the clipboard (when available) and redirects to https://github.com/iboiarkin96/study_bot/issues/new?… with template, title, labels, and body populated.
  11. The body markdown contains the page rel-path, the absolute URL, the chosen feedback type, the user's text, and an "Additional context" placeholder.
  12. Setting DOCS_FEEDBACK_CARD_ENABLED = false hides the FAB and prevents the modal from ever mounting.
  13. Ancestor transform/filter on the column does not pin the FAB — runtime reparents to <body>.
  14. Only one FAB exists in the DOM regardless of how many .docs-report-bug-fab elements the page declared.

Test plan

Smoke (manual)

  1. Open any internal docs page. The FAB is present and pulsing within one second of paint.
  2. Watch the bug for ~15 seconds without hovering: the antennae should flick briefly a few times, left and right at different moments — each spends most of its time still.
  3. Hover the FAB: ambient twitch is replaced by the full ±12° lift, pulse speeds up, colour brightens to accent. Move pointer away: antennae return to 0° and the ambient loop resumes.
  4. Click: the right-side panel opens and the textarea receives focus.
  5. Type "test feedback" + submit. A new GitHub Issue tab opens with the prefilled body.

Reduced motion (manual)

  1. Enable OS-level reduced motion (or DevTools emulation).
  2. Reload. The FAB is static — no radar pulse, no antenna twitch, no body wiggle. Hovering produces no motion either. Click still opens the modal.

Mobile breakpoint (manual)

  1. Resize to ≤ 760 px (iPhone Pro Max ≈ 430 px, iPhone SE ≈ 375 px presets cover the range).
  2. FAB stays 52 × 52 px — no shrink. The modal opens as a bottom sheet (rounded top corners, 85 vh max).
  3. Verify the FAB cluster sits above the iOS home indicator (env(safe-area-inset-bottom)).
  4. Visual parity: the bug FAB and the back-to-top rocket are identical in width, height, glyph size, and right inset on every viewport. Drift is a regression.

Accessibility (manual)

  1. Keyboard-only: Tab to FAB, Enter to open, Tab through fields, Esc to close. Focus returns to the FAB.
  2. Screen reader: announce "Found a bug? Report issue, button" on focus and "Page feedback form, dialog" on open.
  3. Submit invalid (< 10 chars): the live status text is read out as an error.

Containing-block trap (regression)

  1. Apply transform: translateZ(0) to .internal-layout__main in DevTools.
  2. Reload. The FAB must still anchor to the viewport corner — confirms the body-portal mitigation.

Kill-switch (regression)

  1. Toggle DOCS_FEEDBACK_CARD_ENABLED = false.
  2. Reload. No FAB in the DOM, no modal, no listeners. No console errors.
