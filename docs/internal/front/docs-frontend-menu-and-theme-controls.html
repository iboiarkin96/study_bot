Overview

This page is retained as the normative contract for navigation and theme behavior. It should contain stable requirements only, while implementation details are documented in dedicated frontend block pages.

  • Detailed implementation docs: navigation/search/discovery and UI/motion/adaptivity.
  • Scope here: product-level expectations and acceptance criteria.

Scope and breakpoints

Tiers below mirror the canonical viewport breakpoints — the single source of truth for device classes across docs.

  • Desktop: width >= 1025px.
  • Tablet: width 761...1024px.
  • Mobile: width <= 760px.
  • The long-form On this page aside is desktop-only.
  • Desktop TOC includes a compact collapse control: < to hide the list and > to restore it.

Navigation contract

  • Drawer opens from the left and is the primary menu model on tablet/phone internal pages.
  • Menu launcher is attached to the page title row on narrow internal layouts and hidden on desktop.
  • Desktop keeps Project and Code groups in top navigation; narrow layouts move them to drawer top sections.
  • Drawer interactions must preserve keyboard and screen-reader semantics (role, aria-modal, aria-expanded, and close behaviors).

Theme contract

  • Theme toggle is flashlight-style icon control.
  • Theme cycle order is automatic -> light -> dark -> automatic.
  • Explicit dark mode must show lit flashlight state.
  • Each switch shows a non-blocking top-centered status toast, with reduced-motion fallback.

Visual reference

Two contracts at a glance: the theme cycle (top — flashlight toggle stepping automatic → light → dark → automatic, each step announced by a top-centered toast) and the navigation contract by breakpoint (bottom — Project/Code groups live in top nav on desktop, fold into the drawer on tablet and mobile).

Why the dark state lights up. The flashlight is the icon's "off" affordance everywhere except in explicit dark mode, where the beam glows. Automatic and Light both leave the flashlight unlit because neither commits the reader to a dark scheme — only an explicit choice does. This keeps the icon legible at a glance without a separate label.

Why the drawer covers everything below 1025 px. Tablet and mobile both fold the Project and Code groups into the drawer, so the launcher in the title row is the single entry point to the navigation tree on those tiers. The drawer is portaled at z-index 1000 so it sits above the resume banner and the back-to-top FAB while it is open.

Quick actions policy

  • Quick actions and page actions are enabled on desktop widths and disabled on narrower widths.
  • Mobile/tablet navigation relies on drawer and standard links, not command palette.

Acceptance checklist

  1. Phone: Menu opens drawer, Project/Code available in drawer, quick actions hidden, desktop TOC aside hidden.
  2. Tablet: same model as phone for navigation and quick actions policy.
  3. Desktop: top groups visible, Menu hidden, flashlight in title row, desktop TOC aside visible when headings exist.
  4. Theme cycle, lit dark-state, and toast behavior are consistent across sections.

Implementation entrypoints

  • docs/assets/docs-nav.js — top nav render, theme cycle/toast, viewport sync.
  • docs/assets/internal-sidebar.js — internal drawer and sidebar runtime.
  • docs/assets/internal-layout.css — title row, drawer, and internal layout breakpoints.
  • docs/assets/docs-site-nav.css — top-nav groups and flashlight visuals.
