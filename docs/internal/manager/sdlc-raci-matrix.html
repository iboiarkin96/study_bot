SDLC RACI matrix by change type

Overview

This page defines a practical RACI baseline for the Study App delivery workflow. It maps ownership by SDLC stage and by change type, and is intended for planning, design review, and release alignment.

Scope: internal process governance for engineering and documentation work. Product contract details remain in System design, internal API docs, and ADRs.

Legend and usage rules

  • R (Responsible): performs the work.
  • A (Accountable): final approver and owner of outcome (single owner per row).
  • C (Consulted): provides input before decision.
  • I (Informed): notified after decision or delivery.

Escalation path for ownership conflicts: Tech Lead (technical) and Product Manager (scope/priorities). If unresolved, decision is logged in ADR or release notes.

Decision rules

  1. Each row must have exactly one accountable role (A).
  2. When technical risk is high, SEC and SRE move from C to mandatory approval input before release.
  3. Breaking changes require migration notes and release communication in docs and changelog before merge.
  4. Hotfix flow allows post-factum documentation within one business day after stabilization.
  5. Any unresolved ownership dispute is documented as a decision record with named owner and due date.
