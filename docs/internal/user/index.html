User resource (internal hub)

Entry point for engineering documentation of the User HTTP API: what the resource represents, where the contract is defined, and how to jump to operation-level detail. This mirrors how large cloud vendors structure docs — a resource hub plus deep specifications, with OpenAPI as the external contract (ADR 0025).

What this resource is

A User row is keyed externally by (system_user_id, system_uuid) and carries profile fields (name, timezone, invalidation metadata, and so on). The HTTP API under /api/v1/user creates and reads that identity for partner integrations. Runtime behavior, idempotency, and observability are documented in the internal spec linked below — not duplicated here.

Authoritative sources

  • External contract: OpenAPI — OpenAPI (test) or docs/openapi/openapi-baseline.json — field types, examples, and declared errors.
  • Internal specification (full): User HTTP API — internal specification — end-to-end: governance, API surface, layers, data model, idempotency, error mapping, logging, metrics, dependencies.
