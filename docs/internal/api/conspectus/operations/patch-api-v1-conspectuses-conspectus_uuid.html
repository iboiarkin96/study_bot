PATCH /api/v1/conspectuses/{conspectus_uuid}
Status Owner Ivan Boyarkin operationId patchConspectus Last reviewed 2026-04-29 Spec template v1

Overview

Partial update of an existing conspectus’s ETR content fields (title, cue_sheet, dense_paragraph, bullets). Does not change scheduling — for retrieve transitions see POST …/actions/review. Each successful PATCH appends a CONTENT_PATCHED row to conspectus_events and bumps content_version.

1. Business context

The error log workflow can drive a Transform-pass refinement: a learner discovers a weak cue or an unclear paragraph and updates the conspectus content. This endpoint captures that edit without disturbing the spaced-retrieval schedule.

2. User stories

  • US-CONSP-PATCH-1. As a learner, I want to fix a weak cue or rewrite a paragraph without losing my schedule progress.
  • US-CONSP-PATCH-2. As a product owner, I want every content edit auditable so I can analyse why retention regressed for a note.

4. Functional requirements

IDShort descriptionDetailed characteristic
FR-CONSP-PATCH-001 Resolve owner; verify ownership. Resolve user from body; verify conspectuses.owner_client_uuid = users.client_uuid; otherwise 404 CONS_404.
FR-CONSP-PATCH-002 Reject empty patches. Body must contain at least one of title, cue_sheet, dense_paragraph, bullets; otherwise 400 CONS_102.
FR-CONSP-PATCH-003 Patch only ETR content fields. Schedule, IDs, and timestamps cannot be patched. Unknown body fields → 422.
FR-CONSP-PATCH-004 Bump content_version and stamp updated_at. One DB transaction: UPDATE conspectuses with new field values, content_version = current + 1, updated_at = now(); INSERT conspectus_events (CONTENT_PATCHED) with diff.
FR-CONSP-PATCH-005 Enforce idempotency. Per shared rules; namespace PATCH /api/v1/conspectuses/{conspectus_uuid} with concrete UUID.
FR-CONSP-PATCH-006 Do not change scheduling. conspectus_schedules is read-only on this endpoint; schedule_revision stays the same.
FR-CONSP-PATCH-007 Validate field shapes. Same validators as create (cue_sheet object, paragraph 1–4000 chars, bullets 1–20 of 1–500 chars).

6. Authorization & permissions

API key per shared auth. Ownership rule: conspectuses.owner_client_uuid = resolved learner's client_uuid. Cross-tenant access → 404 CONS_404.

10. Side effects

  • On success. One transaction:
    1. UPDATE conspectuses SET fields, content_version = current + 1, updated_at = now().
    2. INSERT INTO conspectus_events (event_type="CONTENT_PATCHED", payload diff).
    3. INSERT INTO idempotency_keys.
    4. Emit conspectus_patched (info).
  • On idempotent replay. No DB writes; replay stored 200; emit conspectus_patch_idempotent_replay (info).
  • On error. Rollback; no partial state.
  • Schedule unchanged. No conspectus_schedules writes.

11. Idempotency & concurrency

  • Namespace: PATCH /api/v1/conspectuses/{conspectus_uuid} with concrete UUID.
  • Key composition: Idempotency-Key + SHA-256 of canonical body.
  • Replay status: 200 OK with stored body.
  • TTL: 24 h.
  • Concurrency: v1 — last writer wins on content fields (Open question Q1).

13. Edge cases checklist

  • Patch with all four fields. Accept; all fields replaced atomically.
  • Patch sets title="". Accept (clears the title).
  • Patch sets cue_sheet={}. Accept; empty object is valid.
  • Body has only owner fields. 400 CONS_102.
  • UUID exists but belongs to U1, request resolves to U2. 404 CONS_404.
  • Concurrent patches. v1 — last writer wins; content_version may have gaps from a client’s perspective.
  • Idempotency replay across content_version bump. Original stored body served; ignores subsequent edits.

17. Open questions

  • Q1. Add expected_content_version for explicit conflict detection (mirror of review’s expected_schedule_revision)? Current draft: no — patches are additive enough that last-writer-wins is acceptable.
  • Q2. Field-level merge for cue_sheet (deep merge by row index) vs whole-replace? v1 chose whole-replace for simplicity.

18. Out of scope

  • Schedule edits — go through POST …/actions/review or a future manual-reschedule endpoint.
  • Bulk patch.
  • JSON-Patch / RFC 6902 syntax — v1 uses simple field-level replace.
  • Field-level merge inside cue_sheet — see Q2.
