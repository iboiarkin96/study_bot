GET /api/v1/conspectuses/due

Conspectus

GET /api/v1/conspectuses/due
Status Owner Ivan Boyarkin operationId listDueConspectuses Last reviewed 2026-04-29 Spec template v1

Overview

List the learner's conspectuses whose next_review_at is at or before the given instant (default: now UTC), optionally filtered by slot. Read-only. Bounded result set per shared pagination rules (≤ 100 items, no cursor in v1).

1. Business context

The retrieval-practice loop in the methodology hinges on knowing what is due. This endpoint is the canonical answer to “what should I review now?” — temporal ordering, no pedagogical ranking, deterministic tiebreakers so the integrator's UI is reproducible (system-design · review suggestion).

2. User stories

  • US-CONSP-DUE-1. As a learner, I want to see the notes due for retrieval so I can pick the next session.
  • US-CONSP-DUE-2. As an integrator, I want a stable ordering so my pagination and UI rendering are deterministic.
  • US-CONSP-DUE-3. As a product owner, I want the evening-drill UX (random sample from Slot A) to be a client concern, not a server one — keeping the API explainable.

4. Functional requirements

IDShort descriptionDetailed characteristic
FR-CONSP-DUE-001 Resolve owner from query. Resolve users.client_uuid from (system_user_id, system_uuid) in query; 404 USER_404 when not found.
FR-CONSP-DUE-002 Filter by next_review_at. WHERE next_review_at <= due_before AND owner_client_uuid = ?; default due_before = now() UTC.
FR-CONSP-DUE-003 Optional slot filter. AND slot = ? when slot is present.
FR-CONSP-DUE-004 Deterministic ordering. ORDER BY next_review_at ASC, conspectuses.created_at ASC, conspectus_uuid ASC (per system-design).
FR-CONSP-DUE-005 Cap at 100. Hard LIMIT 100; integrator can re-poll with a tighter due_before to fetch the next slice (pagination).
FR-CONSP-DUE-006 Return bare array. Response is a JSON array of ConspectusResponse; no envelope, no total count.
FR-CONSP-DUE-007 Read-only. No DB writes; Idempotency-Key header silently ignored if sent.

5. Non-functional requirements

IDShort descriptionDetailed characteristic
NFR-CONSP-DUE-001Latency. P95 ≤ 150 ms, P99 ≤ 400 ms.
NFR-CONSP-DUE-002Index usage. Query covered by composite index (owner_client_uuid, next_review_at) on conspectus_schedules.
NFR-CONSP-DUE-003Cap. Hard LIMIT 100; cap is non-negotiable.
NFR-CONSP-DUE-004Read consistency. Default DB isolation; brief lag vs a just-committed write is acceptable.
NFR-CONSP-DUE-005Backwards compatibility. Adding new optional query params is additive; tightening the cap is breaking.
NFR-CONSP-DUE-006Observability. Standard middleware request_done; no operation-specific business event for reads.
NFR-CONSP-DUE-007Rate limit. Inherits global limit.
NFR-CONSP-DUE-008Cacheability. No cache headers in v1; due-lists change rapidly after a review.

6. Authorization & permissions

API key per shared auth. Learner resolved from query. Result is implicitly scoped to owner_client_uuid; cross-tenant data cannot leak.

13. Edge cases checklist

  • User has zero conspectuses. 200 [].
  • due_before in the past. Accept; returns items due before that point.
  • Calendar-day "today" semantics. Out of scope in v1; client computes its own day boundary using users.timezone if needed.
  • Slot=D filter with rung-specific query. Out of scope — only slot-level filter in v1.
  • 250 due items. Cap at 100; integrator polls again with a tighter due_before.
  • System clock skew. Server uses DB now(); not Python clock.
  • Idempotency-Key sent on this read. Silently ignored.
