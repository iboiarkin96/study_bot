Operation spec — Definition of Done

Internal — _shared

Overview

A spec is Done when every box below is ticked. The list is the single contract the systems analyst owes the developer before implementation starts; the linters in scripts/spec_lint.py and scripts/spec_consistency.py automate the structural checks, and a peer review confirms the content quality. PR cannot merge with a failing lint.

Use the template at _shared/spec-template.html as the starting point. Cross-cutting topics (idempotency, error envelope, auth, pagination, versioning, observability) are linked, never duplicated.

Definition of Done — checklist

Header & metadata

  • Status set to one of draft · in-review · approved · implemented · deprecated; matches the badge in the metadata strip.
  • Owner linked to a portal people page.
  • operationId present, unique, and consistent across body attribute and metadata strip.
  • Last reviewed date is ≤ 30 days old (refresh on each spec change).
  • HTTP method tablet + path displayed and matches the body operationId.

Sections — content

  • 1. Business context — 2–4 sentences anchored to methodology / system-design.
  • 2. User stories — at least one in “As a … I want … so that …” form.
  • 3. Acceptance criteria — every AC has an ID, Given / When / Then; happy + unhappy + at least one edge case covered.
  • 4. Functional requirements — every FR has an ID and is testable.
  • 5. Non-functional requirements — every NFR has a measurable threshold (latency ms, payload bytes, %, …).
  • 6. Authorization — links to _shared/auth.html; ownership rule explicit.
  • 7. Request contract — every parameter has type, format, required, constraints, business meaning, OpenAPI mapping; includes a JSON example.
  • 8. Response contract — same for each success status; example provided per status.
  • 9. State transitions — present if the operation mutates a state machine; otherwise removed (not left empty).
  • 10. Side effects — DB writes, events, downstream calls listed per outcome (success / failure / replay); use None. for read-only.
  • 11. Idempotency & concurrency — namespace, key composition, replay behaviour, concurrency model declared; links to _shared/idempotency.html.
  • 12. Error catalog — every code/key exists in _shared/error-catalog.html; envelope examples for non-trivial errors.
  • 13. Edge cases — at least 5 concrete cases; or explicit None.
  • 14. Observability — log events, metrics, traces, alerts per _shared/observability-conventions.html.
  • 15. Test plan — unit / integration / contract / e2e mapped to AC IDs.
  • 16. OpenAPI authoring hints — follows openapi-authoring-guide; names every reused component and example.
  • 17. Open questions — explicit list with owner + deadline, or literal None.
  • 18. Out of scope — explicit list, or literal None.
  • 19. Traceability — methodology / system-design / ADR / OpenAPI cross-links; at least one ADR.
  • 20. Spec changelog — first entry seeded; subsequent entries added on every contract-affecting change.
  • Page history — at least one row.

Quality gates

  • python scripts/spec_lint.py --paths <file> exits 0.
  • python scripts/spec_consistency.py exits 0 (no FAIL; warnings reviewed).
  • Page renders cleanly: no broken links, no missing diagrams, examples readable, in-page TOC populated.
  • make docs-fix changes nothing afterwards (no drift).
  • Reviewed by at least one developer who agrees the spec is implementable as written.

Status badge meanings

StatusMeaningWhat can change without an ADR
📝 Draft Authoring in progress. Not implementable yet. Anything.
👀 In review Submitted for peer review. Wording, examples; not the contract surface.
✅ Approved Reviewers signed off; implementation may start. Implementation-detail clarifications. Contract changes require a new review.
⭐ Implemented OpenAPI updated; tests green; live. Additive changes only (per versioning).
🚫 Deprecated Sunset announced; new clients should not adopt. Documentation / sunset clarifications only.

Running the lint locally

# Lint every operation spec:
python scripts/spec_lint.py

# Lint one spec:
python scripts/spec_lint.py --paths \
  docs/internal/api/conspectus/operations/post-api-v1-conspectuses-conspectus_uuid-actions-review.html

# Cross-document consistency (operationId, error catalog):
python scripts/spec_consistency.py

# Treat warnings as failures (CI mode):
python scripts/spec_consistency.py --strict-warnings
