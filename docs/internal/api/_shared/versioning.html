API versioning

Internal — _shared

Overview

Single source of truth for what counts as breaking, what is additive, and how new versions are rolled out. Governing ADR: 0004 — API versioning policy.

Scheme — major in URL

All public endpoints are mounted under /api/v{N}/.... N is a positive integer. The current major is v1.

  • Breaking changes require a new major path (/api/v2/...). The previous major continues to serve until the announced sunset.
  • Additive changes ship into the current major in place.
  • There is no minor or patch in the URL. Internal info.version in OpenAPI is bumped per release; externally that version is informational, not part of the path.

Breaking vs additive — matrix

ChangeClassWhy
Add a new endpoint additive Existing clients are unaffected.
Add an optional request field with a sensible default additive Old clients that omit it keep working.
Add a new field to a response object additive Clients are required to ignore unknown fields (a contract obligation).
Add a new value to an enum returned in a response breaking Existing clients’ exhaustive switches break.
Add a new value to an enum accepted in a request additive Old clients that don’t send it are fine.
Make an optional response field required additive Stronger guarantee — clients that already used it keep working.
Make a required response field optional breaking Clients that read it unconditionally crash on missing.
Rename a field breaking Always.
Change a field type or format breaking Almost always; even integer → string formatted breaks parsers.
Tighten validation (shorter max, stricter regex) breaking Previously valid clients get rejected.
Loosen validation (wider range) additive Old clients keep working.
Change the meaning of an existing field breaking Even when the type matches; semantic drift is the worst kind.
Change a 200 to a 201 (or vice versa) breaking Clients branch on status code.
Add a new error code to an existing endpoint additive Provided the HTTP status class stays the same and the envelope shape is unchanged.
Move from bare array to wrapped list response breaking See pagination.

Rolling out a new major

  1. RFC documenting the breaking changes; link from the new major’s OpenAPI info.description.
  2. Mount /api/v2/... alongside /api/v1/...; both serve live traffic.
  3. Announce sunset for v1 with a date — minimum 6 months for paying integrators (per ADR 0004).
  4. Mark v1 operations as deprecated: true in OpenAPI; add Sunset header on responses.
  5. Track usage; remove v1 after sunset. Document the removal in the release notes.

Deprecating an endpoint inside a major

Endpoints can be deprecated without a major bump if the change is removal of an optional capability:

  • Mark deprecated: true in OpenAPI.
  • Set the Sunset response header (RFC 8594) to the planned removal date.
  • Update internal spec status to deprecated.
  • Removal is itself a breaking change → goes with the next major.

Release notes

The internal spec changelog (Section 20 of an operation page) is per-operation and machine-trackable. The public release notes live separately and bundle multiple endpoints into one human narrative — see ADR 0013.
