Error catalog — shared registry

Internal — _shared

Overview

Authoritative registry of every code / key pair the API may return. Every entry on a per-operation Error catalog (Section 12) must also appear here — the consistency lint blocks the PR otherwise. Code home: app/errors/<domain>.py. Envelope shape: _shared/error-envelope.html.

Note. The legacy auto-generated catalog also lives at ../errors.html (synced from app/errors/ via scripts/sync_docs.py). This page is the analyst-facing view: codes are organised by allocation policy, not by source file. Both views must agree — the lint compares them.

Allocation policy

  • Domain prefixes. COMMON_ (cross-cutting), USER_, CONS_ (conspectus), ERR_ (error log). New domains require an ADR.
  • HTTP-aligned 3-digit suffix (e.g. COMMON_400) for codes that map directly to an HTTP status. Reserved range: 000 (validation), 4xx, 5xx.
  • Domain-specific 3-digit suffix in the 1xx range (e.g. USER_101, CONS_102) for business-logic errors that would otherwise share an HTTP status with a generic code.
  • Append-only. Numbers are never reused. Retired codes stay in the catalog with a deprecated badge.

Adding a new error

  1. Pick a domain (existing prefix or new ADR).
  2. Allocate the next free numeric in app/errors/<domain>.py.
  3. Define the constants and add an OpenAPI example in app/openapi/examples/errors.py.
  4. Add a row to the appropriate section above and to the per-operation Error catalog.
  5. Run make docs-fix — markers in ../errors.html regenerate from code.
  6. Run make docs-check / make verify — consistency lint cross-checks this page, the per-operation catalog, and the auto-generated table.
