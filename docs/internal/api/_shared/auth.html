Authentication & authorization

Internal — _shared

Overview

Single source of truth for how the internal API authenticates integrators and authorises access to per-tenant resources. Per-operation specs link here for the global scheme and only describe operation-specific deltas (extra ownership rules, scopes). Governing ADR: ADR 0005.

Scheme — API key

All endpoints under /api/v1/** require a valid API key in the X-API-Key header. The key identifies the integrator (one third-party tenant). The actor inside that tenant — the learner — is identified separately by (system_user_id, system_uuid) in the body or query, depending on the operation.

X-API-Key string required

Opaque ASCII secret issued out-of-band. Treated as bearer-equivalent — keep it server-side, never embed in mobile/web bundles.

Format
Printable ASCII; conventional prefix sk_live_ / sk_test_.
Length
Implementation-defined ≥ 32 chars.
Rotation
Operator-driven; the API does not currently support overlapping rotation in v1 — short downtime expected (Open question for v2).

Out of scope in v1: OAuth 2.0, JWT bearer tokens, mTLS, IP allow-listing, session cookies. Adding any of these requires an ADR.

Actor resolution: integrator vs learner

The two-layer identity model is deliberate: one API key may serve many learners (an integrator is a school, a tutor app, a Telegram Mini App, …).

  • The integrator is resolved from X-API-Key by the auth middleware before the handler runs.
  • The learner is resolved from (system_user_id, system_uuid) in the body or path/query, by the service layer after validation. Resolution returns users.client_uuid; this UUID is the canonical owner key on every row owned by the learner.
  • If the user composite key does not resolve, the operation returns 404 USER_404 USER_NOT_FOUND. Always — never 403 (do not leak existence).

Authorization — ownership

v1 uses a single authorisation rule: ownership. The resolved learner must own every resource the operation touches.

  • Conspectus: conspectuses.owner_client_uuid = users.client_uuid.
  • Error log entry: learning_errors.owner_client_uuid = users.client_uuid; if a referenced conspectus_uuid is set, it must also be owned by the same learner.
  • Schedule: belongs to a conspectus; ownership transitively via conspectuses.

Cross-tenant access (resource exists but belongs to a different learner): always return 404 <DOMAIN>_404. Never 403. The integrator gets the same response whether the resource does not exist or belongs to someone else — by design.

No admin override. v1 has no super-user role. Operator access to production data goes through DB-level audited queries, not the API.

Errors emitted by the auth layer

See _shared/error-envelope.html for the shape; registered in _shared/error-catalog.html.

HTTPCodeKeyWhen
401COMMON_401 SECURITY_AUTH_REQUIRED Missing, malformed, or revoked X-API-Key.
404USER_404 USER_NOT_FOUND Learner composite key does not resolve OR resource not owned by this learner (collapsed to 404 on purpose).

Rate limiting (related)

Rate limits are enforced per X-API-Key by middleware (COMMON_429 SECURITY_RATE_LIMIT_EXCEEDED). The default budget and the sliding-window definition live in ADR 0005. Per-endpoint override requires its own ADR; v1 has none.

Forbidden patterns

  • Returning 403 for cross-tenant access. Always 404 — even if you know the resource exists.
  • Including the API key in URLs / query strings. Header only.
  • Logging the API key. Mask to last 4 chars in any log line.
  • Trusting client-supplied owner_client_uuid in a body. The server resolves it from (system_user_id, system_uuid); ignore any body field that tries to set ownership directly.
