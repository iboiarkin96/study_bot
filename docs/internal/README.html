Internal service documentation

Engineering-facing documentation for the Study App HTTP service: how we work, which policies apply, and where to find API narrative beyond OpenAPI. Use the left navigation to open the API hub, resource hubs, and per-method pages — content is split across pages on purpose so readers are not faced with a single endless document.

Contributors: folder layout and how to add or edit internal HTML pages — How-to: internal service docs layout (STRUCTURE.md is a short pointer to the same content).

Product and teams (working model)

Study App is a learning platform prototype; the repo is structured for clarity of ownership and review. The following squads are a logical split — in a small team one person may wear several hats:

  • Product API — HTTP surface, domain rules, contract tests, and internal specs for resources and operations. Owns alignment between OpenAPI, runtime behavior, and these HTML pages.
  • Platform & reliability — deployment topology, observability defaults, SLOs, and runbooks; collaborates with API on error envelopes and logging fields.
  • Developer experience — docs-as-code in docs/, ADR workflow, local dev and Make targets; keeps navigation and templates maintainable.

Documentation and development practices

  • Contract-first: docs/openapi/openapi-baseline.json (and live /openapi.json) is the sole external contract for paths, schemas, and declared errors.
  • Change behavior and docs together: meaningful HTTP or semantics changes ship with updates to OpenAPI, internal pages, and tests in the same maintenance window (see developer guides).
  • Split by page: landing (this page) for orientation; API hub per surface; resource hub for business + data contract; one page per method when operation-level detail would overload the resource page.
  • User resource: business context, contract summary, and full technical narrative live on one page — User resource — internal specification; method pages link to anchors there.

Audience and authority

Who this is for: contributors and maintainers in this repository. External integrators should rely on OpenAPI and release notes.

Authority: For topics documented under docs/internal/, these pages are the normative narrative where they explicitly describe behavior, per ADR 0026. Global policies remain in ADRs.
