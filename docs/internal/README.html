Role and scope

The ETR Study API is a FastAPI application that provides a stable, documented HTTP surface for clients.

Architectural boundaries

Layered architecture — theoretical background

Layered architecture groups the system into horizontal layers. Each layer exposes a narrow set of capabilities to the layer above and depends on the layer below. The usual dependency rule is: inward-only dependencies — inner layers (domain rules, persistence abstractions) must not depend on outer concerns (HTTP framework details, transport), so business logic stays testable and replaceable.

This pattern is related to ports and adapters (hexagonal architecture): the “application” core defines what the system does; adapters (HTTP, SQL) translate between the outside world and that core.

Contract-first means the HTTP surface is described explicitly (OpenAPI) before or alongside implementation; code must conform. That fits layered boundaries because the outer “adapter” is governed by a stable contract rather than growing ad hoc.

We use layers: HTTP routes, then business logic, then the database. The public API is described first in OpenAPI (“contract-first”): what you see in the spec is what callers should rely on.

Where checks happen (in order):

  1. Field rules for JSON bodies are defined in Pydantic models under app/schemas/ (required fields, types, max length, and so on).
  2. When a request arrives, FastAPI runs those checks before code in app/api/ runs. Handlers usually receive data that already passed this step.
  3. If a check fails, app/validation/ maps the failure to our standard 422 response (code, key, message). It does not redefine the rules—only the error shape for clients. That wiring is registered in app/main.py.
  4. Rules that need the database—such as “this user already exists”—are enforced in app/services/ and surface as business client errors, not as “invalid JSON field” / schema validation. See ADR 0003.

Under normal operation the call chain is: API → services → repositories → database. The table lists who does what; the diagram below is a simplified picture of that path.

Contract and integrators

External developers must use the OpenAPI document as the contract: committed snapshot at docs/openapi/openapi-baseline.json, explorer at OpenAPI (test), live Swagger when the app runs at /docs. Release-to-release changes are summarized in the repository root CHANGELOG.md (ADR 0013).

Versioning rules (URL major version, additive changes within a major) follow ADR 0004.

Observability and operations

Health and metrics expectations follow ADR 0009. Structured logging and request correlation follow ADR 0023.

Metrics in this project are exposed via the /metrics endpoint and are compatible with Prometheus. We collect application-level and HTTP endpoint metrics using prometheus_fastapi_instrumentator. Examples of captured metrics include:

  • Request counts and response time histograms per endpoint and HTTP method
  • HTTP error and exception rates
  • Process uptime, memory, and basic resource usage

Key metrics are scraped by the Prometheus server configured per environment. These include both high-level health and detailed per-handler operations, which are used for real-time monitoring, SLA checks, and alerting. Metric labels (endpoint/method/status code) enable dashboards and queries for per-route trends and troubleshooting. For metric customizations or additions, see app/observability/metrics.py (or equivalent).

Operational recovery steps belong in Runbooks when they are procedural.
