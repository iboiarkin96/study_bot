Internal service documentation

This section holds engineering-facing documentation for the Study App service: business context, behavioral semantics, and operational expectations. It complements — but does not replace — the public HTTP contract in OpenAPI.

Audience and authority

Who this is for: contributors and maintainers working in this repository. External integrators should rely on the OpenAPI snapshot and release notes instead; see ADR 0025.

Authority: For topics explicitly documented here, these pages are the normative narrative for how the team intends the system to behave, per ADR 0026. Policy-level decisions (versioning, security, global error rules) remain in ADRs.

Relationship to OpenAPI and code

  • OpenAPI (docs/openapi/openapi-baseline.json, live /openapi.json) is the sole external contract for paths, schemas, and declared errors.
  • Internal pages explain why and how it behaves in production terms — rules, edge cases, monitoring hooks — without duplicating full schema definitions.
  • Code must implement both; if internal docs and runtime disagree, treat it as a bug and fix docs or code in the same maintenance spirit as contract tests.

Documentation map

This tree follows a vendor-style information architecture: platform context first, then resource hubs (what the thing is, links to the contract), then deep specifications — behavior, idempotency, error and logging semantics, metrics — either on one page or split per method when the API grows. OpenAPI stays the only external contract; internal pages are the engineering narrative (ADR 0026).
