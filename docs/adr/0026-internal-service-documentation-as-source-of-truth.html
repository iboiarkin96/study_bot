ADR 0026: Internal service documentation as source of truth

Ratification

This ADR records norms for internal HTML documentation under docs/internal/: what must appear there, how pages relate to code and OpenAPI, and how we preserve history. Complements ADR 0025 (audience split). The audit trail is the git history for this file on main.

Context

Why this matters: External integrators rely on OpenAPI (ADR 0007). Internally, we still need a single, reviewable place for rules that span layers: business invariants, when an error is considered operational vs validation, how a flow is expected to be monitored, and how HTTP handlers relate to domain services and (if any) asynchronous boundaries. Without that, disagreements default to “read the implementation,” which is slow and version-ambiguous.

This ADR makes docs/internal/ the authoritative narrative for engineering for those topics. It does not duplicate OpenAPI field lists; it interprets and binds them to behavior and operations.

Decision

  1. Location and format. Internal service documentation lives under docs/internal/ as static English HTML using the shared stack (docs/assets/docs.css, docs/assets/docs-nav.js). New pages are added or updated in the same pull request as the behavior change when the change is user-visible or alters invariants described here.
  2. Source of truth for engineering disputes. For topics explicitly covered in docs/internal/, the written page (on main after merge) is the normative reference for how the team intends the system to work. If code disagrees, either the code or the doc must be fixed in a follow-up; silent divergence is treated as a defect. Topics not yet documented remain governed by ADRs, runbooks, and code until a page exists.
  3. What to document (non-exhaustive). Pages should cover, as applicable:
    • Capability or domain slice — purpose, boundaries, key entities.
    • HTTP mapping — reference operationIds or paths that implement the slice; link to OpenAPI rather than copying schemas.
    • Business rules — invariants, ordering assumptions, idempotency and conflict behavior (see also ADR 0006).
    • Errors and observability — which codes or signals warrant alerts vs expected client mistakes; correlation with logs and metrics (see ADR 0003, ADR 0009, ADR 0023).
    • Async and integration — message topics, consumers, outbox patterns, or external systems, when present.
  4. History and traceability. Material pages include a short Document history subsection (table: date, author optional, summary of change). Git history remains the full audit trail; the table is for human scanning. Breaking behavioral changes also appear in root CHANGELOG.md per ADR 0013.
  5. Index. docs/internal/README.html is the entry point: scope, links to ADR 0025/0026, and a curated list of internal pages.
  6. Relationship to ADRs. Cross-cutting policy (versioning, security, idempotency) stays in ADRs. docs/internal/ applies those policies to concrete flows and entities. If a new policy is needed, add or update an ADR; do not bury policy-only decisions in internal prose alone.

Scope

  • In scope: Structure, authority, and minimum expectations for docs/internal/ HTML.
  • Out of scope: Replacing OpenAPI, replacing pdoc under docs/api/, or duplicating runbook procedures (link to docs/runbooks/ instead).

Alternatives considered

  1. Wiki or external tool.
    • Pros: rich editors, permissions.
    • Cons: not versioned with the same PR as code; harder to enforce review alongside behavior changes.
  2. Markdown only under docs/internal/.
    • Pros: faster to write.
    • Cons: this site’s published docs are HTML-first; one stack keeps nav and styling consistent.

Consequences

Positive

  • Engineers can resolve “what did we agree?” without inferring from code alone for documented areas.
  • Onboarding gains a bounded reading path starting at the internal README.

Trade-offs

  • Ongoing maintenance: pages must be updated when behavior changes, or authority of the doc set erodes.
