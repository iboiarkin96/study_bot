ADR 0022: Embedded Swagger UI — static OpenAPI request validation

How to set valid status (click to open)

On <main>, set a single attribute data-adr-weight to the current milestone (−1 … 7). The page derives Current status and paints the Status log from that one number (values are not shown as plain text on the page).

Which number means which step (single global order 0 → 7)
Value Meaning (current milestone)
-1 Status not set (or unknown)
0 Decision — Proposed
1 Decision — Accepted
2 Decision — Superseded
3 Documentation — Draft
4 Documentation — Ready
5 Implementation — Not started
6 Implementation — In progress
7 Implementation — Done

Example: data-adr-weight="4" → Current status is Documentation · Ready; earlier steps in the log are green, later steps gray. Implementation: Done (7) uses green when it is the current value. Out-of-range values are clamped in docs/assets/docs-nav.js.

Context

The documentation site embeds Swagger UI (docs/openapi-explorer.html) against the committed contract snapshot docs/openapi/openapi-baseline.json (see ADR 0007). Contributors and readers often want to explore operations without running the API, database, or network — for example from static hosting or a local file:// page.

Standard Swagger UI Try it out sends real HTTP requests. This embedded page instead only validates JSON bodies against the contract: no network to configured servers.

Decision

The embedded explorer (docs/openapi-explorer.html) always runs in validation-only mode: it validates application/json request bodies using Ajv (2020 dialect, formats) against schemas from openapi-baseline.json, via interceptors — no toggle, no live API from this page. Implementation lives in docs/assets/openapi-sandbox.js (ES module) and uses:

  • @apidevtools/json-schema-ref-parser to dereference a clone of the spec for validation (concrete request-body schemas for matching operations).
  • requestInterceptor / responseInterceptor on Swagger UI to short-circuit Try it out with a synthetic response when validation succeeds, and to normalize the displayed response so Swagger does not mis-render JSON against operation response schemas.
  • Bypass for requests that reload the OpenAPI document itself (same URL Swagger uses for url: loading), so that fetch is not mistaken for an API call.

Out of scope: validating headers, path parameters, query strings, or response bodies. Real HTTP Try it out remains on the running service (e.g. FastAPI /docs), not on this static page.

Swagger UI loading model (contract URL)

Internal #/components/schemas/… references must resolve against the OpenAPI document, not the HTML page URL. The implementation therefore prefers loading the contract via url (relative openapi/openapi-baseline.json on http(s)). For file:// pages, resolving refs via literal file:///…/openapi-baseline.json#/… can fail inside Swagger’s resolver (“Evaluation failed on URI”); the code supplies the same JSON through a blob: object URL so fragment refs resolve in memory.

A fully dereferenced (inlined) document is not passed into Swagger UI: it has been observed to break rendering of the documented Responses section (e.g. React error responses_Responses).

Implementation status

Implementation is in progress (data-adr-weight="6"). Core pieces (interceptors, Ajv, path template matching, blob URL for file://, spec reload bypass) are in place; behavior remains sensitive to Swagger UI version and browser.

Known issues and open risks

Not all combinations of page origin (file:// vs http(s)), Swagger UI patch version, and browser have been fully validated. Despite mitigations above, users may still see resolver warnings, failed rendering in the Responses panel, or console errors that we have not fully eliminated. Treat this page as a best-effort helper for contract exploration, not a second test harness.

  • Contract truth remains make contract-test / CI gates on openapi-baseline.json; the browser validator does not replace server-side validation.

Alternatives considered

  1. Only host Swagger at the live API (/docs) — simpler, but no offline/static-site exploration of the committed snapshot.
  2. Separate CLI or desktop validator — stronger guarantees, higher friction for readers.
  3. Inline spec: only — breaks $ref base on file:// (refs anchor to the HTML file).

Consequences

Positive

  • Readers can sanity-check JSON bodies against the published contract without infrastructure.
  • Aligns narrative docs with the same openapi-baseline.json enforced in CI.

Trade-offs

  • Extra client-side JS, CDN imports (Swagger bundle, Ajv, ref-parser), and maintenance when upgrading Swagger UI.
  • Browser-only validation is not a substitute for automated contract tests.

Compatibility and migration

  • Backward-compatibility impact: none for the API; documentation-only behavior.
  • Rollback: revert or disable openapi-sandbox.js wiring in openapi-explorer.html.

Validation

  • Manual: open docs/openapi-explorer.html, run Try it out on JSON operations; confirm no unintended network calls to the API (only spec load / blob as configured).
  • Contract gates unchanged: make contract-test / CI.
