ADR 0022: Embedded Swagger — static OpenAPI validation (superseded)

How to set valid status (click to open)

On <main>, set a single attribute data-adr-weight to the current milestone (−1 … 7). The page derives Current status and paints the Status log from that one number (values are not shown as plain text on the page).

Which number means which step (single global order 0 → 7)
Value Meaning (current milestone)
-1 Status not set (or unknown)
0 Decision — Proposed
1 Decision — Accepted
2 Decision — Superseded
3 Documentation — Draft
4 Documentation — Ready
5 Implementation — Not started
6 Implementation — In progress
7 Implementation — Done

Example: data-adr-weight="4" → Current status is Documentation · Ready; earlier steps in the log are green, later steps gray. Implementation: Done (7) uses green when it is the current value. Out-of-range values are clamped in docs/assets/docs-nav.js.

Context (historical)

The documentation site was to embed Swagger UI against docs/openapi/openapi-baseline.json (see ADR 0007) so readers could explore the contract without running the API. An additional goal was browser-side validation of JSON request bodies (Ajv, interceptors, no real HTTP to the API from that page).

Superseded decision

The approach described in earlier drafts of this ADR — static validation of application/json bodies in the browser via Ajv, Swagger requestInterceptor / responseInterceptor, and related client-side wiring — is cancelled as a product commitment. It proved brittle (CDN loading, file://, Swagger UI versions, ref resolution) and did not deliver a reliable experience.

What was implemented instead

docs/openapi-explorer.html is labeled OpenAPI (test) in the docs nav. It loads Swagger UI from the CDN and points at the committed openapi-baseline.json for visual exploration only: operations, schemas, and documentation text. Try it out is disabled on this page. No validation runs in the browser; there is no Ajv/ref-parser bundle and no openapi-sandbox.js.

For real HTTP Try it out against a running server, use the application’s interactive docs at /docs (configure CORS for your browser origin).

On hold

A future iteration may revisit browser-side or docs-integrated contract validation (static or against a mock), with clearer scope and tooling. Until then the task is on hold. Contract correctness remains enforced by make contract-test / CI on openapi-baseline.json, not by this page.

Consequences

  • Readers still get a quick visual view of the OpenAPI snapshot from the static site.
  • No claim of client-side validation; no maintenance burden for the previous interceptor/Ajv stack.
  • ADR 0007 / governance of the baseline file is unchanged.
