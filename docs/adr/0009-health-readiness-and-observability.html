ADR 0009: Health, Readiness, and Prometheus Observability

How to set valid status (click to open)

On <main>, set a single attribute data-adr-weight to the current milestone (−1 … 7). The page derives Current status and paints the Status log from that one number (values are not shown as plain text on the page).

Which number means which step (single global order 0 → 7)
Value Meaning (current milestone)
-1 Status not set (or unknown)
0 Decision — Proposed
1 Decision — Accepted
2 Decision — Superseded
3 Documentation — Draft
4 Documentation — Ready
5 Implementation — Not started
6 Implementation — In progress
7 Implementation — Done

Example: data-adr-weight="4" → Current status is Documentation · Ready; earlier steps in the log are green, later steps gray. Implementation: Done (7) uses green when it is the current value. Out-of-range values are clamped in docs/assets/docs-nav.js.

Ratification

Adopted before ADR 0018; no separate ratification trail was required then. Git history for this file records merges into main.

Context

The service had a basic /health endpoint but no explicit readiness probe and no standardized metrics export for latency/error-rate/DB latency in staging and production.

Decision

  • Introduce /live for process liveness checks.
  • Introduce /ready for dependency readiness checks (including DB probe).
  • Expose Prometheus metrics via /metrics.
  • Collect API latency, HTTP status counters, and DB operation latency with low-cardinality labels.
  • Adopt OpenAPI snapshot contract tests (make contract-test) in the quality gate.

Implementation

  • Health/readiness endpoints in app/main.py and schemas in app/schemas/system.py.
  • Metrics collector and SQLAlchemy hooks in app/core/metrics.py.
  • Config toggles and thresholds in app/core/config.py and env/example.
  • Operational commands in Makefile and observability stack in docker-compose.observability.yml.

Operational Model

  • Prometheus uses pull-based scraping from /metrics.
  • Dockerized Prometheus + Grafana is the default local/staging bootstrap path.
  • Production deployment may run the same stack outside Docker (VM/Kubernetes) with the same endpoint contract.

Consequences

Positive

  • Faster diagnosis of incidents via explicit readiness and standardized metrics.
  • Reduced risk of accidental contract drift through snapshot tests.
  • Consistent observability baseline from local to staging/production.

Trade-offs

  • Additional maintenance for metrics and dashboards.
  • Need to enforce low-cardinality labels and avoid exposing sensitive dimensions.
