ADR 0018: ADR lifecycle, ratification, and status badges

Status

Decision: Accepted Implementation: Done Documentation: Ready

Ratification

Bootstrap: this ADR introduces the ratification block; ADRs numbered 0017 and below were adopted under earlier conventions. From ADR 0018 onward, new decisions should link to their discussion Issue and merge PR here.

Context

The project uses HTML ADRs under docs/adr/ and publishes them with the rest of the site (for example GitHub Pages). Contributors need a clear split between decision accepted, implementation progress, and documentation polish, plus a lightweight way to record discussion outcomes without interactive voting in static HTML.

Decision

  1. Each ADR exposes three axes in the Status section using shared CSS badges (see docs/assets/docs.css): Decision (Proposed, Accepted, Superseded), Implementation (Not started, In progress, Done), and Documentation (Draft, Ready). Accepted decisions use the success (green) badge style.
  2. Ratification: for new decisions after this ADR, record links to the GitHub discussion Issue and the merge PR, and the acceptance date, in a dedicated section (see docs/adr/0000-template.html). Consensus is reached in the Issue; the merged ADR is the published record.
  3. Documentation changelog: notable changes under docs/ are summarized in docs/CHANGELOG.md, separate from the root CHANGELOG.md (product/API focus per ADR 0013).
  4. Documentation-only delivery: pull requests that only change documentation (including ADRs) may merge to main before implementation code; implementation follows in separate PRs.

Consequences

Positive

  • Readers see decision vs implementation vs doc readiness at a glance.
  • Discussion and votes remain auditable in GitHub Issues and PRs.
  • Doc history is traceable without overloading the product changelog.

Trade-offs

  • Badge lines must be updated manually when status changes.
  • Older ADRs may lack a Ratification block until edited; they remain valid as adopted under prior process.
