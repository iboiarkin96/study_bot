ADR 0018: ADR lifecycle, status model, and ratification

ADR status on this page

The lifecycle bar uses data-adr-weight on <main> (values −1…7). See ADR 0018 for what each step means, or the ADR template for the full milestone table.

Ratification

This ADR establishes the canonical lifecycle model for ADRs in this repository. Record the discussion Issue (opened with the ADR template; title starts with [ADR]), merge PR, and acceptance date here when the ratification thread is complete.

  • Discussion Issue: see repository issues
  • Merge PR: see git history for this file
  • Accepted: 2026-04-12

Context

Why this matters: An ADR is useless if readers cannot tell whether it is a draft idea, an accepted policy, or fully implemented. A tiny, machine-readable status on each page keeps the docs site honest without maintaining a second spreadsheet of “ADR state.”

Records live as static HTML under docs/adr/. We need one lifecycle: what was decided, how it was ratified (Issue + PR), and progress through documentation and implementation—without inventing a new ticket system only for status.

This ADR defines the canonical status model. It uses one ordered scale (-1 … 7) covering Decision (0–2), Documentation (3–4), and Implementation (5–7). Authors set a single attribute on <main>: data-adr-weight. That value is the current milestone; Current status and the Status log are derived from it (green = passed; yellow = current step except Implementation: Done, which stays green when current; gray = not yet). Rendering is in docs/assets/docs-nav.js (renderAdr).

The ADR HTML template is the author-facing shell: data-adr-weight, optional collapsible value table, and standard sections (Ratification, Context, Decision, …). New ADRs start from that file; they apply the model—they do not redefine it.

How we mark ADR work

Three layers — do not mix them up:

  1. GitHub Issue (discussion). Use the template .github/ISSUE_TEMPLATE/adr_discussion.md. Start the title with [ADR] so these threads are easy to find. Discussion and consensus stay in the Issue. We do not depend on a big label system; the template plus title prefix are the convention.
  2. ADR page on the site (lifecycle). Set data-adr-weight on <main> so readers see draft → done in the docs UI. That is published status only; it does not replace the Issue and PR history.
  3. Ratification. In the merged ADR, record links to this Issue, the merge PR, and the acceptance date — that binds the HTML record to the audit trail.

Roadmap rows in docs/backlog/ use their own status pills. Branch names follow ADR 0017. Those are unrelated to data-adr-weight.

Decision

  1. Status model (normative here). The milestone table is fixed: Decision — Proposed (0), Accepted (1), Superseded (2); Documentation — Draft (3), Ready (4); Implementation — Not started (5), In progress (6), Done (7). Invalid data-adr-weight values are clamped in script. The model was introduced with this ADR; older ADRs may predate it until migrated.
  2. Ratification. For decisions adopted after this ADR, record the discussion Issue (opened with the ADR template and [ADR] title), merge PR, and acceptance date in the ADR’s Ratification section (see the template). ADRs numbered 0017 and below were adopted under earlier conventions; from 0018 onward, this pattern applies.
  3. Documentation changelog. Notable changes under docs/ are summarized in docs/CHANGELOG.md, separate from the root CHANGELOG.md (product/API focus per ADR 0013).
  4. Documentation-only delivery. Pull requests that only change documentation (including ADRs) may merge to main before implementation code; implementation follows in separate PRs.

Consequences

Positive

  • One agreed status semantics for all ADRs; the template and JS stay aligned.
  • Readers see current milestone and full log without leaving the page.
  • Discussion and ratification remain auditable via Issues and PRs.

Trade-offs

  • Authors must update data-adr-weight on <main> when the milestone changes.
  • Legacy ADRs may lack weights or ratification blocks until edited.
