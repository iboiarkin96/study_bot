ADR 0015: Container Image and Local Kubernetes

How to set valid status (click to open)

On <main>, set a single attribute data-adr-weight to the current milestone (−1 … 7). The page derives Current status and paints the Status log from that one number (values are not shown as plain text on the page).

Which number means which step (single global order 0 → 7)
Value Meaning (current milestone)
-1 Status not set (or unknown)
0 Decision — Proposed
1 Decision — Accepted
2 Decision — Superseded
3 Documentation — Draft
4 Documentation — Ready
5 Implementation — Not started
6 Implementation — In progress
7 Implementation — Done

Example: data-adr-weight="4" → Current status is Documentation · Ready; earlier steps in the log are green, later steps gray. Implementation: Done (7) uses green when it is the current value. Out-of-range values are clamped in docs/assets/docs-nav.js.

Ratification

Adopted before ADR 0018; no separate ratification trail was required then. Git history for this file records merges into main.

Context

The service was primarily run from a virtual environment (make run) with optional Docker-only observability stacks. Docker and Kubernetes remain optional for day-to-day feature work; they do not replace the venv workflow. Real deployments typically consume a built image from a registry; contributors learning production-style workflows need a reproducible container image and a minimal Kubernetes deployment to practice probes, declarative configuration, and secrets without relying on a cloud account.

Decision

  • Provide a Dockerfile that installs dependencies from the same requirements.txt as make install, copies application and Alembic assets, runs as non-root user uid 1000, and sets ENTRYPOINT to scripts/container_entrypoint.sh (same shell script as make container-start on the host: alembic upgrade head then Uvicorn). The image does not run make (no venv layout in-container).
  • Default container paths: SQLite file under /data (configurable via SQLITE_DB_PATH), logs under /tmp/logs when not overridden, to avoid permission issues on read-only layers.
  • Kubernetes non-secret configuration: single source file k8s/app.env; k8s/configmap.yaml is generated by scripts/render_k8s_configmap.py (also invoked from make docs-fix). Do not put API keys in the ConfigMap; default local cluster profile is APP_ENV=dev with the built-in dev default key when unset. For qa, use a Secret and optional secretRef (see k8s/secret.example.yaml).
  • Ship example Kubernetes manifests under k8s/: Namespace, generated ConfigMap, Secret template, PVC, Deployment, Service. These target local clusters; they are not a full production chart.
  • Keep SQLite + single replica for this ADR’s scope; document that scaling out requires a shared database.
  • Add Makefile helpers docker-build, k8s-render-configmap, k8s-apply, and developer guide docs/developer/0009-docker-and-kubernetes-local.html.

Scope

  • In scope: image layout, entrypoint behavior, local kubectl apply flow, documentation.
  • Out of scope: hosted registry promotion, Helm, Ingress/TLS, PodSecurity admission policies, horizontal pod autoscaling, replacing SQLite with PostgreSQL.

Alternatives considered

  1. Helm chart only
    • Pros: templating for many environments.
    • Cons: steeper learning curve for beginners; rejected as the primary path until plain manifests exist.
  2. docker-compose for the API
    • Pros: simple multi-container dev.
    • Cons: does not teach Kubernetes primitives; observability compose already exists separately.
  3. Init container for migrations, app container without Alembic
    • Pros: clearer separation.
    • Cons: duplicate image configuration for standalone docker run; entrypoint migration keeps one path for Docker and K8s for now.

Consequences

Positive

  • Single reproducible artifact (image) aligned with CI and release practices.
  • Readiness/liveness probes exercise the same endpoints used in production-style health checks.
  • Secrets and config are separated (Secret vs ConfigMap).

Trade-offs

  • SQLite on a PVC is not suitable for multi-replica HA deployments.
  • Image build must be repeated or loaded into minikube/kind explicitly when the Docker daemon differs from the cluster’s.

Compatibility and migration

  • Backward-compatibility impact: none for Python APIs; additive files only.
  • Migration plan: N/A.
  • Rollback strategy: stop using k8s/ manifests and continue with make run.

Implementation plan

  1. Add Dockerfile, scripts/container_entrypoint.sh, .dockerignore.
  2. Add k8s/*.yaml and Makefile targets.
  3. Document in developer guide and this ADR; link from ADR index.

Validation

  • Technical: docker build -t study-app-api:local . succeeds; optional local kubectl apply + port-forward returns HTTP 200 on /live.
  • Documentation: developer guide and ADR cross-linked.
