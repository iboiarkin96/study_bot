ETR Study API — Engineering Practices

This document contains engineering workflow, delivery rules, policy governance, and operational runbook discipline.

1. Development Guide

1.1 Project Philosophy

  • API contract is the product: request/response/error rules must always match runtime behavior.
  • Documentation evolves with code in the same change set.
  • Centralized source-of-truth files prevent drift as API scales.

1.2 Validation Standards (General)

  • Validate external input at API boundary (schemas/types/validators).
  • Do not duplicate request-shape validation in service/repository layers.
  • Normalize validation failures into stable error codes and keys.
  • Return consistent machine-readable payloads across endpoints.

1.3 Error Governance and Expansion

  • Error code + key are immutable public contracts.
  • Add new codes, do not repurpose existing ones.
  • Keep a fallback code path for unmapped validation failures.
  • Every endpoint should declare explicit error responses in Swagger.

1.4 Documentation Workflow at Scale

  • Maintain source of truth in: app/validation/user.py, app/schemas/errors.py, app/openapi/examples/errors.py, router responses.
  • After API changes run make docs-fix; verify marker blocks only are auto-updated.
  • Keep documentation concise: explain standards and extension path, not full per-code dumps.

1.5 Testing Policy (Mandatory)

  • API changes without tests are considered incomplete and should not be released.
  • For each endpoint, keep at least one success and one failure-path test.
  • Contract testing is mandatory: run OpenAPI snapshot contract test to detect accidental API drift.
  • Contract test must guard response fields, error codes, and schema types via baseline diff.
  • Keep tests deterministic and isolated from mutable local state.
  • Use make test for full run and make test-one path=... for targeted checks.

1.6 Release-Gate Procedure (Mandatory)

  • Before deployment, run make release-check as a quality gate.
  • Pipeline includes: make env-check, make verify, make docs-fix.
  • Deployment must be blocked if any step fails.
  • Use make release DEPLOY_CMD="..." to enforce release gate before real deployment command.

1.7 Code Formatting and Linting

  • Use make format-fix to keep code style consistent before review.
  • Use make lint-check and make type-check as mandatory static checks.
  • Use make verify for combined lint + typecheck + OpenAPI contract + tests + docs sync.
  • Install hooks once with make pre-commit-install and run with make pre-commit-check.
  • Continuous integration: on push and pull requests, GitHub Actions runs make verify (lint, types, OpenAPI, contract tests, tests, then docs-fix) and then pre-commit run --all-files. You must still run make verify-ci locally before push: CI does not assert that your commits match docs-fix output, but docs-check (in verify-ci) does—otherwise GitHub Pages and other clones can show stale generated docs while the build stays green. Standard hosted runners are free for public repositories; private repos receive a monthly Actions allowance on free plans.

1.8 API Security by Default (Mandatory)

  • Every protected route under /api/v1 must pass auth, rate-limit, body-limit, and security-header checks.
  • Auth strategy is configured via API_AUTH_STRATEGY; default mode is mock_api_key.
  • Any endpoint contract must explicitly document 401, 413, and 429 responses when applicable.
  • Do not bypass CORS allowlist or increase limits without decision record in ADR.
  • Security-related config changes require docs update in the same change set.

1.9 Idempotency for Write Operations (Mandatory)

  • Critical write endpoints must support idempotent retries with Idempotency-Key.
  • Reuse of key with same payload must replay original success response.
  • Reuse of key with different payload must fail with explicit conflict response.
  • Idempotency rules and examples must be visible in OpenAPI docs and tests.
  • Policy-level changes are captured in ADR before rollout.

1.10 Observability and Readiness (Mandatory)

  • Service must expose /live and /ready probes.
  • /live reflects process liveness only; dependency checks belong to /ready.
  • /metrics must expose Prometheus metrics for API latency, error rate, and DB latency.
  • Use low-cardinality metric labels only (method, route template, status class/code, operation type).
  • Run make contract-test and make verify before release to protect API contract.

1.11 Environment Profiles and Config Governance (Mandatory)

  • Use explicit app profiles: APP_ENV=dev|qa|prod. Automated tests run under qa rules.
  • Single committed template: env/example → copy to .env once (make env-init).
  • Base config is loaded from .env; profile overrides are loaded from env/<APP_ENV> (e.g. env/dev).
  • Optional ENV_FILE applies last for deployment-specific secrets/config.
  • GET /live must expose effective app_env for operators to verify the running profile.
  • In qa/prod, insecure defaults are blocked (disabled auth, default API key, metrics disabled).
  • In prod, localhost CORS and LOG_LEVEL=DEBUG are blocked by startup validation.

2. Docs as Code Workflow

Documentation is treated as production artifact and evolves together with application code.

2.1 Source of Truth

  • Service behavior and contracts: app/
  • Error mapping: app/validation/user.py
  • OpenAPI examples: app/openapi/examples/
  • UML sources: docs/uml/**/*.puml
  • Generated sections: marker blocks in README.md and docs/system-analysis.html

3. API Versioning Policy (Required)

3.1 Version Model

  • Major version is part of URL prefix (current: /api/v1).
  • Current major is backward-compatible by default.
  • Breaking changes are released only in a new major version (for example /api/v2).

3.2 Breaking Changes (Not Allowed in Same Major)

  • Removing or renaming endpoints, fields, or enum values.
  • Changing field type or semantic meaning.
  • Changing requiredness of existing request fields (optional -> required).
  • Repurposing existing error code/key semantics.

3.3 Allowed Non-Breaking Changes

  • Adding new endpoints.
  • Adding optional fields in request/response payloads.
  • Adding new error codes without changing existing ones.
