Schemas and Contracts Guide

This file describes where API schemas/contracts live and how to evolve them safely.

1. Where schemas live

  • Request/response schemas: app/schemas/
  • Error schemas: app/schemas/errors.py
  • OpenAPI examples: app/openapi/examples/
  • Router contract (responses, summary, description): app/api/v1/

2. Contract rules

  • API contract is public interface.
  • Existing fields and semantics are backward-compatible within same major.
  • Error code and key semantics are immutable.
  • Changes are additive by default.
  • Critical write endpoints define idempotency contract via Idempotency-Key header.

3. Idempotency contract for writes

  • Same key + same payload -> replay original success response.
  • Same key + different payload -> explicit conflict response.
  • Missing key (for required write endpoint) -> explicit business validation error.

4. Error contract

Business error shape:

  • {"code":"...","key":"...","message":"...","source":"business"}

Validation error shape:

  • {"error_type":"validation_error","endpoint":"...","errors":[...]}

Validation rules source:

  • app/validation/
  • Status-first governance matrix: Error Matrix By Status

5. How to evolve schemas

  1. Update schema classes in app/schemas/.
  2. Update endpoint declaration in app/api/v1/....
  3. For write routes, update header contract and idempotency behavior docs.
  4. Update examples in app/openapi/examples/....
  5. Update validation mapping in app/validation/... if needed.
  6. Add/update tests.
  7. Run make verify and make docs-fix.

6. Anti-patterns

  • Inline hardcoded examples in routers/schemas duplicated in many files.
  • Reusing existing error code for new meaning.
  • Silent breaking change without versioning/migration notes.
  • Accepting retried write requests without idempotency contract.
