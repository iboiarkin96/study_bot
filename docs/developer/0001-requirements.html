Developer Requirements Guide

Overview

How product requirements map into code and tests.

Functional requirements

Product-level requirements live under System design → Functional Requirements.

When you implement an endpoint:

  • Tie it to at least one named functional requirement.
  • Keep behavior predictable and covered by tests.
  • Keep the HTTP contract and the database behavior in sync.

Non-functional requirements (NFR)

Non-functional requirements live in docs/internal/analysis/system-design.html#dev-nfr.

Engineering checklist:

  • Performance: avoid N+1 queries; do not serialize fields callers do not consume.
  • Reliability: wrap multi-step writes in a transaction and roll back on any failure.
  • Maintainability: keep the layering strict (api → service → repository).
  • Observability: request and business logs are structured and carry enough context to debug a failure.
  • Environments: safe defaults differ by profile (dev, qa, prod).
  • Contracts: change APIs and errors in additive ways whenever the call site allows it.
  • Idempotent writes: retries deduplicate safely under explicit rules.

Requirement-to-implementation mapping

Before merging a new endpoint, confirm:

  1. The requirement is named in the docstring or in the docs.
  2. Request and response schemas are explicit.
  3. Business rules sit in the service layer.
  4. OpenAPI responses document every error.
  5. Tests cover both success and failure.
  6. Write routes document idempotency: Idempotency-Key, replay, and conflicts.

Done criteria

The endpoint is done when:

  • Behavior matches the requirement.
  • The NFR impact is reviewed.
  • Tests pass.
  • Docs are up to date (make docs-fix).
