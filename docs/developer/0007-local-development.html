Local development (environment and run targets)

How to run the API locally, optionally with the Docker observability stack (Prometheus, Grafana, Blackbox exporter) and, separately, an optional logging stack (Elasticsearch, Kibana, Filebeat) for searchable NDJSON logs. For metrics URLs, logging quick start, and probe checks, the repository root README.md is the short reference; details below complement ADR 0023.

1. First-time setup

  1. make venv then make install (activate .venv if you prefer).
  2. Create .env: make env-init (or copy env/example), then set APP_ENV and secrets as needed.
  3. make migrate to apply Alembic migrations.
  4. Optional: make env-check to verify config and database path.

2. Run targets

Command What it does
make run Starts the FastAPI app with Uvicorn (reload). Reads .env. Use this when you only need the API.
make run-project Runs make observability-up first (renders Prometheus config, starts docker-compose.observability.yml: Blackbox, Prometheus, Grafana), then the same Uvicorn command as make run. Requires Docker. The API process stays in the foreground; stop it with Ctrl+C.
make observability-up / make observability-down Start or stop the observability stack alone. Use when you already run make run in one terminal and want metrics dashboards without restarting the API.

3. Default local ports

Override via .env where documented in env/example.

  • API: 127.0.0.1:8000 (from APP_HOST / APP_PORT)
  • Prometheus: 127.0.0.1:9090
  • Grafana: 127.0.0.1:3001 (container maps host 3001 → Grafana 3000)
  • Blackbox exporter: 127.0.0.1:9115

4. After you stop the API

Containers from make run-project or make observability-up keep running until you run make observability-down (or docker compose -f docker-compose.observability.yml down).

5. Optional logging stack (Elasticsearch, Kibana, Filebeat)

For searchable, structured logs locally, use NDJSON (LOG_FORMAT=json or leave LOG_FORMAT unset—the application default is json; see env/example and app/core/config.py) so the API writes lines under LOG_DIR (default logs/). Then start the Elastic stack (requires Docker; allocate roughly 2 GiB RAM for Elasticsearch and Kibana): 

make logging-up

Filebeat tails the mounted ./logs directory and ships lines to Elasticsearch. Open Kibana at http://127.0.0.1:5601 (default), create a data view whose index pattern is *study-app-logs* (asterisks before and after — Elasticsearch 8 may store documents in backing indices named .ds-study-app-logs-…, which do not match the narrower pattern study-app-logs-*, so searches like request_id: "…" return nothing). Then use Discover to search (filter by field request_id or paste the UUID into the Kibana search bar). CLI check without Kibana: make logging-es-query (last few docs) or make logging-es-query QUERY=<uuid> (same as python scripts/check_es_request_id.py <uuid>). Smoke checks: make logging-smoke. Stop with make logging-down. Licensing and scope: ADR 0023.

Data path (why Discover can be empty)

  1. API on the host writes lines to ./logs/*.log (e.g. app.log). Nothing is sent to Elastic until step 3.
  2. Filebeat (container) reads the same files via the bind mount ./logs → /var/log/study-app and publishes events to Elasticsearch. If the API is not running or writes elsewhere, Filebeat has nothing new to ship.
  3. Elasticsearch stores documents in indices / data streams whose names contain study-app-logs (e.g. .ds-study-app-logs-…).
  4. Kibana Discover reads from Elasticsearch only through a data view whose pattern matches those indices — use *study-app-logs*. Set the time range to include “now” (e.g. Last 15 minutes) after you generate traffic.

Clean slate (reset logs + Elastic)

To remove old noise and old indices: make logging-reset stops the stack, deletes the Docker volume holding Elasticsearch data (all indices and Kibana objects such as saved data views), and deletes logs/*.log. Then run make logging-up, open Kibana → Stack Management → Data views → Create data view, index pattern *study-app-logs*, timestamp field @timestamp. Start the API (make run), hit /ready a few times with curl, wait ~10–20 s, refresh Discover.

6. Sanity checks

  • API: curl -s http://127.0.0.1:8000/live, /ready, /metrics (responses include X-Request-Id)
  • Observability endpoints: make observability-smoke
  • Logging stack (when running): make logging-smoke
