Docker image and local Kubernetes

This guide explains how the API is packaged as a container, how that maps to Kubernetes objects, and how to run the stack on a local cluster for learning and integration tests. Production hardening (registry scanning, multi-replica SQLite constraints, ingress TLS) is out of scope here; see ADR 0015.

Optional tooling and daily development

Docker and Kubernetes in this repository are optional. Normal development uses a virtual environment, make run, and tests — the same workflow as in Local development. You do not need to build an image or apply manifests for every feature.

The development cycle (implement → test → make verify → commit) does not fundamentally change when Docker/K8s exist in the repo: they are an additional path for packaging and for running the stack like a remote environment.

Real deployments usually expect a container image: something must build the image, push it to a registry (Docker Hub, GitHub Container Registry, cloud vendor registry, etc.), and the target environment (Kubernetes, a VM with Docker, or a managed service) pulls that image and runs it with production configuration and secrets. The repo’s Dockerfile is the starting point for that image. Automated build and push to GitHub Container Registry after CI is described in ADR 0021; this guide focuses on the local build and kubectl apply flow. See What a real deployment looks like below.

What a real deployment looks like (high level)

A full production setup is unique to each team, but the pattern is almost always:

  1. Build and test in CI — pipeline runs lint, tests, and (often) builds the Docker image and tags it (for example :1.4.2 or :<git-sha>).
  2. Push to a registry — docker push to a URL your servers can read (authentication required). Nothing is “sent” automatically from git push unless you wire a workflow.
  3. Roll out to an environment — Kubernetes updates a Deployment to the new image tag, or a VM runs docker pull && docker run, or a PaaS (Cloud Run, ECS, etc.) deploys the new revision.
  4. Configuration and data — production APP_ENV, secrets, database URL, and scaling are set in that environment (not in the image). SQLite is fine for demos; real multi-instance setups typically use a shared database (for example PostgreSQL).
  5. Traffic — DNS points to a load balancer or ingress; TLS terminates at the edge; requests reach your service on port 8000 (or behind a reverse proxy).

Local k8s/ manifests are a learning and smoke-test baseline; production clusters add namespaces, quotas, network policies, monitoring, and often Helm or GitOps — all outside this guide.

0. What you need installed

Before building the image or applying manifests, install the following on your machine.

  • Docker Desktop or Docker Engine — installation: Get Docker.
  • kubectl — installation: Kubernetes tools.
  • A local Kubernetes cluster — pick one:
    • Docker Desktop: Settings → Kubernetes → Enable Kubernetes (built-in cluster). See Docker Desktop Kubernetes.
    • minikube — Install & start.
    • kind — Quick start.

Point kubectl at your cluster (kubectl config current-context should show the context you expect) before running the steps below.

1. Docker in one minute

Docker builds an image (a filesystem snapshot + metadata) and runs it as a container (an isolated process on your machine). The API image installs dependencies from the same requirements.txt as make install, copies app/ and alembic/, and starts the app via scripts/container_entrypoint.sh (alembic upgrade head, then Uvicorn). On the host, make container-start runs the same script (with your .venv); the image does not invoke make — there is no .venv inside the container.

  • Image name used in this repo: study-app-api:local (tag is arbitrary).
  • Build: make docker-build or docker build -t study-app-api:local .
  • Run (example): mount a volume for SQLite so data survives container removal: 
    docker run --rm -p 8000:8000 \
            -e APP_ENV=dev \
            -e SQLITE_DB_PATH=/data/study_app.db \
            -e API_MOCK_API_KEY=local-dev-key \
            -v study-app-data:/data \
            study-app-api:local
    Then open http://127.0.0.1:8000/live. For qa/prod rules, non-default API keys and stricter CORS apply (app/core/config.py).

2. Configuration: one place for Kubernetes

Non-secret environment variables for the in-cluster pod are maintained in k8s/app.env (dotenv format). The file k8s/configmap.yaml is generated — do not edit it by hand. Regenerate with make k8s-render-configmap or run make docs-fix / make verify (the docs pipeline runs the same renderer).

The full variable catalog and semantics for local development remain in env/example (copy to .env for make run). The Kubernetes file is a subset plus overrides for containers (APP_HOST=0.0.0.0, SQLITE_DB_PATH=/data/study_app.db, LOG_DIR=/tmp/logs).

Default cluster profile is APP_ENV=dev. The API key is not stored in the ConfigMap: the process uses the built-in dev default from app/core/config.py when API_MOCK_API_KEY is unset. For APP_ENV=qa, you must supply a non-default key via a Kubernetes Secret and add a secretRef on the Deployment — see k8s/secret.example.yaml.

3. Kubernetes mental model

Kubernetes runs containers for you across machines. On a laptop you still use the same ideas: declarative YAML describes desired state; the control plane schedules work.

Object Role in this project
Namespace study-app — groups all objects for this app (k8s/namespace.yaml).
ConfigMap Non-secret env — generated from k8s/app.env into k8s/configmap.yaml.
Secret Optional: only for qa/prod-style keys; see k8s/secret.example.yaml.
PersistentVolumeClaim Disk for /data/study_app.db — SQLite file survives pod restarts (single replica).
Deployment Runs the API pod(s), wires env, mounts the volume, sets CPU/memory limits, configures /live and /ready probes.
Service Stable cluster IP and port 8000 to reach the pod from inside the cluster.

Traffic from your browser does not hit the Service directly until you add an Ingress or use kubectl port-forward (below).

4. Build the image and make it visible to the cluster

  • Docker Desktop Kubernetes: build on the host — make docker-build — the cluster usually sees the same image cache.
  • minikube: eval "$(minikube docker-env)" then make docker-build so the daemon that minikube uses gets the image.
  • kind: make docker-build then kind load docker-image study-app-api:local --name <cluster>.

5. Apply manifests

make k8s-apply renders the ConfigMap from k8s/app.env, then applies namespace, ConfigMap, PVC, Deployment, and Service. For the default APP_ENV=dev flow you do not need to create a Secret first. 

make k8s-apply

Equivalent manual sequence:

make k8s-render-configmap
                  kubectl apply -f k8s/namespace.yaml
                  kubectl apply -f k8s/configmap.yaml
                  kubectl apply -f k8s/pvc.yaml
                  kubectl apply -f k8s/deployment.yaml
                  kubectl apply -f k8s/service.yaml

6. Reach the API from your machine

kubectl -n study-app port-forward svc/study-app-api 8000:8000

Then: curl -s http://127.0.0.1:8000/live, /ready. For protected routes, send X-API-Key with the dev default (local-dev-key) unless you override via Secret.

7. SQLite and replicas

This API uses SQLite. The PVC gives one pod durable storage. Do not scale the Deployment beyond one replica with this setup; for horizontal scaling you would move to a shared database (for example PostgreSQL) — a future architectural change.
