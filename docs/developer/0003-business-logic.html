Business Logic Guide

Overview

How HTTP, services, repositories, and models split responsibility.

Layering model

  • API layer: app/api/v1/ (transport only).
  • Service layer: app/services/ (business rules).
  • Repository layer: app/repositories/ (database access).
  • Models: app/models/ (ORM entities).

Rules:

  • The router does not implement business rules.
  • The service does not depend on FastAPI request primitives.
  • The repository does not return HTTP-shaped structures.

Theoretical background

Layered architecture — theoretical background

Layered architecture stacks the system in horizontal layers, and each layer talks only to the layer below. Inner layers (rules, persistence) never import HTTP or transport details, so they stay testable and the adapters stay swappable.

The shape is close to ports and adapters: the core defines behavior, while HTTP and SQL are adapters that plug into stable ports.

Contract-first means the OpenAPI spec describes the HTTP surface and the implementation must match it. The outer adapter stays bound to a stable contract.

Layer responsibilities (reference)

The public API is described first in OpenAPI (contract-first): what callers see in the spec is what they rely on. The system-wide diagrams (C4) live in System design; a simplified request-path diagram is rendered from docs/uml/architecture/internal_readme_layer_boundaries.puml.

Layer Packages Responsibility Contract and constraints
HTTP API app/api/ FastAPI routers declare paths, dependencies, and Pydantic-typed request and response bodies from app/schemas/; they call services. Handlers stay thin and never duplicate field validation. Matches OpenAPI (operations, declared responses, examples). Maps successful outcomes and framework-level HTTP concerns to the right status codes.
Schemas & 422 contract app/schemas/, app/validation/, handlers in app/main.py app/schemas/ holds Pydantic models and field constraints. app/validation/ translates Pydantic and FastAPI validation failures into the canonical 422 payload for clients. Exception handlers are registered next to the app setup in app/main.py. Error catalog and examples live in OpenAPI. Pydantic error kinds align with the documented code and key values.
Application app/services/ Use cases and business invariants (for example, a duplicate composite key) live here. Services raise HTTPException with business error payloads when a rule fails. Business errors match the governed contract. Layering policy: Developers Docs.
Persistence app/repositories/, DB session in app/core/ Loads and persists entities through SQLAlchemy. Schema changes go through Alembic migrations. Storage details stay out of the public HTTP model. DB constraints stay consistent with the API promises.

Where validation runs

The order of checks:

  1. Field rules for JSON bodies live in Pydantic models under app/schemas/: required fields, types, length limits, and so on.
  2. FastAPI runs those checks before any code in app/api/ executes. Handlers always receive data that already passed this step.
  3. When a check fails, app/validation/ maps the failure into the canonical 422 response (code, key, message). It does not redefine the rules, only the error shape that clients see. The wiring is registered in app/main.py.
  4. Rules that need the database (for example, “this user already exists”) run in app/services/ and surface as business client errors, not as schema-level validation. See ADR 0003.

Responsibility boundaries

Router:

  • Parses and validates the request.
  • Calls the service.
  • Shapes the response model.
  • Declares the OpenAPI responses.

Service:

  • Enforces domain rules.
  • Orchestrates repositories.
  • Raises domain and business errors.

Repository:

  • Isolates query and write mechanics.
  • Keeps persistence details local.

Transactions and consistency

  • Write operations are atomic.
  • Duplicate and conflict checks are deterministic.
  • Every failure path returns the stable error contract.

Logging and observability

  • Request-level logging: middleware in app/main.py.
  • Business event logs: the service layer.
  • Log format and configuration: app/core/logging.py.

Typical change flow

  1. Define or adjust the schema.
  2. Implement the service rule.
  3. Implement the repository operation.
  4. Wire the router.
  5. Add examples and error mappings.
  6. Add tests.
  7. Run the make pipeline (make docs-fix, make verify).
