RFC 0002: Docs Search KPI Policy and SLO Thresholds

Status

  • State: active
  • Date: 2026-04-17
  • Related decision: ADR 0027
  • Related implementation: RFC 0001

Goals and non-goals

This policy defines service-level targets for docs search quality and the operating playbook when quality degrades.

Goals

  • Define measurable, stable KPIs for docs search quality on docs/**/*.html.
  • Set explicit warning and breach thresholds for operational response.
  • Keep KPI review cadence and response actions consistent across releases.

Non-goals

  • Semantic/vector retrieval, typo correction, and multilingual ranking.
  • Replacing RFC 0001 implementation details with policy-level text.

Canonical KPI definitions

  • Zero-result rate = count(search_query where results_count = 0) / count(search_query).
  • Query CTR = count(distinct query_id with at least one search_result_click) / count(search_query).
  • TTFS (time-to-first-success) = distribution of first search_success.time_to_success_ms per session (p50/p75).

Source endpoint: GET /internal/telemetry/docs-search/metrics?window_minutes=<N>.

Targets (rolling 7-day window)

KPI Target Warning threshold Breach threshold
Zero-result rate ≤ 12% > 12% and ≤ 18% > 18%
Query CTR ≥ 45% < 45% and ≥ 35% < 35%
TTFS p50 ≤ 25s > 25s and ≤ 40s > 40s
TTFS p75 ≤ 60s > 60s and ≤ 90s > 90s

Operating cadence

  • Weekly: review KPI snapshot for 7-day and 30-day windows.
  • Release gate: check that KPIs are not in breach before major documentation rollouts.
  • Monthly: record trend note in docs changelog/audit updates when significant shifts appear.

Response playbook

Warning level (one KPI in warning for 2+ days)

  1. Inspect top zero-result queries and top low-CTR queries.
  2. Verify index freshness and payload health: rebuild with python3 scripts/build_docs_search_index.py.
  3. Review ranking regressions after recent content/nav changes.

Breach level (any KPI in breach for 24h)

  1. Open incident task in backlog and assign owner for 48h mitigation.
  2. Run telemetry validation playbook from RFC 0001.
  3. Apply hotfix: index regeneration, ranking rollback, or nav/content correction.
  4. If user-facing quality is materially degraded, publish runbook note and add changelog entry.

Runbook links

Change management

  • Adjust KPI thresholds only with evidence from 30+ days of telemetry and documented rationale.
  • Any threshold change must update this RFC and be referenced in docs/CHANGELOG.md.
  • ADR updates are required only when the architecture itself changes (not just threshold tuning).
