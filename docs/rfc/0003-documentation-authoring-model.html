Goals and non-goals

This RFC pulls together — into a single normative reference — the authoring model the project uses for every kind of documentation page it ships. Earlier ADRs (0001, 0025, 0026, 0007) decided that we use Docs-as-Code, where internal vs external content lives, and which artifact is authoritative for which audience. They did not, in one place, describe the templates a contributor must copy and the rationale behind their shape. This RFC fills that gap.

Goals

  • Name every document type produced in this repository and its canonical template.
  • Specify the required section list for the two heavy templates — the API operation spec and the screen spec — plus the lighter templates (ADR, RFC, runbook, how-to, glossary).
  • Justify each template against published, internationally recognised documentation programs (Stripe, Atlassian Design System, GitHub Primer, Vercel, Linear, IBM Carbon, Material Design, Adobe Spectrum, Salesforce Lightning, Storybook, Mailchimp, GOV.UK Service Manual, Kubernetes, Wikipedia / MDN page-history).
  • Enumerate the lint, audit, and Definition-of-Done gates that turn the templates into a contract.
  • Provide copy-pasteable section catalogues (Appendix A, Appendix B) so a contributor never has to reverse-engineer the template by reading a sibling page.

Non-goals

  • Visual styling. Typography, colour, spacing, motion, and the design tokens are governed by the docs frontend stylesheet (docs/assets/docs.css) and by docs-frontend-token-gallery.html — not by this RFC.
  • OpenAPI authoring rules. Already governed by ADR 0007 and the shared OpenAPI authoring guide. This RFC links them; it does not restate them.
  • Public developer-portal product. Out of scope per ADR 0025.
  • Marketing or end-user help-centre copy. The corpus described here is for engineers and integrators only.

Background

Until this RFC, the project's documentation rules were distributed across seven ADRs and three process pages. A contributor who landed in the repository needed to read ADR 0001 for the governance principle, ADR 0025 for the audience split, ADR 0026 for the internal layout, ADR 0007 for the external-contract rules, the documentation maintenance process for the lifecycle, the documentation style guide for the page skeleton, and finally the two heavy templates (API spec template and screen spec template) to actually start writing. The intent of each artifact was sound; the discoverability was poor and the rationale for the chosen shape was implicit.

The motivation to consolidate is simple: as the surface grows, the cost of an inconsistent page is compounded — readers stop trusting the docs, drift creeps in unnoticed, and the corpus loses its claim of being a source of truth. International programs that have lived through this growth phase (Stripe, Atlassian, Vercel) all converged on fixed-shape templates plus a public rationale. This RFC adopts the same posture for our repository, with this single page as the entry point a new contributor reads first.

Design

1. Document taxonomy

Every page authored under docs/ belongs to exactly one of the following document types. A page that does not match any type is either misfiled or a missing type — open a discussion before inventing a new shape.

Type Audience Authoritative for Template / location
ADR — Architecture Decision Record Engineers (in this repo) One architectural decision, its context, the alternatives considered, and the consequences docs/adr/0000-template.html
RFC — Request for Comments Engineers The specification an ADR intentionally omits: implementation, validation, rollout docs/rfc/0000-template.html
API operation spec Engineers (analysts, developers, reviewers) Behavioural contract of one HTTP endpoint: business rules, request/response, errors, observability docs/internal/api/_shared/spec-template.html
API resource hub Engineers One resource: business context, invariants, endpoint index, anchor targets for per-method pages Pattern documented in ADR 0026 §Repository layout and howto/0002
Screen spec (frontend) Frontend / design / docs maintainers One user-facing screen: anatomy, states, data contract, where to change what docs/internal/front/screens/docs-screen-template.html
Frontend reference Frontend maintainers One cross-cutting frontend system (navigation, motion, tooltips, …) Convention from documentation-process.html § Document types
Runbook On-call / operators Operational procedure for one incident or recurring task docs/runbooks/
How-to Contributors Step-by-step procedure for one repeatable engineering task docs/howto/
Glossary All readers Single canonical definition of every domain term used elsewhere frontend glossary (extend with sections per area as needed)
External API contract Integrators (partners, third-party clients) HTTP request/response surface at a given version OpenAPI under docs/openapi/ — governed by ADR 0007
Python reference Contributors Python module / class / function reference Generated under docs/pdoc/ per ADR 0016

The taxonomy is closed. A contribution that does not fit one of these types is by definition a request to extend the taxonomy — open a discussion issue first (ADR 0018).

2. The API operation spec template

The heavy template at _shared/spec-template.html is the contract that the systems analyst owes the developer before implementation begins. Its 21 required sections are catalogued in Appendix A; the live page is the source of truth for the markup. The lint at scripts/spec_lint.py enforces presence, attribute markers, and content density.

The template's shape is not arbitrary. It is the intersection of three established practices:

  1. Behaviour-first specification — the section order forces the analyst to write the why (business context, user stories, acceptance criteria) before the what (request/response, errors, side effects). This is the same authoring order recommended in GOV.UK Service Manual and the Stripe API reference: a reader who scans only the first two cards must understand whether the operation applies to their problem, before paying the cost of reading the schema.
  2. Contract-first technical spec — request and response sections are field-level tables (name · location · type · format · required · nullable · default · constraints · business meaning · PII class · OpenAPI mapping), not free prose. This shape is borrowed from OpenAPI 3.1 parameter objects and from the Atlassian REST API documentation tabular convention. It survives translation to OpenAPI without information loss.
  3. Operational realism — every spec must declare side effects, idempotency and concurrency, error catalog with envelope JSON, edge cases, and observability. Without these sections, an endpoint passes review but fails in production. The requirement is shaped after the post-mortem culture at companies that publish their incident reviews publicly (Cloudflare, Stripe, AWS).

3. The screen spec template

The frontend twin of the API spec lives at screens/docs-screen-template.html. Its required sections are catalogued in Appendix B. The template is informed by:

  • Atlassian Design System, GitHub Primer, and Material Design: anatomy diagrams as the canonical visual reference — annotated SVG embedded inline, with numbered callouts that the rest of the spec references by number, not by colour or position.
  • Stripe Elements, Adobe Spectrum, and IBM Carbon: component anatomy with named regions (header, body, footer, icon, content slots). Every spec has a Layout map that lists each region with its CSS hook and purpose.
  • Web Components and Salesforce Lightning: data contract — every data-* attribute consumed by the runtime is listed in the spec, so the markup-to-code coupling is reviewable in a single PR.
  • Storybook stories: states gallery (default, hover, focus, error, loading, empty) rendered as a per-state row with a screenshot or anatomy panel.
  • Linear's engineering posts and Notion's eng playbook: "where to change what" — every spec ends with a file-by-file mapping from behaviour to source, eliminating the most common new-contributor question on day one.

4. Visual evidence — screenshots and anatomy diagrams

The screen-spec template carries both a real-screenshot gallery and a hand-authored anatomy diagram. Each answers a different question:

  • Screenshots — “what does this look like right now?” — captured by scripts/capture_screen_specs.py for reproducibility, refreshed in the same PR that changes the screen.
  • Anatomy diagrams — “what are the named regions?” — inline SVG, hand-edited in git, numbered callouts, accessible (<title> + <desc> + caption).

The reasoning for keeping both is the lesson learned by every long-running design system program: screenshots rot on every style tweak and are useless in code review diffs, while diagrams age gracefully and survive cosmetic changes — but only screenshots prove current visual reality. The pair is cheap to maintain because each artifact has a single authoring path and an automatic refresh hook.

5. Diagrams: PlantUML only

Per ADR 0020, every diagram in the repository is PlantUML — sources under docs/uml/**/*.puml, rendered SVG under docs/uml/rendered/*.svg. Mermaid, Graphviz, Lucid, draw.io, and image-only diagrams are not admitted. The reason is auditability: PlantUML is text in git, diff-readable in code review, and rendered by a deterministic build step. Templates reference rendered SVGs; they do not embed alternative formats.

6. Plain-language editorial standard

Every page in this repository is written for engineers whose first language is rarely English. The editorial rules in documentation-style-guide.html § Plain language are normative: short sentences, common vocabulary, one idea per sentence, defined acronyms on first use, active voice for procedures. The standard is modelled on GOV.UK Content design, Stripe writing principles, and Mailchimp Voice & Tone.

7. Page history is required

Every non-redirect page ends with a Page history table. This is borrowed directly from Wikipedia's "View history", MDN's "Page history", and Notion's "Page updates" — each of which discovered independently that an attached, human-readable change log is what turns a page from "current" into "audited". git log remains the full audit trail; the table is a scannable surface for a future reader who needs to know when the page was last verified against reality.

8. Single source of truth, co-located with code

Every topic has exactly one canonical page. Other pages link, never restate. When two pages contradict, the canonical page is fixed and the other becomes a link. This is the only rule that survives across the documentation programs we modelled this on (Stripe, Atlassian, Vercel, Storybook, Linear); it is also the rule most often violated under deadline pressure, which is why the lint and review checklist enforce it explicitly.

Why this template — international references

The two heavy templates (API operation spec, screen spec) and the lighter ones (ADR, RFC, runbook, how-to) are not novel. Each section in each template traces back to a published convention from a documentation program known for its quality. The table below is the explicit map; it is also the answer to the question "why this template and not another?" — every choice has a precedent, and the cost of deviating from it is well understood.

Implementation

The templates and the rules around them already exist in the repository. This RFC's implementation work is therefore discoverability and enforcement, not new authoring.

  1. Land this RFC at docs/rfc/0003-documentation-authoring-model.html. The page is the canonical entry point a contributor reads before touching docs. Add it to the RFC index (rfc/README.html) and to the internal sidebar (docs/assets/internal-sidebar.js).
  2. Cross-link from the entry-points. ADR 0001, ADR 0025, ADR 0026, style guide, and documentation-process all link to this RFC as their single rationale-and-template index.
  3. Templates as concrete artifacts — keep them where they already are; this RFC links to them, never duplicates them:
    • docs/internal/api/_shared/spec-template.html — API operation spec.
    • docs/internal/api/_shared/spec-definition-of-done.html — DoD checklist.
    • docs/internal/front/screens/docs-screen-template.html — screen spec.
    • docs/adr/0000-template.html — ADR.
    • docs/rfc/0000-template.html — RFC.
  4. Lint hookup. scripts/spec_lint.py already validates the API spec template; add a one-line check that data-spec-template="v1" is present in metadata so future template versions are machine-detectable. make docs-check already runs as part of make verify per ADR 0001.
  5. PR template entry. The PR template asks "Did you copy the right template? Did you fill every required section? Did you add a Page history row?" — a single bullet pointing here.

Validation

Local checks

  • make docs-fix — normalises HTML; idempotent.
  • make docs-check — fails when committed output is stale.
  • python scripts/spec_lint.py docs/internal/api/<resource>/operations/*.html — section-level lint for API specs.
  • python scripts/spec_consistency.py — cross-checks error keys against the global error catalog.

CI / automated checks

  • make verify in CI runs docs-check; a failing run blocks merge.
  • Search index build (ADR 0027) detects orphan pages — a page not linked from the sidebar surfaces as a low-rank result and triggers an audit.

Operational checks

  • Quarterly screen-spec re-verify per documentation-process § Audit cadence.
  • Glossary scan on every release: any new domain term must have a glossary entry in the same PR.
  • Annual taxonomy review: confirm no new document type has accumulated outside the closed list in §1.

Rollout plan

  • Rollout phases: single PR. The artifact is documentation; there is no service to deploy.
  • Feature flags: none.
  • Rollback strategy: revert the PR. No persistent state, no migration.
  • Backward compatibility: none required. The RFC consolidates rules that already apply; existing pages remain valid until they are next edited, at which point the contributor brings them up to spec.

Change management

This RFC is the canonical statement of which templates we use and why. It is updated, not superseded, when a template gains or loses a section, when a new document type is introduced, or when an industry reference is replaced by a more relevant one. A new RFC is opened only if the underlying model itself changes — for example, a move away from co-located Docs-as-Code, or a decision to host documentation on a third-party platform.

  • Update this RFC when:
    • A required section is added, removed, or renamed in the API spec template or screen spec template.
    • A new document type is admitted into the closed taxonomy in §1.
    • The list of industry references in §"Why this template" is revised.
  • Open a new RFC when:
    • The repository moves away from static HTML under docs/.
    • A separate developer-portal product is introduced (currently out of scope per ADR 0025).
    • The single-source-of-truth principle for internal docs is revisited.
  • Owner after merge: the docs maintainer listed in data-maintainer-ids on this page.

Appendix A — API operation spec template, section catalogue

The live template at _shared/spec-template.html is the structural source of truth. The following list is a copy-pasteable reference for the 21 sections a real spec must contain (in this order). Sections marked required must be present and non-empty; sections marked optional may be removed when not applicable, but never reordered.

  1. Business context (required) — 2–4 sentences on the user problem and product outcome.
  2. User stories (required) — “As a <role>, I want <capability>, so that <outcome>.”
  3. Acceptance criteria (required) — Given / When / Then scenarios with IDs AC-<RESOURCE>-<NN>.
  4. Functional requirements (required) — bullet list with IDs FR-<RESOURCE>-<NN>.
  5. Non-functional requirements (required) — latency, throughput, availability, payload, concurrency, freshness, cacheability — each with a measurable threshold and ID NFR-<RESOURCE>-<NN>.
  6. Authorization & permissions (required) — link to _shared/auth.html; describe operation-specific deltas only.
  7. Request contract (required) — field-level table (name · location · type · format · required · nullable · default · constraints · business meaning · PII class · OpenAPI mapping) plus a JSON example.
  8. Response contract per success status (required) — same field-level table per 200 / 201 / 202 / 204.
  9. State transitions (optional) — PlantUML state diagram + transition table; required when the operation drives a state machine.
  10. Side effects (required) — DB writes, events emitted, downstream calls, cache invalidations, audit entries, listed as "on success / on failure / on idempotent replay".
  11. Idempotency & concurrency (required) — namespace, key composition, replay status code, concurrency strategy.
  12. Error catalog (required) — table of HTTP · code · key · when · log-event · envelope JSON; cross-checked against _shared/error-catalog.html.
  13. Edge cases checklist (required) — empty list, missing optional field, unicode, max-length, timezone crossover, concurrent writes, retry storm, soft-deleted/archived resource, partial failure.
  14. Observability requirements (required) — log events (name + level + fields), metrics (name + labels), trace spans, alerts and thresholds.
  15. Test plan (required) — unit · integration · contract (Schemathesis / Dredd) · end-to-end — each scenario links back to its AC-… ID.
  16. OpenAPI authoring hints (required) — schemas to reuse, examples to attach, security scheme, tags, what to hide from external integrators.
  17. Open questions (required) — bullet list with question · best guess · owner · deadline; or the literal word None.
  18. Out of scope (required) — explicit non-goals; or None.
  19. Traceability (required) — methodology, system-design, ADRs, tickets / RFCs, OpenAPI operationId.
  20. Spec changelog (required) — reverse-chronological, per contract intent (breaking · additive · fix · doc-only).
  21. Page history (required) — per-edit table (date · change · author).

The header strip carries the metadata visible to readers: HTTP method tablet, summary, status badge, availability (v1 · Internal / v1 · External), owner, operationId, last-reviewed date, and template version (v1). Body data attributes data-spec-status, data-spec-owner, data-spec-operation-id are machine-readable and consumed by the lint.

Appendix B — Screen spec template, section catalogue

The live template at screens/docs-screen-template.html is the structural source of truth. Required sections, in order:

  1. Title block — page title with the Screen badge; clear noun phrase.
  2. At-a-glance — status (draft · in-progress · stable · deprecated), primary URL, owner, audience, tracked features (DFEAT-xxx), last-verified date.
  3. Visual gallery — real screenshots covering desktop-light, desktop-dark, tablet (761–1024 px), mobile (≤760 px), and every transient state (intro, modal, loading, empty, error). Tier names follow the canonical viewport breakpoints. Files in ./assets/<screen-id>-<variant>.png; each <figure> gets a single-line <figcaption> and is wrapped in a link to the full-size PNG.
  4. Anatomy diagram — inline SVG with numbered callouts; legend below the figure; <title> + <desc> + <figcaption>; viewBox, no fixed dimensions; multi-phase screens stack two panels in one SVG.
  5. Layout map — text walk-through of every region; each region lists its CSS hook and purpose.
  6. Behaviour cards — one section per major behaviour (e.g. boot intro overlay, hero region, card sections); ordered by user-perceived priority.
  7. Data contract — every data-* attribute and class state used by the runtime; column shape: attribute · element · meaning · runtime consumer.
  8. Interaction logic — page lifecycle, event contract, timing budget, animation sequencing.
  9. Responsive behaviour — breakpoints with the layout change at each.
  10. Accessibility & motion policy — reduced motion, ARIA, focus, contrast, theme parity.
  11. Where to change what — file-by-file mapping from behaviour to source.
  12. Acceptance checklist — items a reviewer can tick post-change.
  13. Known edges and design decisions — non-obvious choices and historical fixes.
  14. Related internal docs — links to glossary, architecture map, sister specs.
  15. Page history — per-edit table.
