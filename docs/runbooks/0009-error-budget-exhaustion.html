Runbook: Error Budget Exhaustion and SLO Alerts

Trigger

  • Prometheus alerts StudyAppErrorBudgetExhausted, StudyAppErrorBudgetCritical, StudyAppFast5xxBurn, StudyAppHighHttp5xxRate, StudyAppLatencySLOViolation, or StudyAppReadyProbeFailing are firing.
  • Grafana SLO panels show HTTP error budget burn above 1, or sustained high 5xx rate, or readiness probe at 0.
  • External SLA breach or customer-visible outage correlated with the above.

What “error budget exhausted” means

For the Study App API, the internal SLO allows at most 0.1% of requests to return HTTP 5xx over a rolling 30-day window (99.9% success). The recording rule job:study_app:http_5xx:burn30d compares actual 5xx share to that allowance. When burn is above 1, the team has consumed more than the allotted error budget for the window.

Policy details and SLI definitions: ADR 0011: SLOs, SLAs, error budget, and monitoring alerts.

Fast triage (order)

  1. Readiness / dependencies — If StudyAppReadyProbeFailing fires, open Observability scrape failing patterns: confirm API is up, database reachable, curl -i http://127.0.0.1:8000/ready returns 200.
  2. 5xx and latency — Open Grafana dashboard Study App Observability and Prometheus Alerts / Graph.
  3. Recent change — Identify deploys, config changes, migrations, or traffic spikes in the incident window.
  4. Rollback — If a new release is the likely cause, roll back to the last known-good version per your deployment process.
  5. Saturation — Check CPU/memory, connection pools, and rate limits if latency SLO fires without clear 5xx spikes.

Response when budget is exhausted or critical

  • Incident — Treat sustained burn or customer impact as an incident: assign an incident lead, use a shared channel, and document timeline.
  • Feature freeze — Until stability is restored, defer non-essential releases and risky changes except fixes for the incident.
  • Root cause — After mitigation, capture root cause and follow-up items (tests, alerts, ADR).
  • External SLA — If a published SLA exists, follow customer communication and escalation processes defined for that SLA.

Exit criteria

  • Firing alerts are resolved or acknowledged with a documented exception.
  • 5xx rate and latency panels are back within SLO targets on the Grafana dashboard.
  • Error budget burn is trending down or a new window policy is agreed in writing.
