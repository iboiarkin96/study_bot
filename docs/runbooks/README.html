CI & Local Quality

CI and local quality failures

Runbooks for the gates that block a commit or CI run: failing tests, pre-commit hooks, the full make verify gate, and OpenAPI contract drift.

  • 0001 — Tests failing: make test or CI test stage fails; triage by DB state, fixtures, and assertions.
  • 0004 — Pre-commit failing: hooks fail on commit; staging conflicts, ruff-format, or mypy issues.
  • 0005 — Quality check failing: make verify fails on lint, types, tests, or docs sync stage.
  • 0007 — OpenAPI contract test failing: snapshot drift vs baseline; intentional vs accidental contract changes.
Data & Migrations

Data and migration failures

Alembic errors, schema mismatches, or DB state issues that block startup or break tests.

  • 0002 — Migrations failing: make migrate errors, bad autogenerated diffs, or history mismatch.
Security & API

Security and API guardrails

Auth, rate-limit, body-size, CORS, and security header failures on protected endpoints.

  • 0006 — API security failing: 401, 429, 413 spikes, or CORS errors; recovery by category.
Observability & Reliability

Observability and reliability incidents

Log pipeline failures, Prometheus scrape outages, and SLO alert responses including error-budget exhaustion.

  • 0003 — Logging failing: no log lines, bad LOG_DIR path, or missing write permission on logs/.
  • 0008 — Observability scrape failing: Prometheus target DOWN, empty /metrics, or Grafana panels show No data.
  • 0009 — Error budget exhaustion: SLO alerts firing; sustained 5xx burn, latency violations, or readiness probe failures.
Docs Frontend

Docs frontend incidents

Issues with the docs site runtime: missing TOC, broken navigation, or layout regressions.

  • 0010 — In-page TOC missing: "On this page" sidebar absent or empty; page shell, heading structure, and docs-nav.js checks.

Operational rules

  • If you find a policy or architecture gap, add or update an ADR.
  • If it is a process or step gap, update the runbook (this one or another).

Mini-SLA

  • During an incident: add short notes or a checklist to the runbook within 30 minutes.
  • After the fix: publish a clean runbook update within 24 hours.
  • Same incident twice in 30 days: update the runbook and add a preventive action (docs or ADR).
