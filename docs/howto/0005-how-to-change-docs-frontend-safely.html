How to change docs frontend safely (playbooks)

Overview

This page provides operational playbooks for common docs frontend changes. Each playbook includes where to edit, what to verify, and what documentation must be updated.

Related maps: architecture map, quick entry by role, navigation/search/discovery, UI/motion/adaptivity, and feedback/editorial workflow. Diagram-specific behavior is documented in diagrams and lightbox.

How to use Feature IDs in change requests

  • Reference impacted features by ID from the architecture ledger (for example, DFEAT-005 for Command Palette).
  • When a change spans multiple features, list all IDs in planning and QA notes.
  • If a request introduces new behavior, assign a new ID in the architecture map before implementation.
  • Use IDs in docs/CHANGELOG.md entries to make behavior changes traceable over time.

Playbook: add a new page action

  1. Add action definition in buildDocsPageActions() inside docs/assets/docs-nav.js.
  2. If action should appear in command palette, ensure it has clear group and keywords fields.
  3. If action opens external URL, set external: true so runtime applies safe link attributes.
  4. Validate toolbar rendering via injectDocsPageActions() and palette rendering via installDocsQuickActionsUi().
  5. Check desktop-only policy: action should not appear when viewport is below the page-actions breakpoint.

Feature IDs: DFEAT-006 (Page actions toolbar), DFEAT-005 (Command Palette).

Verify: keyboard execution, mouse execution, focus return, and no regressions in hotkeys.

Playbook: add a new feedback type in Page feedback

  1. Update the feedback type options list in injectDocsFeedbackCard() in docs/assets/docs-nav.js.
  2. Update inline guidance in the same form if the new type needs extra wording.
  3. Update .github/ISSUE_TEMPLATE/docs_feedback.md so template guidance matches UI options.
  4. Check that issue prefill payload created by docsFeedbackIssueUrl() includes the selected type.
  5. Validate cadence expectations in .github/workflows/docs-feedback-cadence.yml remain aligned.

Feature IDs: DFEAT-013 (Page feedback form).

Verify: form validation, issue link prefill, clipboard fallback, and live-status messaging.

Playbook: change breakpoint policy

  1. Update JS breakpoints in docs/assets/docs-nav.js and docs/assets/internal-sidebar.js where applicable.
  2. Update CSS media queries in docs/assets/docs-site-nav.css, docs/assets/internal-layout.css, and docs/assets/docs.css.
  3. Re-check policies for quick actions, top-nav groups, drawer mode, and desktop-only TOC visibility.
  4. Confirm Menu button and flashlight placement still align with title-row behavior on internal pages.
  5. Update docs pages that encode breakpoint contracts: #1 contract and UI/adaptivity page.

Feature IDs: DFEAT-001, DFEAT-002, DFEAT-005, DFEAT-009.

Verify: phone, tablet, desktop snapshots and keyboard behavior in each mode.

Playbook: add a new maintainer (Edited by workflow)

  1. Create profile page under docs/internal/portal/people/<slug>/index.html with required data-person-* attributes.
  2. Add maintainer id to page body attribute data-maintainer-ids on target documentation pages.
  3. Run make docs-fix to regenerate docs/assets/docs-portal-data.js.
  4. Confirm Edited by widget resolves profile correctly and avatar stack/list render as expected.
  5. If unknown-id warning appears, verify profile id and re-run generation.

Feature IDs: DFEAT-014 (Edited by), DFEAT-015 (Portal ownership lists).

Verify: profile link, image path, maintainer pages list, and portal group placement.

Playbook: change diagram lightbox behavior

  1. Update runtime logic in docs/assets/docs-diagram-lightbox.js (open/close, zoom, keyboard, pan, focus behavior).
  2. Update styling in docs/assets/docs.css for .diagram-expand-btn and .docs-diagram-lightbox* selectors.
  3. Keep binding contract stable for img.diagram--uml unless IA-level migration is planned.
  4. Run keyboard and pointer checks: Escape, +/-, Cmd/Ctrl+0, wheel zoom, drag pan.
  5. Update diagrams/lightbox documentation page and architecture ledger (DFEAT-017) if behavior changes.

Feature IDs: DFEAT-017.

Verify: focus return after close, body scroll lock while open, and no regressions on pages with multiple diagrams.

Playbook: change search telemetry or search-index lifecycle

  1. For event model changes, update telemetry emitters in docs/assets/docs-nav.js and keep event naming backward-compatible where possible.
  2. For endpoint contract changes, verify metadata configuration path and fallback behavior on local and hosted docs.
  3. For ranking or corpus changes, regenerate docs/assets/search-index.json and run query smoke tests on key docs topics.
  4. Update docs pages that define operational contracts: search/discovery and telemetry workflow.
  5. Record material telemetry/search behavior changes in docs/CHANGELOG.md.

Feature IDs: DFEAT-004, DFEAT-005, DFEAT-006, DFEAT-012.

Verify: search quality, keyboard interaction, telemetry payload integrity, and graceful behavior when index/endpoint is unavailable.

Cross-cutting safety checklist

  1. Run make docs-fix and make docs-html-check.
  2. Check light and dark themes for every changed surface.
  3. Check reduced-motion mode for all changed animations.
  4. Run keyboard-only pass for changed interactions.
  5. Update relevant docs pages in docs/internal/front/ and add docs/CHANGELOG.md entry for substantive behavior changes.
