How to change docs frontend safely (playbooks)

Overview

Playbooks for recurring docs-frontend changes. Each one names where to edit, what to verify afterwards, and which docs to keep in sync.

Related maps: architecture map, quick entry by role, navigation, search, and discovery, UI, motion, and adaptivity, feedback and editorial workflow. Diagram-specific behavior lives in diagrams and lightbox.

How to use Feature IDs in change requests

  • Reference impacted features by their ID from the architecture ledger (for example, DFEAT-005 for the Command Palette).
  • When a change spans multiple features, list every ID in the planning and QA notes.
  • If a request introduces new behavior, assign a new ID in the architecture map before implementation starts.
  • Use Feature IDs in docs/CHANGELOG.md entries so behavior changes stay traceable over time.

Playbook: add a new page action

  1. Add the action definition in buildDocsPageActions() inside docs/assets/docs-nav.js.
  2. If the action belongs in the command palette, give it explicit group and keywords fields.
  3. If the action opens an external URL, set external: true so the runtime applies the safe link attributes.
  4. Validate toolbar rendering through injectDocsPageActions() and palette rendering through installDocsQuickActionsUi().
  5. Confirm the desktop-only policy: the action does not appear when the viewport is below the page-actions breakpoint.

Feature IDs: DFEAT-006 (Page actions toolbar), DFEAT-005 (Command Palette).

Verify: keyboard execution, mouse execution, focus return, and no regressions in existing hotkeys.

Playbook: add a new feedback type in Page feedback

  1. Update the feedback-type options inside injectDocsFeedbackCard() in docs/assets/docs-nav.js.
  2. Update inline guidance in the same form when the new type needs extra wording.
  3. Update .github/ISSUE_TEMPLATE/docs_feedback.md so the template guidance matches the UI options.
  4. Confirm the issue prefill payload built by docsFeedbackIssueUrl() includes the selected type.
  5. Re-check the cadence settings in .github/workflows/docs-feedback-cadence.yml for alignment.

Feature IDs: DFEAT-013 (Page feedback form).

Verify: form validation, issue link prefill, clipboard fallback, and live-status messaging.

Playbook: change the breakpoint policy

  1. Update the JS breakpoints in docs/assets/docs-nav.js and, where relevant, in docs/assets/internal-sidebar.js.
  2. Update the CSS media queries in docs/assets/docs-site-nav.css, docs/assets/internal-layout.css, and docs/assets/docs.css.
  3. Re-check the policies for quick actions, top-nav grouping, the drawer mode, and the desktop-only TOC.
  4. Confirm that the Menu button and the flashlight still align with the title-row behavior on internal pages.
  5. Update the docs that encode breakpoint contracts: #1 contract and UI/adaptivity page.

Feature IDs: DFEAT-001, DFEAT-002, DFEAT-005, DFEAT-009.

Verify: phone, tablet, and desktop snapshots; keyboard behavior in each mode.

Playbook: add a new maintainer (Edited by workflow)

  1. Create the profile page under docs/internal/portal/people/<slug>/index.html with the required data-person-* attributes.
  2. Add the maintainer ID to data-maintainer-ids on every page they own.
  3. Run make docs-fix to regenerate docs/assets/docs-portal-data.js.
  4. Confirm the Edited by widget resolves the profile and that the avatar stack and list render as expected.
  5. If an unknown-id warning appears, re-check the profile ID and rerun the generator.

Feature IDs: DFEAT-014 (Edited by), DFEAT-015 (Portal ownership lists).

Verify: profile link, image path, the "Maintained pages" list, and portal group placement.

Playbook: change the diagram lightbox behavior

  1. Update the runtime logic in docs/assets/docs-diagram-lightbox.js (open and close, zoom, keyboard, pan, focus return).
  2. Update styling in docs/assets/docs.css under the .diagram-expand-btn and .docs-diagram-lightbox* selectors.
  3. Keep the binding contract for img.diagram--uml stable, unless the change is part of an IA-level migration.
  4. Run keyboard and pointer checks: Escape, + / -, Cmd/Ctrl+0, wheel zoom, and drag pan.
  5. Update the diagrams and lightbox docs page and the architecture ledger entry DFEAT-017 when behavior changes.

Feature IDs: DFEAT-017.

Verify: focus returns after close, the body scroll lock holds while the lightbox is open, and pages with multiple diagrams stay regression-free.

Playbook: change search telemetry or the search-index lifecycle

  1. For event-model changes, update telemetry emitters in docs/assets/docs-nav.js and keep event names backward-compatible where possible.
  2. For endpoint contract changes, verify the metadata configuration path and the fallback behavior on local and hosted docs.
  3. For ranking or corpus changes, regenerate docs/assets/search-index.json and run query-smoke tests on key topics.
  4. Update the docs that hold the operational contracts: search and discovery and telemetry workflow.
  5. Record material telemetry or search-behavior changes in docs/CHANGELOG.md.

Feature IDs: DFEAT-004, DFEAT-005, DFEAT-006, DFEAT-012.

Verify: search quality, keyboard interaction, telemetry payload integrity, and graceful behavior when the index or endpoint is unavailable.

Cross-cutting safety checklist

  1. Run make docs-fix and make docs-html-check.
  2. Check the light and dark themes on every changed surface.
  3. Check reduced-motion mode for every changed animation.
  4. Run a keyboard-only pass on every changed interaction.
  5. Update the relevant pages under docs/internal/front/ and add a docs/CHANGELOG.md entry for any substantive behavior change.
