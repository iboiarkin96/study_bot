Beginner walkthrough: add an endpoint and matching docs

An execution plan for adding a new write endpoint across every layer. The example target is POST /api/v1/contract; the steps generalise to any new resource.

Goal and scope

Target endpoint: POST /api/v1/contract.

Done means:

  • The route is registered and visible in Swagger.
  • Schemas and OpenAPI examples cover request, business errors, and validation errors.
  • Business logic lives in the service layer; the repository handles persistence only.
  • Validation mapping uses stable, entity-prefixed codes and keys.
  • Tests, docs, and quality gates all pass.

Prepare the domain model and the database

Skip this section if the endpoint does not need new tables or columns.

  1. Add or update the ORM model in app/models/core/ (for example contract.py).
  2. The project relies on package-level imports (from app.models.core import *). Register the new model in app/models/core/__init__.py so Alembic discovers it.
  3. Generate a migration: 
    .venv/bin/python -m alembic revision --autogenerate -m add_contract_table
  4. Apply the migration: 
    make migrate

Add the repository layer

File: app/repositories/contract_repository.py.

Typical methods:

  • create(...)
  • get_by_...(...) when deduplication or business checks need a lookup

Repository rules:

  • Data access only.
  • No HTTP exceptions.
  • No FastAPI imports.

Add the service layer (business logic)

File: app/services/contract_service.py.

The service is responsible for:

  • Applying business rules.
  • Calling the repository.
  • Raising business errors with a stable code and key.

Naming convention for error codes on this entity:

  • Keys carry the entity prefix, for example CONTRACT_CREATE_*.
  • Codes follow the entity prefix plus a number, for example CONTRACT_101.

Add request and response schemas

File: app/schemas/contract.py.

Define:

  • ContractCreateRequest
  • ContractCreateResponse

Schema rules:

  • Validation lives at the API boundary.
  • Every field has a clear description.
  • Names are stable; OpenAPI consumers depend on them.

Add validation mapping for the entity

File: app/validation/contract.py.

The mapper covers:

  • Field- and type-specific mapping to stable validation codes.
  • A fallback rule for validation errors that have no explicit mapping.

Naming examples:

  • Codes: CONTRACT_001, CONTRACT_002, …
  • Keys: CONTRACT_CREATE_FIELD_REQUIRED

Wire the mapper into the central validation entry point. The project supports two wiring styles; pick whichever exists today:

  • Route-based dispatch inside the validation builder, or
  • A small registry (endpoint -> mapper) in app/validation/__init__.py.

Add OpenAPI examples

Files:

  • app/openapi/examples/contracts.py (new)
  • app/openapi/examples/__init__.py — add the new exports

Cover:

  • Request body examples.
  • Business error examples.
  • Validation error examples.

Add the API router

File: app/api/v1/contract.py.

The router:

  • Declares APIRouter(prefix="/contract", tags=["Contract"]).
  • Adds @router.post("").
  • Uses the request and response schemas above.
  • References the OpenAPI examples.
  • Declares an explicit responses map for error codes.
  • Documents and enforces Idempotency-Key behavior, since this is a write endpoint.

Register the router in the app: update app/main.py with app.include_router(..., prefix="/api/v1").

Add tests

File: tests/api/v1/test_contract_create.py.

Minimum scenarios:

  1. Success: 201.
  2. Business failure: 400.
  3. Validation failure: 422 with the entity-prefixed code and key.
  4. Idempotent replay: same key plus same payload returns the same success response.
  5. Idempotency conflict: same key plus a different payload returns a conflict.

Update tests/conftest.py when fixtures need new tables or seed data.

Update the docs

Sync every doc artifact:

make docs-fix

Run quality gates

Before each commit:

  1. make format-fix
  2. make lint-check
  3. make type-check
  4. make test
  5. make pre-commit-check

Before opening the PR or deploying:

  1. make verify
  2. make release-check

Final checklist

  • [ ] The endpoint is registered and appears in Swagger.
  • [ ] Schemas and OpenAPI examples are complete.
  • [ ] Business logic is isolated in the service layer.
  • [ ] The repository handles persistence only.
  • [ ] Validation mapping exists for contract.
  • [ ] Error keys and codes use the CONTRACT_* prefix.
  • [ ] Tests cover success, business failure, and validation failure.
  • [ ] Idempotency scenarios are covered for the write endpoint.
  • [ ] Docs are synced.
  • [ ] Quality gates pass.
