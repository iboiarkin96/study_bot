Onboarding: from zero to endpoint and docs

Bring the project up locally, add one endpoint across every layer, and ship the matching docs in the same PR.

Goal

By the end of this guide:

  • The API runs locally and answers a health probe.
  • One new endpoint exists across the router, schema, service, and repository layers.
  • OpenAPI examples and internal HTML docs cover the endpoint.
  • make verify passes from a clean working tree.

Phase 1. Local setup and first successful call

  1. Create the virtual environment and install dependencies: 
    make venv && make install
    The command ends with Dependencies installed.
  2. Initialize local config: 
    make env-init
    You see .env created from env/example. If .env already exists, keep the current file.
  3. Apply database migrations: 
    make migrate
    Output ends with Migrations applied.
  4. Start the API: 
    make run
    Uvicorn binds to 127.0.0.1:8000 by default, or to the host and port set in .env.
  5. From a second terminal, hit the health endpoint: 
    curl -i http://127.0.0.1:8000/live
    Expected response: HTTP/1.1 200 OK and a JSON status body.

Phase 2. Implement one endpoint across layers

Build the endpoint in the order below. The diff stays reviewable and follows the project's layering.

  1. Add a migration and the matching schema change, if the endpoint needs new tables or columns.
  2. Add request and response models in app/schemas/.
  3. Add data-access methods in app/repositories/.
  4. Add service logic and business rules in app/services/.
  5. Add the router handler in app/api/v1/ with validation and response mapping.
  6. Update OpenAPI examples in app/openapi/examples/ where relevant.

For a worked example, follow Add POST /api/v1/contract.

Phase 3. Document the endpoint

  1. Update the resource hub and operation pages under docs/internal/api/ (resource index.html and any operation page the change affects).
  2. When you add or rename a page under docs/internal/, update the sidebar source docs/assets/internal-sidebar.js.
  3. Match the structure in the internal service docs layout guide.
  4. Regenerate every doc artifact: 
    make docs-fix

Phase 4. Ready for review

  1. Run the full local gate. This is the target CI runs: 
    make verify
    It covers drift detection, baseline accessibility, the OpenAPI contract, and the test suite.
  2. Confirm the diff contains both the code change and the matching docs change for the new endpoint.
