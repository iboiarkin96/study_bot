Backlog: priorities and status

A single prioritized list of engineering work, from critical to nice-to-have. Estimates are rough calendar-day ranges assuming steady focus (weekends excluded) at 3 / 2 / 1 hours per day—not commitments, but a shared planning baseline. Update each item’s status by changing the pill class (see legend). This roadmap markup is separate from ADR lifecycle (data-adr-weight on ADR pages — see ADR 0018).

See also: Architecture & quality assessments (indexed maturity reports under docs/audit/, ADR 0024).

Status legend

Work item states
To do — queued; not started
In progress — actively in progress
Done — completed and accepted
Blocked — waiting on external dependency or decision

How to update status in this page (README.html)

  1. Status colors are defined in docs/backlog/backlog.css using CSS custom properties.
  2. To change the palette, edit the --status-* variables; all pills on the page follow them.
  3. In each backlog item heading (<h2>), find <span class="status-pill status-pill--..."></span> and replace the suffix after status-pill-- with one of: todo, in-progress, done, blocked.

P0 — critical CI pipeline: quality gates, coverage threshold, docs-check Done

1) Summary
Automate the same checks developers run locally (make verify or equivalent) in continuous integration: linters, types, tests with a minimum coverage floor, OpenAPI/doc consistency, and optional contract tests—so main stays releasable.
2) Problem & value
Problem: Strict local workflows do not protect the default branch if CI is missing or weaker; regressions and doc drift merge silently.
Value: Shifts defects left; gives reviewers a green pipeline signal; matches industry practice (quality gates per commit/PR).
3) Scope & deliverables
GitHub Actions (or equivalent): run formatter/linter, mypy, pytest with coverage and fail-under, OpenAPI check, contract tests, docs-check; cache dependencies for speed; fail the build on violations. Optionally run the same hooks in CI that pre-commit uses locally. Document how to reproduce CI locally in contributor docs.
4) Rough estimate (calendar days)
3 h/day ~4 days (~12 h)
2 h/day ~6 days
1 h/day ~12 days
5) Delivered (to date)
.github/workflows/ci.yml on push/PR to main or master: Python 3.11, pip cache, make verify then pre-commit run --all-files. Coverage enforced via pytest-cov and fail_under in pyproject.toml. Local pre-push drift check: make verify-ci. Contributor note in engineering practices (CI / reproduce locally).

P1 DB uniqueness, race safety, and documented concurrency behavior Done

1) Summary
Encode business invariants in the schema (unique constraints), map database conflict errors to the API error contract, and document concurrent scenarios (double submit, retries, idempotency) so behavior under load is explicit.
2) Problem & value
Problem: “Check-then-act” without unique indexes races under concurrency and causes duplicate rows and production incidents.
Value: Data integrity at the database layer; predictable client handling; less tribal knowledge for support and new engineers.
3) Scope & deliverables
Add constraints on natural keys; consistent HTTP/status codes for conflicts; developer doc table: scenarios (duplicate create, idempotent retry) and expected outcomes.
4) Rough estimate (calendar days)
3 h/day ~3 days (~10 h)
2 h/day ~5 days
1 h/day ~10 days
5) Delivered (to date)
Alembic/SQLAlchemy unique indexes and constraints on natural keys (e.g. systems, timezones, users, idempotency_keys composite uniqueness per ADR 0006); HTTP/error contract for conflicts and idempotency replay; ADR 0006 plus developer docs and error matrix for retry and idempotency semantics.

P3 Distributed rate limiting with Redis (multi-instance) To do

1) Summary
Replace per-process in-memory rate limiting with a shared store (Redis) so limits are consistent across horizontally scaled API instances, with the same HTTP 429 contract clients already expect.
2) Problem & value
Problem: Per-replica counters let abusive traffic multiply by instance count; fairness breaks under autoscaling.
Value: Standard pattern for APIs behind load balancers; prerequisite for fair abuse protection at scale.
