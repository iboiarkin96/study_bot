How to use: Filter guide and backlog model

Use quick presets for fast managerial views. Use detailed filters for exact slicing by block, priority, and status.

Methodology to avoid duplication: data-group answers where work happens (frontend/backend/devops/docs), while data-tags answers what kind of work it is (bug/feature/research/tech-debt). Do not repeat domain names in tags.

Quick filters

  • All — shows every task, no preset restrictions.
  • My focus — open tasks with priority P0 or P1.
  • Open — only active states: to do, in progress, blocked.
  • Blocked — only tasks currently blocked by dependency/decision.
  • High risk — priority P0 items tagged bug or tech-debt.

Delivery groups (data-group)

  • frontend — UI, client integration, frontend tooling
  • backend — API/service/domain/data-layer logic
  • devops — CI/CD, runtime, infra, observability, security ops
  • docs — internal/external docs, guides, specs readability

Task tags (data-tags)

  • bug — defects, regressions, and broken flows
  • feature — new capability or meaningful product improvement
  • research — pilot, spike, or validation before full rollout
  • tech-debt — refactor, maintainability, hardening, process quality

Priority levels

  • P0 — critical (release blocker or production risk)
  • P1 — high (next sprint / near-term delivery)
  • P2 — medium (planned improvements)
  • P3 — low (later or optional improvements)

Risk and confidence

  • data-risk: high, medium, low — delivery/coordination risk for the task.
  • data-confidence: high, medium, low — confidence in estimate/scope accuracy.

Status workflow

Work item states
To do — not started
In progress — someone is working on it
Done — finished and accepted
Blocked — waiting on someone or a decision
Rejected — we decided not to do this (out of scope or obsolete)

How to edit backlog cards (README.html)

  1. For each item set data-tags, data-priority, data-status, and data-group on the card section.
  2. Allowed group values: frontend, backend, devops, docs.
  3. Allowed tag values: bug, feature, research, tech-debt.
  4. Allowed priorities: P0, P1, P2, P3.
  5. Allowed risk values in data-risk: high, medium, low.
  6. Allowed confidence values in data-confidence: high, medium, low.
  7. Allowed status values in data-status: todo, in-progress, blocked, done, rejected.
  8. Status pill text and status modifier class are auto-synced from data-status in backlog.js; keep heading pill as <span class="status-pill"></span>.
  9. Optional KPI fields for better cockpit analytics: data-blocked-days="N" or data-blocked-at="YYYY-MM-DD" (for SLA risk).
  10. Optional flow fields: data-done-days-ago="N" or data-done-at="YYYY-MM-DD", plus data-actual-hours="N" for factual effort.
  11. ETA default is derived from the recalibrated estimate at 3 h/day. Optional override fields (days, decimals allowed): data-eta-days="A-B" or numeric data-eta-min-days="A" + data-eta-max-days="B".
