Load testing (tools/load_testing)

Real HTTP requests, including writes to the database. Policy and scenarios: Developer: API load testing, ADR 0012.

The runner does not call the Python handler directly (that would be a different kind of test): requests go over the network like curl so metrics capture the same middleware and status codes. To avoid duplicating the endpoint contract, user bodies are built via UserCreateRequest from app.schemas.user; the path comes from USER_HTTP_BASE_PATH in app.api.v1.user.

Architecture (overview)

The runner loads Python modules from scenarios/, builds a plan from weights, and sends requests. The chain matches the load-planning idea: 

flowchart LR
  subgraph scenarioTree [scenarios_directory]
    W[weights_py]
    U[user_create_py]
    O[observability_http500_py]
  end
  R[runner_py]
  R --> scenarioTree
  R --> H[httpx_Client]
  H --> API[FastAPI]
  API --> DB[(database)]

Scenario weight formula

Final weight of one named scenario:

flowchart TD
  GW["GROUP_WEIGHTS[group]"]
  SH["SHARE_OF_GROUP[file]"]
  MX["MIX[scenario_key]"]
  F["weight = GW * SH * MX"]
  GW --> F
  SH --> F
  MX --> F

In each scenario file MIX sums to ~1.0; across files in one group SHARE_OF_GROUP sums to ~1.0; GROUP_WEIGHTS in weights.py sums to ~1.0 overall.

Running

From the repository root (virtualenv activated):

python -m tools.load_testing.runner --dry-run
python -m tools.load_testing.runner --total-requests 200
python -m tools.load_testing.runner --total-requests 200 -q
LOAD_TEST_BASE_URL=http://127.0.0.1:8000 python -m tools.load_testing.runner --delay-ms 0

During a real run the terminal prints a "Start" line and periodic progress [N/total] scenario → HTTP code; without it output would only appear at the end (long gaps due to --delay-ms). The -q / --quiet flag disables this progress.

Environment variables: LOAD_TEST_BASE_URL, LOAD_TEST_API_KEY, LOAD_TEST_DELAY_MS (default ~1 s between requests to avoid hitting the API limit), LOAD_TEST_VERBOSE=1 or -v / --verbose — log which modules were skipped because the group weight is zero (e.g. observability_5xx).

If the log shows mismatches and 404 / 429

This is usually not a runner bug. 404 on /__loadtest/http500 — the route exists only when LOADTEST_HTTP_500=true on the server; by default group weight observability_5xx in scenarios/weights.py is 0, so that scenario does not run. 429 — rate limit. Options: increase --delay-ms; for local-only load raise the limit on the API side — make run-loadtest-api (starts API with a high limit, waits for /ready, runs python -m tools.load_testing.runner, stops API; see tools/load_testing/run_with_local_api.sh (warning and "continue?" prompt before start; cancel skips the run; for CI set LOADTEST_SKIP_CONFIRM=1). Default limit for runs is 1000000000 requests per window (API_RATE_LIMIT_REQUESTS_LOADTEST). Long-lived server with the same limit — make run-loadtest-api-serve. Or temporarily raise API_RATE_LIMIT_REQUESTS / API_RATE_LIMIT_WINDOW_SECONDS in .env (not for prod).

Directory layout

  • scenarios/weights.py — GROUP_WEIGHTS (group shares: user, observability_5xx, …)
  • scenarios/user/create.py — POST /api/v1/user, MIX, SCENARIOS, SHARE_OF_GROUP
  • scenarios/observability/http500.py — GET /__loadtest/http500 (requires LOADTEST_HTTP_500=true)
  • scenarios/conspectus/ — placeholder for new endpoints (README.txt inside)

How to add an endpoint

  1. Create tools/load_testing/scenarios/<entity>/<name>.py.
  2. Set GROUP (key from weights.py), SHARE_OF_GROUP, MIX (sums to ~1.0), SCENARIOS (key → callable(ctx) -> BuiltRequest).
  3. For a new entity, add the group share in GROUP_WEIGHTS.
  4. Scenario keys are globally unique (e.g. conspectus.create.ok).

Multiple files in one group

For example user/create.py and user/get.py: SHARE_OF_GROUP across those files should sum to ~1.0; in each file MIX sums to ~1.0.

422 on user create

See user_payload.apply_break_field and fields in scenarios/user/create.py.
